OpenAI は **2026年6月4日**、ChatGPT のメモリを改善する新しい仕組みとして **dreaming** を発表した。発表によると、ChatGPT は利用者が直接やり取りしていない間に過去の会話を参照し、利用者にとって役立つ可能性のある洞察をまとめる。まず米国の Plus / Pro 利用者から提供される更新で、長期的な個人化を強める方向のプロダクト変更である。

この話は、単なる便利機能として読むと危ない。以前の [ChatGPTセッション管理](/blog/openai-chatgpt-active-sessions-codex-2026/) は「どの端末から使っているか」を扱った。今回のメモリ刷新は「ChatGPT が利用者をどう覚えるか」を扱う。さらに [OpenAI Skills統制](/blog/openai-chatgpt-skills-governance-compliance-2026/) や [Offline検索](/blog/openai-offline-web-search-chatgpt-workspaces-2026/) と同じく、便利な機能ほど管理者、法務、現場が先に運用線を引くべき更新である。

## 事実: dreaming は何を変えるのか

OpenAI の発表では、dreaming は ChatGPT が過去の会話から学び、利用者の関心や文脈に合わせた支援をしやすくする仕組みとして説明されている。従来のチャットは、利用者が質問した瞬間に文脈を取り出して応答する体験が中心だった。今回の発表は、ChatGPT が会話の外側でも過去のやり取りを整理し、より長い時間軸で利用者を支援する方向を示している。

対象はまず Plus / Pro の米国利用者であり、すべての地域や企業プランに即時展開される話ではない。日本企業が今日から全社員に同じ機能を前提に設計できるわけではない。ただし、OpenAI が ChatGPT を短期的な質問応答ツールから、継続的に利用者を覚える業務アシスタントへ寄せていることは明確だ。

ここで重要なのは、メモリと会話履歴を一つに混ぜないことだ。OpenAI の Memory FAQ は、保存されたメモリと、過去チャットを参照する機能を分けて説明している。保存メモリは、利用者が覚えてほしいことや ChatGPT が役立つと判断した情報である。一方、Reference chat history は、過去の会話から応答に役立つ文脈を参照する設定である。dreaming の価値も、この長期文脈をどう扱うかにかかっている。

## 事実: メモリには削除と制御がある

OpenAI は、利用者が ChatGPT に何を覚えているか尋ねたり、特定のメモリを削除したり、設定からメモリを管理したりできると説明している。Temporary Chat を使えば、通常のメモリへ残さずに会話する選択肢もある。これは、個人化を強める一方で、利用者が覚えられたくない情報を扱うための基本線である。

ただし、削除操作を過信してはいけない。Memory FAQ は、保存メモリとチャット履歴の関係、削除後の扱い、モデル改善への利用可否が設定やプランにより変わることを説明している。企業利用では、利用者本人の画面操作だけで統制が完結するとは考えず、ワークスペース設定、データ保持、ログ、利用規程と合わせて見る必要がある。

Enterprise / Edu では、管理者がメモリ関連機能を制御できる。これは日本企業にとって重要だ。部署ごとの利用方針、機密情報の入力禁止、採用・人事・医療・金融など高感度業務での利用範囲を決めるとき、個人ユーザーの好みだけに任せる設計は危うい。個人化AIは、便利になるほど業務情報を覚えやすくなる。

## 分析: 日本企業では説明責任が先に来る

ここからは分析だ。

日本企業で ChatGPT のメモリを導入する場合、最初の論点は技術的な精度ではなく説明責任である。利用者に対して、ChatGPT が何を覚える可能性があるのか、何を覚えさせてはいけないのか、削除や Temporary Chat をいつ使うのかを説明できなければ、個人化の価値より不安が先に立つ。

特に、営業、CS、採用、法務、開発、情シスでは入力される情報の種類が違う。営業なら顧客名や商談履歴、採用なら候補者情報、法務なら契約条項、開発なら未公開仕様やリポジトリ情報が混ざる。これらを一律に「ChatGPT が覚えると便利」と扱うのは乱暴だ。業務ごとに、覚えてよい利用者嗜好、覚えてはいけない顧客・個人・機密情報を分ける必要がある。

この点は、以前の [OpenAI新セキュリティ設定](/blog/openai-advanced-account-security-codex-2026/) とも接続する。アカウントの入口を強くしても、利用者が機密情報を長期メモリに残す運用ならリスクは残る。逆に、メモリをすべて禁止すれば安全に見えるが、全社導入時の使い勝手や定着率は落ちる。現実的には、機能を全開放するか全面禁止するかではなく、業務カテゴリごとに許可範囲を決めることになる。

## 分析: メモリは「個人化」と「台帳化」を分ける

ChatGPT のメモリは、利用者の好みを覚える用途には向いている。たとえば、回答の長さ、文体、よく使うフレームワーク、普段の役割、作業の優先順位などだ。開発チームなら、レビュー観点、使う言語、チームの命名規則、ドキュメント形式を覚えると便利になる。

一方で、顧客ごとの契約条件、社員の評価、未公開決算、障害の詳細、個人情報をメモリに任せるべきではない。これらは、ChatGPT の個人化メモリではなく、権限管理されたCRM、チケット、文書管理、ナレッジベースで扱うべき情報だ。AIに覚えさせることで便利になる情報と、会社の台帳として管理すべき情報を分ける必要がある。

この分離は、[OpenAI Skills統制](/blog/openai-chatgpt-skills-governance-compliance-2026/) で扱った「業務手順の部品化」と似ている。Skills は会社標準の作業を部品として管理する。一方、メモリは利用者ごとの文脈を補助する。会社標準を個人メモリに寄せすぎると、異動や退職で再現性が落ちる。個人化と業務標準は、役割を分けて設計したほうがよい。

## 実務: 導入前チェックリスト

日本企業が今回の発表を受けて準備するなら、まず対象プランと地域を確認する。今回の dreaming は米国 Plus / Pro からの展開であり、日本の Enterprise 利用に同時適用されるとは限らない。発表の範囲と、自社ワークスペースで実際に使える設定を切り分ける。

次に、ユーザー向け説明を作る。ChatGPT が覚えてよい情報、覚えてはいけない情報、削除方法、Temporary Chat の使い方、メモリを無効化する場面を短くまとめる。長い規程だけでは現場に届かない。オンボーディング資料、社内FAQ、AI利用ガイドに入れるほうがよい。

三つ目は、管理者設定の棚卸しだ。Enterprise / Edu でメモリ関連設定をどう扱うか、部署単位で差をつけるか、監査や問い合わせ時に誰が確認するかを決める。ここは [ChatGPTセッション管理](/blog/openai-chatgpt-active-sessions-codex-2026/) と同じく、利用者本人任せにしすぎない設計が必要になる。

四つ目は、禁止入力の明文化である。顧客の個人情報、未公開契約、医療・人事・労務情報、ソースコード上の秘密情報、アクセスキー、社外秘の障害内容をメモリへ残さない。通常チャットでの入力可否と、長期メモリへ残してよいかは別の判断である。

## まとめ

OpenAI の ChatGPT メモリ刷新は、個人化AIの実用性を上げる更新である。dreaming により、ChatGPT は過去の会話をより長期的に活用し、利用者に合った支援をしやすくなる。ただし、企業利用では「便利になった」で終わらせてはいけない。

日本企業が見るべきなのは、メモリをどの業務で許可し、どの情報を残さず、利用者に削除や Temporary Chat をどう説明するかである。セッション、検索、Skills、メモリは別々の機能に見えるが、企業導入ではすべて同じ問いに戻る。ChatGPT に何を任せ、何を覚えさせ、後からどう説明できるかである。
