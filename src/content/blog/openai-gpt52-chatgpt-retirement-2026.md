OpenAI は 2026年6月12日の ChatGPT Release Notes で、**GPT-5.2 models を ChatGPT から退役させた**と発表した。対象は GPT-5.2 Instant、GPT-5.2 Thinking、GPT-5.2 Pro で、既存会話は対応する GPT-5.5 モデルで自動的に続く。新モデルの発表ではなく、ChatGPT のモデル棚から旧世代を外す更新である。

この更新は、5月に扱った [OpenAI o3/GPT-4.5退役、ChatGPT運用棚卸し](/blog/openai-o3-gpt45-chatgpt-retirement-2026/) の続きとして読むと分かりやすい。o3 / GPT-4.5 退役は旧モデル名と Canvas 前提手順の棚卸しだった。今回は、より最近まで使われていた GPT-5.2 系の会話が GPT-5.5 系へ移るため、長期会話、GPTs、社内プロンプト集、研修資料の見直しが主題になる。

また、[GPT-5.5 InstantでChatGPT標準運用はどう変わるか](/blog/openai-gpt-55-instant-chatgpt-default-2026/) で整理した通り、OpenAI は標準体験を GPT-5.5 系へ寄せている。今回の退役はその後始末ではなく、ChatGPT を業務基盤として使う企業が、モデルライフサイクルをどう管理するかを問う更新だ。

## 事実: 6月12日に何が変わったか

OpenAI のリリースノートでは、2026年6月12日時点で GPT-5.2 models は ChatGPT で利用できなくなったと説明されている。対象は GPT-5.2 Instant、GPT-5.2 Thinking、GPT-5.2 Pro である。既存の GPT-5.2 会話は、対応する GPT-5.5 モデルで自動的に継続される。

ここで重要なのは、会話そのものが消える話ではない点だ。ユーザーは過去の会話を開けるが、その続きを書くモデルは GPT-5.5 系に変わる。つまり、同じ会話スレッドでも、過去の出力と今後の出力で文体、推論の深さ、回答の長さ、ツール利用の判断が変わる可能性がある。

OpenAI は、GPT-5.2 退役が以前から告知されていたとも説明している。後継モデルが出たあと、ChatGPT ではモデルを概ね 90 日残す方針が示されている。これは利用者にとって移行期間を読みやすくする一方、企業側には「いつまでも旧モデルを前提にしない」管理を求める。

関連する Help Center では、GPT-5.5 Thinking が ChatGPT の各種ツールをサポートし、GPT-5.5 Instant は Canvas を除く現在の ChatGPT ツールをサポートすると説明されている。つまり、GPT-5.2 から GPT-5.5 へ移るときは、単にモデル名だけでなく、どの作業を Instant / Thinking / Pro のどれに寄せるかを見直す必要がある。

## 事実: API変更と混同しない

今回の主語は ChatGPT である。OpenAI の ChatGPT Release Notes は、ChatGPT の画面で使えるモデルと既存会話の扱いを説明している。API 本番システムのモデル指定が、6月12日の ChatGPT 更新だけで自動的に置き換わると読むべきではない。

この切り分けは日本企業ではかなり大事だ。社内では、ChatGPT の利用者、API を組み込む開発チーム、情シス、法務、AI 推進部門が同じ「OpenAI のモデル」として話しがちである。しかし、ChatGPT のモデル棚と API のモデルライフサイクルは同じではない。ChatGPT 側の退役を見て API 改修を急ぐと、不要な作業が発生する。

一方で、API チームが無視してよい話でもない。ChatGPT で試したプロンプトを API 実装へ移すチームでは、ChatGPT 側の出力が GPT-5.5 系へ変わることで、評価時の基準がずれる。検証環境では ChatGPT の結果を参考にしつつ、本番 API は固定モデル、評価データ、ロールバック手順で管理するほうが安全である。

この考え方は、以前の [OpenAIのモデルピッカー刷新と推論制御の見直し](/blog/openai-chatgpt-model-picker-reasoning-controls-2026/) と同じ線上にある。利用者向け UI は分かりやすく整理されるほど、裏側の運用では「どの用途にどのモデル相当を使うのか」を言語化する必要が増える。

## 分析: 日本企業が棚卸しすべきもの

ここからは分析だ。

日本企業がまず確認すべきなのは、GPT-5.2 というモデル名がどこに残っているかである。社内 wiki、Notion、Google Drive、Slack、研修資料、プロンプト集、GPTs の説明文、問い合わせ対応マニュアルを検索し、`GPT-5.2`、`5.2 Thinking`、`5.2 Pro` といった記述を拾う。

次に見るべきは、長期会話である。ChatGPT を相談相手として継続的に使う部門では、数週間から数カ月にわたる会話スレッドが業務の文脈を持っていることがある。企画、採用、法務、開発設計、営業提案、カスタマーサポート改善などでは、過去のやり取りを前提に次の相談をする。既存会話が GPT-5.5 系で続くなら、出力の違いを受け入れられるか確認したい。

GPTs も棚卸し対象になる。GPTs の説明や社内案内に「GPT-5.2 Thinking で使う」「5.2 Pro 推奨」と書かれている場合、利用者は退役後に迷う。GPTs の owner、用途、想定モデル、検証日を持たないまま増えている組織では、今回のようなモデル退役が管理不全をあぶり出す。

研修資料も影響を受ける。新入社員向け AI 研修や部門別ハンズオンで、画面キャプチャ付きのモデル選択手順を配っている場合、モデル棚が変わるだけで資料が古くなる。これは小さな手間に見えるが、全社展開では問い合わせ増につながる。更新できない資料は、具体的なモデル名ではなく「日常作業は Instant、深い検討は Thinking、最終確認は Pro」のような用途分類に寄せるほうが長持ちする。

## 開発チームへの意味

開発チームにとって、今回の更新は「GPT-5.2 が使えなくなった」だけではない。ChatGPT での探索、API 実装、Codex や IDE での開発支援を分けて管理する必要がある、という合図である。

まず、ChatGPT を使ったプロンプト検証は、モデル名と日付を記録したほうがよい。6月11日に GPT-5.2 でよかったプロンプトが、6月13日に同じ会話で GPT-5.5 系へ移って同じ出力になるとは限らない。検証メモには、ChatGPT のモデル、実行日、想定用途、合格基準を残す。

次に、API 実装では固定モデルと可変モデルを分ける。`chat-latest` のように最新体験へ追随する指定は検証環境では便利だが、顧客向け本番や社内の重要業務では出力の安定性が問題になることがある。固定モデル、評価データ、比較環境、ロールバックの責任者を決めておくべきだ。

さらに、開発支援では ChatGPT と Codex の役割を分けたい。ChatGPT は調査、設計相談、レビュー観点の整理に向く。一方で、実際のコード変更や検証はリポジトリ、CI、レビューの中で行う必要がある。[OpenAI新セキュリティ設定でChatGPTとCodex運用はどう変わるか](/blog/openai-advanced-account-security-codex-2026/) で扱ったように、アカウント保護、セッション管理、コード実行面はモデル選択とは別の統制である。

## 実務チェックリスト

まず、社内資料を検索する。`GPT-5.2`、`GPT-5.2 Thinking`、`GPT-5.2 Pro`、`Legacy Models`、`モデルピッカー` で横断検索し、該当箇所を「高リスク業務」「教育資料」「単なる古いメモ」に分ける。

次に、重要な長期会話を確認する。特に、法務、採用、営業提案、経営企画、開発設計のように成果物へ直結する会話は、GPT-5.5 系で続きを実行したときに出力の長さ、根拠の出し方、結論の出し方が変わらないか見る。

3つ目に、GPTs と共有プロンプトの owner を確認する。owner がいない GPTs は、モデル退役時に誰も説明を直さない。少なくとも社内配布している GPTs には、用途、想定モデル分類、最終確認日、問い合わせ先を持たせる。

4つ目に、API 利用チームへ「今回の主語は ChatGPT」と共有する。API 側は別途 OpenAI Platform のモデルドキュメント、価格、deprecation notice を見て管理する。ただし、ChatGPT で検証した出力を本番 API の期待値として使う場合は、モデル差分を明記する。

最後に、次の退役に備える。OpenAI は後継モデル公開後、ChatGPT で旧モデルを概ね 90 日残す方針を示している。つまり、今後もモデル退役は例外ではなく通常運用になる。情シスや AI 推進部門は、モデル名、用途、owner、検証日、代替方針を持つ軽量な台帳を作っておくとよい。

## まとめ

OpenAI の 2026年6月12日更新は、GPT-5.2 Instant / Thinking / Pro を ChatGPT から退役させ、既存会話を対応する GPT-5.5 モデルで継続させるものだった。新機能発表ではないが、ChatGPT を日常業務に使っている日本企業には実務影響がある。

見るべき点は、モデル性能の優劣だけではない。長期会話、GPTs、社内プロンプト集、研修資料、API 検証メモが GPT-5.2 という名前に依存していないかを確認することだ。ChatGPT のモデル棚は今後も動く。今回を機に、モデル名ではなく用途とリスクで運用を整理しておきたい。
