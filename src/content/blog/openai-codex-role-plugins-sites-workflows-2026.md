OpenAI は 2026年6月2日、Codex を「すべての役割、ツール、ワークフロー」に広げる更新として、役割別プラグイン、Sites preview、annotations の改善を発表した。これを単なる開発者向け拡張機能の追加として読むと、実務上の意味を取り逃がす。今回の焦点は、Codex がコードを書く補助だけでなく、調査、資料化、共有、レビュー、業務アプリ接続まで含む作業面へ進んだことにある。

この流れは、すでに扱った [OpenAI Codex Goalモード](/blog/openai-codex-goal-appshots-browser-2026/) と連続している。Goal mode は長い作業の完了条件を Codex に持たせる更新だった。さらに [Codex Windows対応](/blog/openai-codex-windows-profiles-usage-2026/) は、企業端末と GUI 操作をどう渡すかを示した。今回の役割別プラグインと Sites は、その作業結果をチームや業務部門へ広げる更新として読むべきだ。

## 事実: 2026年6月2日のCodex更新で何が変わったか

OpenAI の発表は、Codex をソフトウェア開発者だけでなく、データ分析、クリエイティブ制作、プロダクトデザイン、営業、投資調査、投資銀行業務のような知識労働にも使える作業エージェントとして位置付けている。発表では、六つの役割別プラグイン、Codex が作った成果物を共有しやすくする Sites preview、そしてコンテキスト内でコメントや修正意図を伝える annotations が中心に置かれている。

役割別プラグインは、従来の「必要な連携を個別に入れる」発想から一歩進む。たとえば営業、データ分析、クリエイティブ制作、プロダクトデザイン、投資調査のような業務では、使うデータ、参照する文書、承認者、出力物が違う。OpenAI は、こうした役割ごとのツール接続やワークフローを Codex 側へまとめ、利用者がゼロから構成しなくても業務文脈を持たせられる方向を示した。

Sites preview は、Codex が作った情報やページを、コードリポジトリの中だけでなく共有可能な成果物として扱う入口になる。OpenAI の説明では、Codex がページ、ダッシュボード、プロトタイプ、説明資料のような形で作業結果を見せ、関係者がそれに対してコメントし、再度 Codex に修正させる流れが意識されている。

annotations も重要だ。これは [Codexモバイル遠隔操作](/blog/openai-codex-mobile-remote-access-2026/) で見た承認待ちの短縮と近い。人間が「この部分を変えてほしい」「この数字を確認してほしい」と対象を指し示し、Codex がその文脈を作業へ戻す。テキストで長く説明するより、対象物に直接コメントできるほど、AIエージェントの再作業は現実的になる。

## 事実: プラグインは「接続」から「業務パッケージ」へ進む

今回の更新で一番大きいのは、プラグインが単なる外部サービス接続ではなく、役割別の作業パッケージになり始めた点だ。開発者向けのプラグインなら、GitHub、Figma、Slack、Notion、Google Drive のような接続先を増やすだけでも価値がある。しかし業務部門で使う場合は、接続先の数より、どの役割にどの権限で何をさせるかが重要になる。

日本企業ではここが特に重い。営業資料、稟議、投資調査、請求、顧客サポート、社内FAQは、部署ごとに権限と監査の扱いが違う。Codex がロール別プラグインを通じて複数の業務アプリに触るなら、管理者は「便利な連携を入れる」だけでは足りない。誰がどの plugin action を使えるか、どのデータを読み取れるか、どの成果物を外部共有できるかを決める必要がある。

これは [Codex利用枠更新](/blog/openai-codex-plan-credits-limits-2026/) の論点ともつながる。Codex は ChatGPT plan、workspace identity、credits、data controls、connected services の中で運用される製品になっている。役割別プラグインが広がるほど、導入判断は「開発者に配るか」ではなく、「業務部門ごとにどのワークフローを許すか」へ移る。

## 事実: Sites previewとannotationsは共有物の反復を変える

Sites preview は、Codex の出力を「会話の返答」ではなく「チームが見て直せる共有物」に近づける。これはプロダクトチーム、事業企画、マーケティング、カスタマーサクセスで効きやすい。たとえば、顧客向けFAQの下書き、営業資料の比較表、社内ポリシーの説明ページ、キャンペーンの計画ページ、データ集計の簡易ダッシュボードを Codex が作り、関係者が見ながら修正指示を返す流れが考えられる。

ただし、共有物になるほど責任も明確にしなければならない。Codex が作ったページが正しそうに見えても、数字、法務表現、顧客名、価格、個人情報、未公開情報が含まれるなら、人間のレビューは必須だ。特に日本企業では、社外向け資料と社内検討資料の境界、委託先との共有範囲、個人情報保護、下請法や金融規制などの業界ルールが関係する。

annotations はこのレビュー過程を短くする一方で、責任を消すものではない。コメントを付ける人、承認する人、Codex に再作業させる人、最終公開する人を分けておかないと、便利な反復がそのまま責任の曖昧さになる。AI が修正したから正しいのではなく、誰が確認したかを残す必要がある。

## 分析: 日本企業では開発者ツールではなく業務AI基盤として扱う

ここからは分析だ。

OpenAI は Codex を、開発者だけの道具から、仕事の進め方を変える作業エージェントへ広げようとしている。役割別プラグイン、Sites、annotations は、そのための部品だ。重要なのは、どれも「AIが賢く答える」機能ではなく、人間の業務プロセスの中に AI を入れるための UI と接続設計である。

日本企業が見るべき最初の論点は、部門別の導入単位だ。全社一斉に Codex のプラグインを開放するより、まずは対象部門を絞るほうが現実的である。開発部門なら仕様書、Issue、Figma、リポジトリ、ローカル確認。営業企画なら提案資料、製品FAQ、価格表、CRM。データ分析なら Snowflake や Tableau のような分析基盤、クリエイティブなら Figma や Canva のような制作面が候補になる。ただし人事や法務のように機微情報が多い部門へ将来広げる場合は、最初から広く接続させるべきではない。

二つ目は、AIが作る共有物の分類だ。Sites preview のような機能が入ると、成果物は会話の中に留まらない。URL、ページ、ダッシュボード、資料として人に見せられる形になる。すると、レビュー前の下書きなのか、社内限定の説明なのか、顧客に送ってよいものなのかを区別する必要がある。

三つ目は、プラグインの標準化だ。個人が好きな連携を入れる状態では、便利さは出ても統制は弱い。役割別に必要なプラグインを標準化し、不要な plugin action は無効化し、管理者が変更履歴を見られるようにする。これは情シスだけでなく、業務オーナー、法務、セキュリティ、現場リーダーが関わる設計になる。

## 導入前に決めるチェックリスト

第一に、役割別プラグインの利用範囲を決める。部門、職種、対象業務、接続先、読み取り権限、書き込み権限を分ける。最初は読み取り中心にし、外部送信や公開に関わる操作は人間承認を必須にする。

第二に、Sites preview の成果物分類を作る。下書き、社内レビュー用、顧客共有可、公開不可のようなラベルを決め、Codex が作ったページを誰が確認し、どこへ置き、いつ削除するかを明確にする。

第三に、annotations のレビュー責任を決める。コメントを付けた人が承認者なのか、修正依頼者なのか、単なるレビュアーなのかを分ける。AI がコメントを反映した後、誰が最終確認するかを残す。

第四に、プラグインと credits を一緒に見る。業務部門へ広げると、開発者だけの利用より利用量の予測が難しくなる。部門別の利用枠、用途別の上限、追加利用の承認、教育用のサンドボックスを設計する。

第五に、接続先のデータ分類を棚卸しする。Google Drive、Notion、Slack、CRM、HR、財務、契約管理に触れるなら、顧客情報、個人情報、未公開価格、契約条件、認証情報が混ざる。Codex に渡してよい範囲を、システム単位ではなくデータ種類単位で決めるべきだ。

## まとめ

OpenAI Codex の 2026年6月2日更新は、開発者向け新機能というより、Codex を業務AI基盤へ広げるための更新である。役割別プラグインは、業務ごとの接続と作業型をまとめる。Sites preview は、AIの出力を共有物へ変える。annotations は、人間のレビューと再作業の往復を短くする。

日本企業にとって重要なのは、すぐに全社展開することではない。まずは、部門別に安全なワークフローを選び、プラグインの権限、Sites の共有範囲、annotations の承認責任を決めることだ。Codex が仕事の入口に入るほど、便利さではなく、誰が何を見せ、誰が直し、誰が承認したかが導入品質を決める。
