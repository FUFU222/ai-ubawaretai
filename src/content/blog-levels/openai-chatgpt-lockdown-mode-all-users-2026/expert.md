OpenAI が ChatGPT Release Notes の 2026年6月4日更新で Lockdown Mode をすべての logged-in users に広げたことは、セキュリティ担当者にとって地味だが重要な変更である。ChatGPT は単なる文章生成ツールから、Web検索、Apps、Connectors、ファイル生成、データ分析、画像生成、Deep Research、Canvas のような外部接続を伴う作業面へ広がっている。そのぶん、外部コンテンツに含まれる悪意ある指示が、会話内のデータを別経路へ流すリスクも現実的になる。

この更新は [OpenAI新セキュリティ設定](/blog/openai-advanced-account-security-codex-2026/) のような認証強化とは違う。Advanced Account Security はアカウントの入口を強くする。Lockdown Mode は、ログイン後の ChatGPT が外部接続や高度機能を使う範囲を狭める。さらに [ChatGPTセッション管理](/blog/openai-chatgpt-active-sessions-codex-2026/) は残存ログイン、[OpenAI Skills統制](/blog/openai-chatgpt-skills-governance-compliance-2026/) は業務部品の権限と監査、[OpenAIのOffline検索](/blog/openai-offline-web-search-chatgpt-workspaces-2026/) は検索経路の扱いをそれぞれ扱う。これらを一つの「セキュリティ設定」として雑にまとめると、運用設計を誤る。

## 事実: Lockdown Modeの目的

OpenAI の Help Center は、Lockdown Mode を prompt injection による data exfiltration のリスクを減らすための設定として説明している。プロンプトインジェクションは、モデルの能力が上がるほど深刻になる。理由は、AI が外部ページを読み、接続アプリに触れ、ファイルを作り、分析を実行し、複数のツールを組み合わせるほど、外部コンテンツ内の指示が「読むべき情報」なのか「従うべき命令」なのかを誤る余地が増えるからである。

Lockdown Mode の実務上の価値は、AI の判断を完全に信頼するのではなく、そもそも危険な経路を閉じる点にある。外部送信や外部接続を伴う機能を絞れば、悪意ある指示が成功しても、実際にデータを持ち出せる経路が減る。これは DLP のような後段検知とは異なる。会話の実行環境そのものを制限する前段防御である。

ただし、OpenAI の説明が示すように、Lockdown Mode では Web検索、画像生成、ファイル生成、データ分析、Canvas、Deep Research、Apps、Connectors などに影響が出る。セキュリティの強化は、作業能力の低下と引き換えになる。したがって、企業は「安全だから常時オン」と単純化するのではなく、作業分類ごとに使い分ける必要がある。

## 事実: 全ログインユーザー展開の意味

2026年6月4日の release notes で重要なのは、Lockdown Mode が all logged-in users に利用可能になった点である。これは Enterprise 管理者だけの話ではない。個人Plus、個人Pro、Business、Enterprise、Edu、PoC用アカウント、外部委託先の個人アカウントなど、正式統制の外側に残りやすい利用者にも説明が必要になる。

日本企業ではこの境界が特に重要だ。正式なEnterpriseワークスペースではSSO、管理者設定、監査ログ、接続制御を設計できる。しかし実際には、部署単位の先行利用、個人契約、業務委託先の検証利用、採用候補者とのやり取り、開発者の個人アカウントによるPoCが残る。Lockdown Mode が広く使えるなら、企業は「正式環境の管理者だけが知っていればよい」ではなく、利用者本人に具体的な使い分けを求められる。

この意味では、Lockdown Mode は中央統制だけでなく、利用者教育のテーマでもある。社内AIガイドラインには、機密情報を含む会話では Lockdown Mode を使う、公開情報の調査と機密整理を同じ会話に混ぜない、接続アプリを使う会話では入力情報を制限する、といった具体的な手順を入れるべきである。

## 分析: 脅威モデルは「入力」ではなく「接続面」で考える

ここからは分析だ。

従来の生成AI利用ルールは、「機密情報を入力しない」「個人情報を入れない」という入力禁止型になりがちだった。しかし ChatGPT が外部アプリやWeb検索に広がると、リスクは入力だけでは説明できない。重要なのは、会話がどの接続面を持っているかである。

たとえば、同じ顧客名を含む相談でも、外部検索なしの文章整理と、Web検索、Drive接続、CRM接続、ファイル生成を含む作業ではリスクが違う。前者は主に会話内データの保持とモデル利用の問題であり、後者は外部コンテンツからのプロンプトインジェクション、接続先権限、意図しない書き込みや送信の問題も含む。

Lockdown Mode は、この接続面を狭めるための設定として理解するべきだ。入力禁止の代替ではない。高感度情報を扱うなら、まず入力してよい情報を限定する。そのうえで、外部接続を伴う機能を必要最小限にする。さらに、Apps や Connectors の管理者設定、利用ログ、DLP、接続先側の権限を組み合わせる。Lockdown Mode はこの中の一層である。

## 分析: 高リスク業務の初期適用パターン

日本企業で最初に適用候補になるのは、機密性が高く、外部検索を必要としない業務である。たとえば、社外秘文書の要約、契約条項の社内論点整理、セキュリティ事故の初動メモ、人事評価の文章整形、未公開の価格・製品計画のレビュー、顧客別の問い合わせ対応方針である。

これらの業務では、ChatGPT に求める価値は外部情報の取得ではなく、入力された文脈を整理し、論点を分け、表現を整えることにある。したがって、Web検索やAppsを止めても大きな支障が出にくい。むしろ、外部接続を止めることで「この会話は内部文脈だけで処理する」という説明がしやすくなる。

一方、公開情報を集める市場調査、競合調査、技術調査、最新仕様の確認、画像生成、ファイル出力、データ分析を伴う作業では、Lockdown Mode が作業を詰まらせる可能性がある。この場合は、作業を二段階に分けるのが現実的だ。第一段階では通常モードで公開情報を集める。第二段階では Lockdown Mode の別会話で、自社の機密情報を含む評価や意思決定メモを作る。

## 分析: 管理者設定と利用者設定を混同しない

Lockdown Mode が利用者に提供されることと、管理者が組織全体の接続面を設計することは別である。Enterprise 管理者は workspace settings、Apps、Connectors、Data controls、Retention、Compliance Logs などを組み合わせて、会社として許可する利用範囲を決める必要がある。

利用者本人が Lockdown Mode をオンにできるとしても、接続先の許可、外部アプリの OAuth、Drive や Slack 側の権限、共有フォルダ構成、外部委託先のアクセス範囲は管理者側で制御しなければならない。利用者設定は最終防御ではなく、業務場面に応じた追加防御である。

ここは [OpenAI Skills統制](/blog/openai-chatgpt-skills-governance-compliance-2026/) と同じ構図だ。ユーザーが便利な機能を使えるようになるほど、組織側は role、owner、アクセス、ログ、削除、レビュー期限を管理する必要がある。Lockdown Mode も、単にユーザーへ「オンにしてください」と言うだけでは運用にならない。どの業務で必須か、誰が例外を承認するか、制限で作業できない場合にどう切り替えるかまで決める必要がある。

## 分析: CodexとMCPは別の権限境界を持つ

開発組織では、Lockdown Mode を ChatGPT の一設定として閉じず、Codex や MCP と分けて説明することが重要である。Codex はコード、リポジトリ、端末、ブラウザ確認、プラグイン、MCP サーバー、環境変数、Git 認証情報と結びつく。ChatGPT の Lockdown Mode を有効化しても、Codex CLI の認証情報、ローカルMCP設定、接続済みGitHubリポジトリ、CIトークン、クラウド認証情報は自動で制限されない。

したがって、開発チームのAI利用ガイドでは、ChatGPTで機密相談をする場面、Codexにコード作業を任せる場面、MCPで社内ツールに接続する場面を分けるべきである。ChatGPT Lockdown Mode は、会話上の外部接続制限である。Codex の安全運用には、リポジトリ権限、ブランチ保護、レビュー必須化、Secrets管理、ローカル環境の分離、MCPサーバーの allowlist、監査ログが必要になる。

特に日本企業では、開発者が個人アカウントや小規模PoCで先にAIコーディングを試し、その後に正式導入へ移ることが多い。この移行期に「ChatGPTでは Lockdown Mode があるから安全」と説明すると危険である。ChatGPT と Codex は同じOpenAIアカウント文脈を持つことがあっても、実行面のリスクは違う。

## 実務: セキュリティチームが決めるべき項目

第一に、Lockdown Mode を必須にする情報分類を決める。個人情報、顧客別情報、未公開財務、未公開製品計画、契約交渉、脆弱性情報、インシデント対応、人事労務、医療・金融・公共系の機微情報を含む場合は、通常モードで外部接続を伴う作業を禁止または承認制にする。

第二に、作業を分離する手順を作る。公開情報の調査、社内情報の評価、成果物の生成、外部共有を一つの会話で混ぜない。公開情報調査は通常モード、機密評価は Lockdown Mode、外部共有前は人間レビュー、という流れにする。

第三に、Apps と Connectors の管理を別紙で持つ。接続先、許可部署、読み取り・書き込み、外部共有、ログ確認者、停止条件を一覧化する。Lockdown Mode があるから接続先管理を省略してよい、という判断は避ける。

第四に、事故報告のテンプレートを用意する。利用者が外部ページ、接続アプリ、ファイル、機密情報を同じ会話で扱った場合、どの日時、どの会話、どの接続先、どの情報分類、Lockdown Mode の状態、出力の異常有無を報告するかを決める。

第五に、教育資料を短くする。利用者には「外部接続を止める安全側モード」「機密情報を扱うときの別スレッド」「調査と機密整理を混ぜない」という3点を先に伝える。詳細な脅威モデルは管理者向け資料に置けばよい。

## まとめ

ChatGPT Lockdown Mode の全ログインユーザー展開は、ChatGPT の業務利用が外部接続前提へ広がったことへの防御策として読むべきである。プロンプトインジェクションによるデータ流出は、単にユーザー教育だけでは抑えにくい。AI が外部コンテンツを読み、外部アプリに触れるほど、実行面の制限が必要になる。

ただし、Lockdown Mode は認証、セッション、メモリ、Apps、Codex、MCP、監査ログを置き換えない。日本企業が取るべき対応は、Lockdown Mode を高リスク業務の安全側設定として位置づけ、通常モードとの使い分け、接続先管理、開発環境の権限境界を分けて説明することである。今回の更新は、ChatGPT をより多くの業務へ入れるために、外部送信の管理線を明文化するタイミングを示している。
