OpenAI が 2026年6月4日に発表した ChatGPT の memory / dreaming 更新は、プロダクト体験の改善であると同時に、企業利用の統制設計を再点検させる更新である。発表の中心は、ChatGPT が過去の会話をより長い時間軸で参照し、利用者に合わせた支援をしやすくすることだ。まず米国の Plus / Pro 利用者から始まるため、日本の Enterprise 環境でただちに同じ挙動が前提になるわけではない。

しかし、設計論点は今から動かせる。ChatGPT が「その場の応答」から「利用者を継続的に覚えるアシスタント」へ進むなら、企業はメモリの許可範囲、削除、説明責任、管理者設定を先に整える必要がある。以前の [ChatGPTセッション管理](/blog/openai-chatgpt-active-sessions-codex-2026/) はアカウントと端末の入口管理だった。今回の論点は、ログイン後に ChatGPT が利用者の文脈をどう蓄積し、どこまで業務に使えるかである。

## 事実関係: dreaming、保存メモリ、会話履歴参照を分ける

OpenAI の発表は、dreaming を ChatGPT のメモリ改善として位置づけている。利用者が直接質問していない間にも、過去の会話から役立つ可能性がある文脈を整理し、より個人に合った助言へつなげる方向の更新である。これは、短いプロンプトに対して毎回ゼロから応答する体験とは違う。利用者の過去の関心、作業、好みを継続的に活用する体験に近い。

一方、OpenAI Help Center の Memory FAQ は、保存メモリと過去チャット参照を分けて説明している。保存メモリは、利用者が明示的に覚えてほしいと伝えた情報や、ChatGPT が今後の応答に役立つと判断した情報である。Reference chat history は、過去の会話から応答に使える文脈を参照する設定である。企業のリスク評価では、この二つを混同しないほうがよい。

保存メモリは、比較的説明しやすい。たとえば「この人は短い回答を好む」「普段はTypeScriptを使う」「社内向けの文章は丁寧語にする」といった情報である。会話履歴参照はより広い。過去のチャットに含まれた業務内容、関心、繰り返し出てくるプロジェクト名、相談パターンが応答に反映されうる。dreaming は、この広い長期文脈をより積極的に活用する方向として読むべきだ。

## 制御面: 削除、Temporary Chat、Enterprise設定

OpenAI は、利用者が ChatGPT に何を覚えているか聞けること、メモリを削除できること、設定でメモリを管理できることを説明している。Temporary Chat を使えば、通常のメモリへ残さずに会話する選択肢もある。個人利用では、このセルフサービス操作が重要になる。

企業利用では、さらに管理者設定が必要になる。Enterprise / Edu では、管理者がメモリ関連機能を制御できる。ここは [OpenAI Skills統制](/blog/openai-chatgpt-skills-governance-compliance-2026/) と同じ構造だ。便利なAI機能が増えるほど、個人ユーザーの操作だけではなく、ワークスペースとして何を許可するかを決める必要がある。

削除の運用も単純ではない。保存メモリを消すこと、チャット履歴を消すこと、モデル改善への利用設定を変えること、監査ログや保持ポリシーを扱うことは、同じ「消す」でも意味が違う。日本企業がユーザー向けガイドを作るなら、メモリ削除、チャット削除、Temporary Chat、管理者への問い合わせを分けて説明したほうがよい。

## リスク評価: 便利な個人化と不適切な蓄積

メモリの価値は明確だ。エンジニアなら、普段使う言語、レビュー観点、社内ドキュメントの形式、回答の好みを覚えることで作業効率が上がる。営業なら、提案書の文体や商談準備の進め方が安定する。経営企画なら、よく見る指標や資料の構成を覚えさせることで、下書きの質が上がる。

問題は、個人化に向く情報と、会社の正式な台帳に入れるべき情報が混ざることだ。顧客の契約条件、社員評価、候補者情報、未公開の財務情報、障害の詳細、アクセスキー、ソースコードの秘密情報は、ChatGPT のメモリに任せるべきではない。これらはCRM、HRIS、文書管理、チケット管理、Secrets管理で扱う情報である。

日本企業では、現場が便利な使い方を先に見つけ、統制が後追いになることが多い。ChatGPT メモリでも同じことが起きる可能性がある。利用者が「この顧客はこういう条件を好む」「この社員はこういう配慮が必要」といった文脈を覚えさせると、短期的には便利だが、情報管理としては不適切になりやすい。

## 内部統制: メモリをAI利用規程にどう入れるか

AI利用規程には、通常「入力してはいけない情報」「出力の確認」「著作権や個人情報の扱い」が書かれる。しかし、メモリ機能が強くなると、それだけでは足りない。入力してよい情報でも、長期メモリに残してよいとは限らないからだ。

規程では、少なくとも三つの区分を作るべきである。第一に、覚えさせてよい情報。回答の形式、利用者の役割、一般的な技術スタック、公開済みの社内用語などだ。第二に、通常チャットでは扱えてもメモリには残さない情報。特定顧客名、案件名、候補者名、障害番号などである。第三に、通常チャットにも入れてはいけない情報。秘密鍵、未公開重要情報、個人情報、機密契約などだ。

この区分は、[Offline検索](/blog/openai-offline-web-search-chatgpt-workspaces-2026/) で扱った外部情報の取得制御とも似ている。AIが外部を検索する場合は、どの情報源を使うかを管理する。AIが利用者を記憶する場合は、どの内部文脈を残すかを管理する。方向は逆だが、統制の考え方は同じである。

## 管理者・法務・現場の役割分担

情シスや管理者は、対象プラン、地域、ワークスペース設定、メモリの有効化範囲、ログや問い合わせ導線を確認する。特に、Plus / Pro の個人利用と Enterprise の管理対象利用が混在する会社では、同じ ChatGPT でも設定とデータ扱いが異なることを説明する必要がある。

法務とコンプライアンスは、メモリに残してはいけない情報カテゴリを定義する。ここでは、日本の個人情報保護、契約上の秘密保持、業界規制、労務情報の扱いを踏まえる必要がある。メモリは会話体験の一部に見えるが、実態としては利用者文脈の保持である。個人データや機密情報の扱いと無関係ではない。

現場部門は、業務別の実例を作る。営業なら「商談資料の文体は覚えてよいが、顧客別の未公開価格条件は覚えさせない」。開発なら「レビュー観点は覚えてよいが、秘密鍵や未公開脆弱性は入れない」。人事なら「社内研修の文体は覚えてよいが、個人評価や候補者情報は入れない」。抽象規程だけでは使われないため、具体例が必要だ。

## Codexや開発組織への波及

今回の主語は ChatGPT だが、開発組織にも影響する。Codex やコーディングエージェントを使うチームでは、AIが開発者の好み、リポジトリの文脈、レビュー観点を覚えることに価値がある。一方で、未公開仕様、脆弱性、認証情報、顧客データを長期的な個人文脈へ混ぜることは避けるべきだ。

ここでは、[OpenAI新セキュリティ設定](/blog/openai-advanced-account-security-codex-2026/) と [ChatGPTセッション管理](/blog/openai-chatgpt-active-sessions-codex-2026/) の考え方がそのまま効く。入口の認証、端末セッション、メモリ、ローカルCLI、APIキー、MCP接続は別々に棚卸しする必要がある。メモリを無効化しても、端末に残る認証情報が安全になるわけではない。逆に、セッションを切っても、ワークスペースのメモリ設定が適切とは限らない。

開発チームでは、メモリに残してよいのは「作業スタイル」と「公開または社内で共有可能な標準」に寄せるのが現実的だ。チームのテスト方針、コードレビューの一般観点、ドキュメントの形式などは候補になる。個別リポジトリの秘密、顧客固有のバグ、脆弱性の詳細、環境変数は候補にしない。

## 導入手順: まず小さく試す

日本企業が dreaming やメモリ改善を前提に準備するなら、いきなり全社展開しないほうがよい。まず、機密度が低く、効果を測りやすい業務で試す。社内FAQの文章作成、公開資料の要約、研修資料の下書き、開発者向け一般レビュー観点の調整などが向いている。

次に、利用者にメモリ操作を教える。何を覚えているか確認する、不要なメモリを削除する、Temporary Chat を使う、メモリ設定を確認する。この操作は、セキュリティ研修の一部として扱ってよい。メールの誤送信防止やパスワード管理と同じく、AI時代の基本操作になる。

最後に、月次または四半期で運用を見直す。利用者からの問い合わせ、誤って覚えさせた情報、削除依頼、業務別の便利な使い方、禁止入力の事例を集める。メモリ機能は静的な設定ではなく、プロダクト更新と現場利用に合わせて見直す運用対象である。

## まとめ

OpenAI の ChatGPT memory / dreaming 更新は、AIアシスタントが利用者をより深く理解する方向への重要な一歩である。個人利用では、より自然で継続的な支援につながる。企業利用では、個人化の価値と、業務情報の不適切な蓄積を分けて管理する必要がある。

日本企業にとっての実務ポイントは、メモリを禁止するか開放するかではない。保存してよい文脈、保存してはいけない文脈、削除とTemporary Chatの手順、管理者設定、業務別の例外を決めることだ。セッション、検索、Skills、メモリを一体のAI統制として扱える会社ほど、個人化AIを安全に活用しやすくなる。
