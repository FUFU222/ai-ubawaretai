OpenAI の 2026年6月2日発表は、Codex をソフトウェア開発支援から、部門横断の作業エージェントへ広げるための更新として読むべきだ。役割別プラグイン、Sites preview、annotations は、どれも単体では UI や連携の改善に見える。しかし組み合わせると、Codex が「コードを直すAI」から「業務アプリ、共有物、レビュー指示をまたいで作業するAI」へ進んでいることが分かる。

このシリーズでは、[OpenAI Codex Goalモード](/blog/openai-codex-goal-appshots-browser-2026/) で長時間作業と視覚文脈を、[Codex Windows対応](/blog/openai-codex-windows-profiles-usage-2026/) で企業端末と GUI 操作を、[Codexモバイル遠隔操作](/blog/openai-codex-mobile-remote-access-2026/) で離席中の承認導線を扱ってきた。今回の更新は、その上に「業務部門ごとの接続と成果物共有」を重ねるものだ。

## 事実: Codexは役割、ツール、ワークフローへ広がる

OpenAI は今回の発表で、Codex を every role, tool, and workflow に広げると説明している。発表の焦点は、職種や業務に合わせてキュレーションされた六つのプラグイン、Codex が生成した成果物を共有しやすくする Sites preview、そして対象物に対して文脈付きの指示を返す annotations である。

重要なのは、OpenAI が Codex を開発者だけの作業面として語っていない点だ。発表では、データ分析、クリエイティブ制作、プロダクトデザイン、営業、公開株投資、投資銀行業務のようなナレッジワークが意識されている。つまり Codex は、リポジトリを読み、コードを編集し、PR を出すエージェントであると同時に、文書、社内ナレッジ、業務アプリ、共有ページをまたいで仕事を進めるエージェントへ拡張されている。

この方向性は、OpenAI Developers の Codex release notes とも整合する。Codex はすでに Appshots、Goal mode、in-app browser annotations、Computer Use、remote control、Profiles、usage statistics のように、作業の継続、画面文脈、端末操作、利用量可視化を積み上げてきた。今回の発表は、それらを部門横断の業務ワークフローへ接続する位置付けに見える。

## 分析: 役割別プラグインはIDと業務境界の問題である

ここからは分析だ。

役割別プラグインは、単に「便利な連携が増えた」という話ではない。企業導入では、役割とは権限、データ分類、承認経路、出力物の責任範囲を意味する。営業担当、データ分析担当、クリエイティブ担当、投資調査担当、エンジニア、PdM、CS 担当は、同じ AI エージェントを使っていても、許される操作が違う。

日本企業で特に問題になるのは、職務分掌と実際の SaaS 権限が一致しないケースだ。Google Drive、Slack、Notion、CRM、HR、会計、契約管理の権限は、現場の便宜で広めに付いていることがある。人間なら暗黙のルールで見ない資料を、AIエージェントは検索結果やプラグイン経由で拾う可能性がある。したがって、Codex の role-based plugins は、ID管理、グループ、データ分類、plugin action の許可範囲とセットで設計する必要がある。

この論点は [Codex利用枠更新](/blog/openai-codex-plan-credits-limits-2026/) ともつながる。Codex は ChatGPT plan、workspace identity、credit pool、connected services、data controls の上で運用される。役割別プラグインが入ると、費用と権限の両方が部門別に動く。開発部門の長時間エージェント利用と、営業企画の資料生成、データ分析のレポート作成、投資調査の比較資料を同じ上限と同じ監査で扱うのは粗い。

## 分析: Sites previewは「AIの出力」を統制対象に変える

Sites preview の意味は、Codex の出力が会話ログから共有可能なページへ移ることだ。これは大きい。チャットの返答なら、その場の利用者が読むだけで済む。しかし page、dashboard、prototype、brief、landing draft、policy explainer のような形になると、他者が参照し、転送し、意思決定に使う可能性が出る。

日本企業では、生成物の状態管理が必要になる。下書きなのか、レビュー済みなのか、社内限定なのか、顧客共有可能なのか、公開可能なのか。Codex が作った Sites をそのまま社外共有すると、誤った価格、未確認の法務表現、古い仕様、個人情報、機密資料の断片が混じるリスクがある。

つまり Sites preview は、生成AIの出力を単なる「回答」から「情報資産」に変える。情報資産になれば、保存期間、アクセス権、レビュー履歴、削除、監査、再利用のルールが必要になる。社内ポータル、ナレッジベース、営業資料、顧客向けFAQ、社内教育ページに Codex の出力を使うなら、この分類を先に決めるべきだ。

## 分析: annotationsはレビュープロセスを短くするが、責任は残る

annotations は、AIエージェントとのやり取りを「文章で説明する」から「対象に直接指示する」へ近づける。フロントエンド QA であれば、画面の余白、文字、色、ラベル、導線を直接指摘できる。資料作成であれば、表の項目、言い回し、数字、注記、見出しの位置を指定しやすい。業務ページであれば、関係者が同じ成果物を見ながら修正指示を返せる。

ただし、annotations はレビュー責任を自動化しない。むしろ責任の線引きを明示する必要が増える。コメントを書いた人はレビュアーなのか、承認者なのか、単なる依頼者なのか。Codex がコメントを反映した後、誰が再確認するのか。最終的に公開する人は誰か。この区別がないと、AIが反映した変更が「誰かが見たはず」の状態で流れてしまう。

開発現場では、これは PR レビューと似ている。コメント、修正、再確認、承認、マージの順番がある。業務部門の AI 生成物にも同じような状態遷移が必要になる。Codex の annotations は、その状態遷移を短くできるが、状態遷移そのものを不要にはしない。

## 日本企業での導入設計

第一に、ロール設計を細かくしすぎない。最初から全職種ごとに完全なプラグインセットを作ると運用できない。まずは開発、営業企画、サポート、社内ナレッジ、データ分析のように、リスクと価値が読みやすい領域から始める。法務、人事、財務のような機微情報が濃い領域へ将来広げる場合は、読み取り対象と出力物をかなり絞ったほうがよい。

第二に、plugin action を読み取り、下書き作成、社内共有、外部送信、書き込みに分ける。AIエージェント導入で危ないのは、読み取りと書き込みを同じ粒度で許すことだ。最初は読み取りと下書き作成に限定し、外部送信や本番データ更新は人間承認に寄せる。

第三に、Sites の分類ラベルを作る。たとえば `draft`、`internal-review`、`approved-internal`、`external-ready`、`restricted` のような状態を決める。ラベルがないと、AI が作ったページが社内外でどう扱われるべきか分からない。

第四に、annotations のログと承認を分ける。すべてのコメントを重い承認にする必要はないが、最終公開や顧客共有に関わる変更は、誰が承認したかを残すべきだ。AI がどのコメントを反映したか、反映しなかったかも確認できるとよい。

第五に、credits と利用量をワークフロー単位で見る。業務部門へ Codex を広げると、費用の伸び方は開発部門と違う。資料生成、調査、再作業、ページ生成、プラグイン検索はそれぞれ消費が違う。個人別利用量だけでなく、業務フローごとの価値と失敗率を見るほうがよい。

第六に、社内テンプレートを用意する。Codex に「この資料を作って」ではなく、目的、対象読者、参照してよい情報、禁止情報、出力形式、レビュー者、公開可否を書く。役割別プラグインがあっても、良い依頼の型がなければ結果は安定しない。

## リスク: 業務AI化はシャドーIT化しやすい

今回の更新で警戒すべきリスクは、Codex が便利になりすぎて、個人や部門が独自に業務AI化を進めることだ。プラグインで接続し、Sites でページを作り、annotations で直し、関係者に共有する。この流れは速い。しかし速いほど、情報管理、監査、公開判断、データ保持、契約上の制約が後追いになりやすい。

日本企業では、生成AIの導入ルールが ChatGPT の会話利用を前提に作られていることがある。しかし Codex がプラグイン、Computer Use、Sites、annotations を持つと、会話利用のルールだけでは足りない。AI がどのアプリを使うか、どの画面を見るか、どの成果物を作るか、誰に共有するかを扱う必要がある。

一方で、過度に禁止すると現場は別のツールへ流れる。したがって、実務的な落としどころは、禁止ではなく標準化だ。許可されたプラグイン、許可されたデータ範囲、許可された Sites の用途、承認が必要な操作、禁止する接続先を明文化する。これにより、現場は安心して試せる。

## まとめ

OpenAI Codex の 2026年6月2日更新は、Codex が企業の業務AI基盤へ向かう節目である。役割別プラグインは、外部接続を職務別ワークフローに変える。Sites preview は、AI出力を共有可能な情報資産へ変える。annotations は、人間のレビューとAIの再作業を同じ成果物上で結ぶ。

日本企業が見るべきなのは、新機能の便利さだけではない。部門別ロール、plugin action、データ分類、Sites の共有範囲、annotations の承認責任、credits の管理をひとつの導入設計として扱うべきだ。Codex が仕事の中心へ近づくほど、成功条件はモデル性能ではなく、業務境界と責任境界をどれだけ先に設計できるかになる。
