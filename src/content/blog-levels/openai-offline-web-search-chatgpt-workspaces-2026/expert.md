OpenAI Help Centerに追加・更新されている **Offline web search for ChatGPT workspaces** は、生成AIの企業導入で見落とされがちな論点をかなり正面から扱っている。論点はモデル性能ではない。**ChatGPTが外部情報を取りに行く経路を、管理対象ワークスペースでどう制御するか** である。

通常のAI検索の議論では、「最新情報が取れるか」「出典が付くか」「検索結果の質が高いか」が前に出る。しかし企業導入では、それだけでは足りない。検索クエリそのものが機密になる場合がある。たとえば、未発表の提携先、買収候補、訴訟相手、脆弱性調査対象、退職者対応、医療・教育・公共領域の相談内容は、調べていること自体が意味を持つ。

Offline web searchは、その問題に対して、ライブWeb検索を常に許可するのではなく、OpenAIのインデックス済み・キャッシュ済みWebコンテンツへ寄せる設定として説明されている。これは[OpenAI新セキュリティ設定でChatGPTとCodex運用はどう変わるか](/blog/openai-advanced-account-security-codex-2026/)で扱った認証強化と同じく、OpenAIが企業向けの統制面を厚くしている流れの一部だ。ただし、便利な安全機能として単純に評価すると危ない。これは検索能力を増やす機能ではなく、検索のtrust boundaryを狭める機能である。

## 事実: OpenAIが示した検索経路の違い

まず事実を切り分ける。

OpenAIのOffline web search説明では、この機能は対象となるChatGPTワークスペース向けのWeb検索設定だとされている。ChatGPTがライブ検索を行う代わりに、OpenAIがすでにインデックスまたはキャッシュしたWebコンテンツを使う。対象は一部のEnterprise、Edu、Healthcare、Teachers、regulated、federal workspace configurationsで、利用可否はプラン、契約、ワークスペース構成、管理者設定に依存する。

ここで重要なのは、OpenAIがこれをAPI Platformの話ではないと明示している点だ。つまり、Responses APIや独自RAG基盤の検索仕様が変わったのではない。ChatGPTという管理対象アプリケーションの中で、Web search behaviorをどう制限するかの話である。

通常のChatGPT Search for Enterprise and Eduでは、ChatGPTが必要に応じて検索クエリを作り、Web結果や専門データプロバイダを使って回答する。OpenAIの説明では、Enterprise/Eduワークスペースでは、プロンプトに基づく検索クエリがBingへ送られる場合がある。ただし、そのデータはユーザーアカウントから切り離され、ユーザーID、アカウントID、デバイスID、セッションID、IPアドレスは付かない。IPアドレスから得た大まかな位置情報が精度向上のために共有される場合はあるが、IPアドレス自体は共有されないと説明されている。

Offline web searchは、ここでライブ外部検索の依存を下げる。検索時点でライブ外部検索プロバイダへ問い合わせるのではなく、OpenAIのインデックスやキャッシュにある範囲を使う。この設計は、OpenAIが言うとおり、より厳しいガバナンス、コンプライアンス、データ取り扱い要件を持つ組織向けだ。

## 事実: 鮮度、カバレッジ、監査証跡は弱くなる

Offline web searchの代償は明確だ。OpenAIは、安定した一般Web調査には向くが、リアルタイム性、特定URLの取得保証、監査グレードの証跡が必要なワークフローには向かないと説明している。

この制約は実務上かなり大きい。特定URLを根拠にしたい場合でも、そのページがOpenAIのインデックスまたはキャッシュに存在しなければ取得できない。ページが新しい、人気が低い、ログインが必要、JavaScriptに依存している、CDNやbot対策が強い、robots.txtでクロールが制限されている、サイト構造が深いといった理由で欠落する可能性がある。

また、いつキャッシュされたかを常に確認できるわけではない。そのため、特定時点のページ内容を証明する業務には使いにくい。これは法務、監査、セキュリティ、調達では致命的な場合がある。たとえば、あるクラウドベンダーの価格表が5月4日時点でどうだったか、ある規制当局ページが提出締切前にどう書かれていたか、ある脆弱性情報が公開時にどの影響範囲を示していたかを確認するなら、Offline web searchの回答だけでは足りない。

OpenAIは、特定資料が重要なら、資料をアップロードする、必要箇所を貼り付ける、別URLを提供する、許可されている場合はライブ検索を使う、といった代替を案内している。これは、AI検索を証跡システムと混同するな、という実務上の警告として読むべきだ。

## 考察: これは検索機能ではなくデータ流出面の制御である

ここからは考察だ。

Offline web searchの本質は、検索品質の向上ではない。むしろ、検索品質や鮮度を一部犠牲にしてでも、ライブ外部検索というデータ流出面を狭めることにある。生成AIの企業導入では、入力データの漏えいばかりが注目されるが、検索クエリもまたデータである。ユーザーが「A社との契約解除リスクを調べて」「B製品の脆弱性悪用事例を調べて」「C候補者の訴訟履歴を調べて」と入力した時点で、検索語はかなり濃い業務文脈を含む。

通常のEnterprise/Edu向けChatGPT Searchでは、OpenAIは検索クエリがユーザーやアカウントから切り離されると説明している。これは重要な保護だが、外部検索プロバイダへのライブ問い合わせが発生し得ること自体を嫌う組織もある。金融、医療、教育、公共、重要インフラ、知財部門、M&A部門では、その感覚は自然だ。

Offline web searchは、そうした部門に対して「検索を完全に止める」以外の選択肢を与える。これは日本企業にとって価値がある。日本の大企業では、便利な機能を全面禁止するより、リスク部門だけ制限したいことが多い。全社一律でChatGPT Searchを止めると現場の生産性は落ちる。しかし全社一律でライブ検索を許すと、法務・監査説明が難しくなる。ロール単位の権限管理とOffline web searchを組み合わせると、その中間を設計しやすい。

## Lockdown Modeとの関係を誤解しない

Offline web searchは、Lockdown Modeとも関係する。OpenAIのLockdown Mode説明では、ライブWeb browsingはキャッシュ済みコンテンツに制限される。さらにDeep Research、Agent Mode、Canvas networking、ChatGPTからのファイルダウンロードなども制限される。目的は、プロンプトインジェクションによるデータ外部流出を、ネットワークアクセスの制限によって決定的に抑えることだ。

ただし、Lockdown Modeを万能防御として扱ってはいけない。OpenAIは、Lockdown Modeがプロンプトインジェクションを文脈へ入る前に防ぐものではないと説明している。キャッシュ済みコンテンツ自体に悪意ある指示が含まれる可能性はある。Lockdown Modeは、主に外部ネットワークリクエストを使った流出経路を制限する。つまり、攻撃を消すのではなく、攻撃後の送信経路を狭める設計だ。

さらに、Lockdown ModeはCodexのネットワークアクセスには影響しないと説明されている。ここは開発組織にとって重要だ。ChatGPT側の検索やツールを制限しても、Codexや別の開発者エージェントが同じ方針になるとは限らない。以前の[OpenAI新セキュリティ設定でChatGPTとCodex運用はどう変わるか](/blog/openai-advanced-account-security-codex-2026/)では、Advanced Account SecurityがChatGPTとCodexの認証面にまたがる点を扱ったが、Lockdown Modeは同じようには広がらない。

したがって、ChatGPTのOffline web search、Lockdown Mode、Codexのネットワーク制御、社内プロキシ、DLP、端末管理は別々に棚卸しする必要がある。1つのOpenAI設定で全AIツールの外部通信が制御されると考えるのは危険だ。

## 日本企業の設計パターン

日本企業で現実的なのは、検索設定を部署・用途別に分けることだ。

第1のパターンは、高リスク部署にOffline web searchまたはLockdown Modeを割り当てる設計である。役員室、法務、人事、監査、医療・教育・公共系部門、M&Aや知財に関わるチームが対象になる。ここでは、最新情報よりも、検索クエリや外部接続経路を抑える価値が高い。一般的な公開情報の下調べ、用語整理、安定した制度理解には使えるが、証跡が必要な確認は公式資料アップロードまたは別の承認済み調査手順に回す。

第2のパターンは、調査・マーケティング部門に通常のライブ検索を残す設計である。ニュース、競合発表、価格改定、SNS反応、イベント情報の把握では鮮度が重要だ。ここにOffline web searchを入れると、ユーザーは古い情報を新しいものとして扱うリスクがある。むしろ、ライブ検索を許可しつつ、出典確認、社外秘入力の禁止、調査結果の二次確認を教育したほうがよい。

第3のパターンは、開発者・セキュリティ担当に専用経路を作る設計である。脆弱性情報、API仕様、SDK変更、障害情報は最新性が命だ。Offline web searchは向かない。公式ドキュメントへの直接アクセス、社内ブラウザポリシー、承認済み検索、ソースアップロード、社内ナレッジベースを組み合わせるべきだ。この観点は、[OpenAIがAWSへ。BedrockでCodexとManaged Agents、日本企業は何を見るべきか](/blog/openai-aws-bedrock-codex-managed-agents-2026/)で扱ったクラウド統制にも通じる。AIの実行面だけでなく、根拠情報の取得面も統制対象になる。

第4のパターンは、社内ナレッジ検索とWeb検索を明確に分ける設計である。ユーザーが社内ファイルを接続している場合、ChatGPTは接続ソースを優先し、必要に応じてWeb検索を使う場合がある。ここで「社内情報だけで答えてほしい」業務と、「社外情報も混ぜてよい」業務を分けないと、回答の根拠が混ざる。Google Workspace側でも、[Google Workspace Intelligenceの管理者向け制御は日本企業のAI導入をどう変えるか](/blog/google-workspace-intelligence-admin-controls-2026/)で扱ったように、AIが社内データへ近づくほど権限設計が重要になる。OpenAIでも同じだ。

## ユーザー教育で必ず伝えるべきこと

Offline web searchを導入するなら、ユーザー教育が欠かせない。設定だけ変えても、ユーザーが挙動を理解していなければ誤用が起きる。

まず、「検索できないURLがある」と伝える必要がある。Offline web searchでは、特定URLがキャッシュにない場合、ChatGPTは取得できないか、取得できないと答える。ユーザーはそれを障害と誤解しやすい。重要な資料はアップロードまたは貼り付けるという運用をセットにするべきだ。

次に、「検索結果は古いことがある」と伝える必要がある。OpenAIは、ページごとの更新SLAを示していない。キャッシュが新しい場合もあるが、特定ページがいつ更新されたかは保証されない。最新情報を扱う業務では、ライブ検索または公式ページ確認へ切り替える基準を用意する。

さらに、「キャッシュ検索でも悪意ある指示は入り得る」と伝える必要がある。OpenAIは、Indexed or cached contentにも不正確な情報や悪意ある指示が含まれ得ると注意している。ユーザーは、出典を確認し、AIの回答をそのまま社内決定に使わない。特に接続アプリや書き込みアクションが有効な環境では、承認プロンプトを丁寧に読む必要がある。

最後に、「Offline web searchは監査証跡ではない」と伝えるべきだ。特定時点のページ内容を証明したいなら、アーカイブ、公式PDF、保存済みファイル、社内記録、承認済み証跡管理システムを使う。ChatGPTの検索回答は、調査の起点にはなっても、監査証拠そのものにはしにくい。

## Privacy FilterやAdvanced Account Securityと一緒に見る

Offline web search単体では、企業AIの安全運用は完成しない。むしろ、OpenAIがこの数週間で出している複数の管理機能を組み合わせて初めて意味が出る。

[OpenAIがPrivacy Filterを公開。日本企業は生成AI前段のPIIマスキングをどう内製化するか](/blog/openai-privacy-filter-pii-redaction-2026/)で扱ったPrivacy Filterは、AIへ送る前に個人情報や秘密情報を検出・マスキングする前処理部品だった。これは入力面の制御である。Advanced Account Securityは、アカウント侵害や復旧経路の弱さを抑える認証面の制御だった。Offline web searchは、外部情報取得面の制御である。

この3つを並べると、OpenAIの企業向けセキュリティ管理はかなり実務的な方向へ進んでいる。モデルの拒否設定だけに頼るのではなく、入力を減らし、本人確認を強め、検索・外部通信を制限し、必要な部署には強いモードを割り当てる。これは日本企業がAIガバナンスを作るときの基本形に近い。

一方で、まだ人間が決めるべきことも多い。どの部署を高リスク扱いにするのか。ライブ検索を誰に許すのか。特定URLが読めないときの代替手順は何か。キャッシュ検索で作った回答をどのレベルまで業務利用してよいのか。これらはOpenAIの設定だけでは決まらない。会社の業務分類、情報分類、監査要件、契約条件に合わせて決める必要がある。

## まとめ

Offline web searchは、ChatGPTの検索を便利にするための機能というより、企業ワークスペースで検索経路を管理するための機能だ。ライブ検索を完全に止めると不便だが、全員にライブ検索を許すにはリスク説明が重い。そこに、OpenAIのインデックス済み・キャッシュ済みWebコンテンツを使う中間の選択肢が入ってきた。

日本企業にとっての実務価値は、部署ごとに検索リスクを分けられる点にある。高リスク部署はOffline web searchやLockdown Modeへ寄せ、鮮度が必要な部署はライブ検索を残し、証跡が必要な業務は公式資料やアップロード資料を使う。この切り分けができれば、ChatGPTを単なる便利ツールではなく、管理された業務基盤として広げやすくなる。

ただし、Offline web searchは最新性、URL取得保証、監査証跡を犠牲にする。ここを理解しないまま導入すると、古い情報を信じたり、特定資料を読めない理由が分からなかったりする。管理者は、設定だけでなく、ユーザー教育、代替手順、利用ログ、接続アプリの権限まで一緒に設計する必要がある。
