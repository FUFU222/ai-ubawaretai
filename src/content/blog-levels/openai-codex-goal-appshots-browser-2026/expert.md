OpenAIの2026年5月21日のCodex更新は、個別機能だけを見ると地味です。しかし、Goal mode、Appshots、in-app browser annotations、locked computer useをまとめて読むと、Codexの設計思想がかなりはっきりします。OpenAIはCodexを、コード補完や一問一答の開発支援ではなく、長時間の作業を継続し、画面文脈を読み、人間の承認を受けながら前へ進む開発エージェントへ寄せています。

この文脈は、[OpenAI Codex Security](/blog/openai-codex-security-workflow-2026/)や[OpenAI CodexのDell連携](/blog/openai-dell-codex-hybrid-onprem-2026/)と同じ線上にあります。前者は脆弱性調査と修正案、後者はハイブリッド/オンプレミス環境との接続を扱いました。今回の更新は、その企業向け基盤化を日々の開発UIに落とすものです。[Codexのモバイル遠隔操作](/blog/openai-codex-mobile-remote-access-2026/)で見えた承認導線の拡張とも重なります。

## 事実: 更新の中心は作業継続性と視覚文脈

OpenAI Help Centerのリリースノートでは、2026年5月21日のCodex更新として、Appshots、Goal mode一般提供、in-app browser annotations、locked computer use、Browser use改善が示されています。ここでの共通点は、モデルそのものの性能ではありません。Codexが作業を理解し、継続し、検証し、人間とやり取りするための周辺UIと運用機能です。

Goal modeは、Codex app、IDE extension、CLIで使えるようになった長時間タスク向けの機能です。OpenAI Developersでは、Goal modeは多段階の作業や明確な完了定義が必要な場面に向くと説明されています。`/goal`で開始し、ゴール文が最初の指示であり、同時に完了基準にもなります。これはエージェントに「やること」だけでなく「終わり方」を与えるUIです。

Appshotsは、macOSのCodex appから前面ウィンドウをスレッドに送る機能です。画像としてのスクリーンショットに加え、アプリが外部に出せるテキストも含められます。OpenAIは、APIドキュメント、メール、カレンダー、画像編集画面、エラー、設定パネルなどを例に挙げています。つまり、Codexが読む文脈はリポジトリだけでなく、ユーザーが見ている業務画面にも広がります。

in-app browserは、Codexとユーザーが同じブラウザプレビューを見るための機能です。ローカル開発サーバー、ファイルベースのプレビュー、ログイン不要の公開ページに向いており、画面上の要素や領域にコメントできます。今回のリリースノートでは、ブラウザ注釈のスタイルフィードバックが強化されたことも示されています。OpenAI Developersの説明では、フォント、テキスト、余白、色などをより細かく指定できる。

Computer Use側では、locked computer useが加わっています。Macがロックされた後でも、条件を満たすComputer Useタスクを接続デバイスから継続できる機能です。ただし、OpenAIはこれを一般的なリモートアンロック手段ではないと明記しています。activeでtrustedな短いComputer Useターンだけを対象にし、それ以外は手動アンロックを求める設計です。

## 分析: Goal modeはプロンプトではなく作業契約である

ここからは分析です。

Goal modeを単なる長文プロンプト機能として見ると、価値を読み誤ります。重要なのは、ゴール文がCodexの作業中に参照され続ける完了条件になる点です。これは、開発組織にとっては「作業契約」に近い。

たとえば、リファクタリングを依頼するとします。悪いゴールは「このモジュールをきれいにして」です。良いゴールは、「公開APIを変えず、対象ディレクトリ内だけを変更し、既存テストと新規テストを通し、パフォーマンスに影響が出る変更は避け、判断が必要なら質問する」です。後者は、Codexに自律的に動く余地を与えながら、組織として守る境界も示しています。

この性質は、企業導入で重要になります。日本企業では、AIコーディングの失敗は「コードが少し違う」だけでは済みません。業務仕様、監査、委託先との責任分界、リリース手順、セキュリティ基準に触れることがあります。Goal modeを使うなら、チーム標準のゴールテンプレートを作るべきです。バグ修正、UI修正、テスト追加、脆弱性対応、ドキュメント更新、依存関係更新で、それぞれ完了条件が違うからです。

この点は[Codex LabsとSI連携](/blog/openai-codex-labs-enterprise-2026-04-21/)の論点ともつながります。企業がCodexを広げるなら、個人がうまいプロンプトを書く段階から、組織が再利用できる作業型を持つ段階へ移る必要があります。Goal modeは、その作業型をUIとして支える機能になり得ます。

## 分析: Appshotsは便利だが情報分類を要求する

Appshotsは、開発者体験としてはかなり強い機能です。Slackの会話、社内ツールの画面、Figmaや画像編集アプリ、エラー表示、APIドキュメントをそのままCodexに渡せれば、説明コストは下がります。特に、非エンジニアがCodexに文脈を渡す場面では効果が大きいでしょう。

しかし、企業利用では真っ先に情報分類の問題が出ます。Appshotは「画面を見せる」機能です。画面には顧客名、メールアドレス、契約金額、内部URL、アクセストークン、障害情報、未公開ロードマップが映り込むことがあります。さらに、アプリが提供するテキストも添付される場合があります。見た目には一枚のスクリーンショットでも、実際にはより多いテキスト文脈が渡る可能性があります。

したがって、Appshotsの運用ルールはスクリーンショット共有ポリシーに近い形で作るべきです。許可する画面、禁止する画面、マスキングが必要な項目、テストデータへの切り替え、顧客環境ではなく検証環境を使う原則を決める。これは情シスや法務だけの話ではありません。プロダクト開発チームが日常的に守れる簡単なルールに落とさないと、使われなくなるか、危ない使われ方になります。

OpenAIのドキュメントも、Appshotsは必要な場合に限り機密コンテンツで使い、スクリーンショットやドキュメント共有と同じようにレビューするべきだと説明しています。ここを軽く見ないほうがいいです。AIエージェントは、文脈を渡すほど賢くなりますが、同時に渡してはいけない情報も増えます。

## 分析: ブラウザ注釈はデザインQAの往復を短くする

in-app browser annotationsは、フロントエンド開発にかなり直接効きます。従来のAIコーディングでは、UIの違和感を言語化する必要がありました。「カードの余白が少し広い」「モバイルでラベルが詰まっている」「ツールチップがデータ点にかぶる」といった指摘は、スクリーンショットだけでも伝わりにくい。画面上の対象へコメントし、スタイル値まで指定できるなら、Codexは修正対象を絞りやすくなります。

これは日本のプロダクト組織で実務価値があります。デザイナー、PdM、QA、エンジニアの間で、画面レビューの往復が多いからです。特に管理画面、SaaS、業務アプリでは、派手なデザインより、密度、余白、折り返し、表の見やすさ、フォームエラー、モバイル時の崩れが品質に直結します。

ただし、in-app browserは万能ではありません。OpenAI Developersは、サインイン状態、通常ブラウザのプロファイル、Cookie、拡張機能、既存タブはサポートしないと説明しています。つまり、認証済みの本番管理画面をそのままCodexに触らせる用途ではありません。ローカル開発サーバー、未認証プレビュー、公開ページ、ファイルベースの検証に向いています。

ここで設計すべきなのは、QA環境の作り方です。ログイン不要の確認ルート、テストデータ、StorybookやプレビューURL、固定viewport、スクリーンショット差分、アクセシビリティ検査を組み合わせると、Codexに任せやすい。逆に、本番環境にしか状態がない、ログインが複雑、テストデータがない、画面ごとの期待値が言語化されていない場合、ブラウザ注釈だけでは解決しません。

## 分析: locked computer useは端末統制の機能である

locked computer useは、最も慎重に扱うべき更新です。離席中や移動中にCodexがデスクトップアプリを使い続けられるなら、長時間タスクの停止は減ります。モバイル遠隔操作と組み合わせれば、ユーザーは外出先から判断を返し、Codexは接続されたMac上で作業を続けられる。

しかし、これは端末統制の問題です。Computer Useは、ファイル編集やシェル実行とは違い、GUIアプリやシステム状態に触れます。OpenAIの説明では、Computer UseにはScreen RecordingとAccessibilityの権限が必要です。これらは、企業のMac管理では非常に重い権限です。画面を見られる、クリックできる、入力できるということは、対象アプリによってはソースコード以外の業務情報にも触れます。

locked computer useはさらに、画面ロック後の挙動を含みます。OpenAIは、Appleのauthorization plug-inを使い、activeでtrustedなComputer Useターンかどうかを確認し、短い範囲に限定すると説明しています。この設計は慎重ですが、導入側が何もしなくてよいという意味ではありません。

日本企業で試すなら、まず管理済みMac、少人数、低機密なアプリ、明確なアプリ許可リストから始めるべきです。MDM、画面録画権限、Accessibility権限、ロック時のネットワーク、ログ取得、紛失時のリモートワイプ、利用者教育を確認する。特に、個人所有端末やBYODで広げるのは避けたほうがよいでしょう。

## 実装チームが作るべき運用テンプレート

この更新を本当に使うなら、機能をオンにする前に運用テンプレートを作るべきです。

第一に、Goal templateです。バグ修正、UI修正、テスト追加、依存関係更新、ドキュメント更新、セキュリティ調査で、成功条件、禁止範囲、検証コマンド、レビュー観点を定義します。これはプロンプト集ではなく、作業のDefinition of Doneです。

第二に、Visual context policyです。Appshotsで共有してよい画面、マスキングが必要な画面、禁止画面を分類します。顧客データ、社内チャット、給与、契約、認証情報、管理者画面は原則避ける。必要な場合はテストデータや専用環境へ切り替えます。

第三に、Browser QA profileです。Codexに検証させるURL、viewport、テストアカウント不要の状態、Storybook、デザイン期待値、スクリーンショット保存場所を決めます。in-app browserでできることと、通常ブラウザや人間QAに残すことを分けます。

第四に、Computer Use approval policyです。どのアプリを許可するか、常時許可を認めるか、ロック後利用を認めるか、誰が設定できるか、どの作業では禁止するかを決めます。ここは情シス、セキュリティ、開発基盤チームの共同設計が必要です。

第五に、Audit and review pathです。CodexがGoal modeで完了したとしても、最終的な完了はPR、レビュー、テスト、必要ならセキュリティ確認を通過した時点です。Codexの作業ログ、入力したAppshots、ブラウザコメント、Computer Useの利用範囲を、どこまで追えるかも確認します。

## まとめ

今回のCodex更新は、OpenAIがCodexを企業の実作業へ近づけるうえで重要な一歩です。Goal modeは作業の完了条件を持たせ、AppshotsはMac上の視覚文脈を渡し、in-app browser annotationsはUI修正のフィードバックを具体化し、locked computer useは離席後の継続作業を可能にします。

ただし、これらはすべて統制とセットです。長時間作業を任せるなら、ゴール設計が必要です。画面を渡すなら、情報分類が必要です。ブラウザで検証するなら、検証環境が必要です。ロック後もComputer Useを動かすなら、端末管理と権限設計が必要です。

日本の開発組織にとって、最初の実験対象は明確です。ローカルWebアプリのUI修正、テスト追加、ドキュメント更新、低機密な不具合調査のように、成功条件を明文化しやすく、レビュー可能な差分に落ちる作業から始める。その上で、Codexを個人ツールではなく、開発基盤の一部として管理する準備を進めるべきです。
