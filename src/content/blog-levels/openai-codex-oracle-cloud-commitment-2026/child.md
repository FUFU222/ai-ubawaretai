OpenAI は 2026年6月10日、Oracle Cloud の既存契約を通じて OpenAI models と Codex を使いやすくする発表をしました。Oracle 側も、Oracle Cloud Marketplace と Oracle Universal Credits を使って、OpenAI models を調達できることを説明しています。

これは「新しいAIモデルが出た」という話ではありません。日本企業にとっては、OpenAI や Codex を買う方法が増える話です。すでに Oracle Cloud を使っている会社なら、まったく新しい契約を作るのではなく、既存のクラウド契約や予算の中で OpenAI 利用を考えられる可能性があります。

## 何が変わるのか

大きなポイントは、Oracle Cloud の既存コミットメントを OpenAI models や Codex の利用に使えるようにする経路が示されたことです。Oracle Cloud Marketplace から調達できれば、請求や購買の流れを Oracle Cloud 側に寄せられます。

会社では、新しいサービスを使うときに多くの確認が必要になります。契約先はどこか、支払いはどこから出るのか、誰が承認するのか、どの部署の予算にするのか、セキュリティ確認は済んでいるのか。Oracle Cloud をすでに使っている会社では、今回のような Marketplace 経由の選択肢があると、検討を始めやすくなります。

ただし、買いやすくなることと、安全に使えることは同じではありません。Codex に社内コードやドキュメントを見せるのか、API で顧客向け機能に使うのか、誰が利用量を管理するのかは、別に決める必要があります。

## なぜ日本企業に関係するのか

日本企業では、AIの導入が技術部門だけでは終わりません。購買部門、法務、情シス、セキュリティ、経理、現場の開発チームが関わります。特に大企業では、既存クラウド契約をどう使うかが重要です。

たとえば、すでに OCI を使っている会社では、OpenAI を個別に契約するより、Oracle Cloud の請求や予算管理に載せたほうが社内説明しやすい場合があります。逆に、既存の Oracle Cloud 予算を使うなら、ほかのクラウド利用と費用が競合するため、AI 利用の上限や配賦を決める必要があります。

## 気をつけること

まず、Oracle 経由で買うものと、OpenAI 側で管理するものを分けて考える必要があります。Oracle Cloud Marketplace は購買経路です。一方、Codex の利用者、API キー、モデル選択、利用上限、ログ、データの扱いは、OpenAI 側や自社の管理で決めることが多く残ります。

次に、Codex と API 利用を分けることも大切です。Codex は開発者がコードレビュー、テスト作成、修正作業などに使うものです。OpenAI models の API は、社内アプリや顧客向けサービスに組み込まれることがあります。どちらも OpenAI ですが、リスクと管理方法は違います。

最後に、既存のクラウド予算を使えるからといって、すぐ全社展開しないことです。まずは小さなチームで、どの用途に使うのか、どれくらい費用がかかるのか、どのデータを渡してよいのかを確認するほうが安全です。

## まとめ

今回の発表は、OpenAI models と Codex を企業が買いやすくする動きです。特に Oracle Cloud をすでに使っている日本企業にとっては、既存の契約や予算を活用できる可能性があります。

しかし、調達しやすくなるだけでは十分ではありません。Codex に何をさせるか、API でどのデータを扱うか、誰が費用を管理するか、どこにログを残すかを決める必要があります。今回のニュースは、OpenAI を使うかどうかだけでなく、AI利用の買い方と管理方法を見直すきっかけとして読むのがよいです。
