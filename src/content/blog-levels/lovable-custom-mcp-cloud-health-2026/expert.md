Lovableの2026年5月24日から25日にかけての更新は、AIアプリビルダーを業務利用するチームにとって見逃しにくい。Custom MCP serversが全プランで利用可能になり、Lovable Cloud database health checkも追加された。これは、生成AIでアプリを作る体験が、プロンプトとUI生成だけでなく、外部文脈の接続、クラウドDBの運用、エージェント間連携へ広がっていることを示す。

この更新を読むときは、「Lovableが無料プランにもMCPを開いた」という表面的な見方だけでは足りない。実務上の主題は、AIアプリビルダーが社内ツール、チケット、ドキュメント、分析データ、DB、外部AIエージェントとどうつながるかである。つながるほど便利になるが、同時に権限境界、データ分類、操作ログ、退職者対応、クレジット消費、障害初動を設計しなければならない。

この点は、[OpenAI Codexプラグイン戦略](/blog/openai-codex-plugins-platform-strategy-2026/)や[Google Gemini API Docs MCP](/blog/google-gemini-api-docs-mcp-agent-skills-2026/)で見た流れと同じだ。AIエージェントの価値はモデル単体ではなく、どの文脈を安全に渡せるか、どのツールを実行させるか、どの証跡を残すかで決まる。Lovableはその流れを、コードを書く人だけでなく、プロダクトマネージャー、デザイナー、業務部門が触るAIアプリ開発面へ持ち込んでいる。

## 事実: Changelogの二つの更新

Lovableの公式Changelogは、2026年5月25日付でCustom MCP serversが全プランで利用可能になり、有料プラン不要でConnectorsからChat connectorsへ追加できると説明している。5月24日付ではLovable Cloud database health checkが追加され、チャットからDB状態を診断できるようになった。

database health checkの説明では、接続数、メモリ、ディスク使用量、稼働時間などが要約対象として挙げられている。目的は、遅いクエリ、タイムアウト、接続数の増加、メモリ圧迫、ディスク不足、compute limitなど、よくあるDB問題の切り分けを早くすることだ。AIアプリビルダーで作ったアプリが本番に近づくほど、これは重要になる。

Custom MCP serversについては、LovableのChat connectorsドキュメントが実務的な位置づけを説明している。MCPはAIシステムが外部ツール、サービス、データソースへ接続するための標準であり、Lovableでは外部のチーム文脈をアプリ生成に取り込むために使われる。ドキュメントには、Notion、Linear、Jira、Miroのようなツールの文脈を使い、プロトタイプやフローを標準に沿って作る例が示されている。

ここでのポイントは、Lovableが単に外部情報を検索するのではなく、接続済みツールをアプリ生成の作業文脈として扱うことだ。チケットの受け入れ条件、Notionの仕様、Miroの図、内部APIの応答が入れば、AIが生成するアプリはチームの現実に近づく。一方、誤った接続、過剰な権限、古い仕様書、機密データが入れば、生成物もリスクを含む。

## 事実: 二種類のMCP経路を混同しない

Lovable周辺には、少なくとも二つのMCP経路がある。第一はChat connectorsである。これはLovable Agentが外部ツールへアクセスするための経路だ。社内のチケット管理、ドキュメント、データソースをLovableへ渡すために使う。

第二はLovable MCP serverである。これは研究プレビューとして提供され、外部のMCP対応AIクライアントからLovableを操作するための経路だ。ドキュメントでは、ワークスペースやプロジェクトの取得、ファイル一覧、ファイル読み取り、プロジェクト作成、Lovable Agentへのメッセージ送信、デプロイ、接続済みMCP serverの一覧や追加などが説明されている。

この違いは、セキュリティレビューで非常に重要である。Chat connectorsは「Lovableがどこへ行けるか」の問題だ。Lovable MCP serverは「誰がLovableを外から動かせるか」の問題である。前者はデータ持ち込みと外部ツール実行、後者はプロジェクト操作とデプロイの権限に直結する。

Lovable MCP serverのドキュメントは、接続スコープがユーザーのアカウント全体に及び、接続したクライアントがそのユーザーの権限でプロジェクトを読んだり編集したりできること、tool callが実際のクレジットを消費し実プロジェクトを編集すること、DB queryはフル権限で実行されることを注意点として示している。これは研究プレビューとして当然の注意だが、企業利用ではそのままリスク登録すべき内容である。

## 分析: AIアプリビルダーにもゼロトラストの発想が必要になる

ここからは分析である。

AIアプリビルダーは、従来の開発ツールよりも導入の入口が軽い。プロダクトマネージャーや業務部門が直接プロンプトを入れ、画面を作り、DBを持つアプリを動かせる。Custom MCP serversが全プランに開かれると、その軽さのまま社内ツール接続まで進みやすくなる。

これは強力だが、従来の開発統制を飛び越えやすい。通常の社内システム連携なら、API申請、OAuthアプリ登録、権限レビュー、ログ設計、テスト環境、本番環境、監査証跡を踏む。しかしAIアプリビルダーでは、個人が「ちょっと便利にするため」にMCP serverを追加し、社内チケットや顧客情報を文脈として使う可能性がある。

したがって、Lovableのようなツールにもゼロトラストの発想が必要になる。誰が接続するのか、何を読めるのか、何を書けるのか、いつ失効するのか、どのログで追えるのか、退職や異動でどう外すのかを、最初から前提にする。AIツールだから特別にゆるくするのではなく、AIツールだからこそ接続範囲を小さく始める。

[Cursor Cloud Agentの開発環境管理](/blog/cursor-cloud-agent-dev-environments-2026/)でも、AIエージェントを動かす環境、リポジトリ設定、秘密情報、監査ログが論点になった。Lovableでは対象がよりプロダクト試作や業務アプリに寄るが、構造は同じだ。AIが作業するなら、作業面を管理対象にする必要がある。

## 分析: database health checkはSRE代替ではなく初動圧縮

Lovable Cloud database health checkは、ローコード/AIアプリビルダーの運用上の弱点をよく突いている。小さく始めたアプリが使われ始めると、最初に出る問題はモデル性能ではなく、DB接続、クエリ、容量、権限、データ設計であることが多い。チャットから状態を要約できるなら、開発者でないメンバーでも初動の言語化がしやすい。

ただし、これはSREやDBAの代替ではない。health checkの結果は、原因候補の整理には役立つが、根本的なインデックス設計、クエリ最適化、データ保持方針、バックアップ、監視アラート、復旧訓練までは別に必要になる。顧客向けアプリなら、障害時のステータス通知や問い合わせ対応も必要だ。

実務では、health checkを「障害対応フローの入口」として定義するのがよい。たとえば、ユーザーから遅延報告が来たら、まず担当者がLovableにhealth checkを実行させる。出力をチケットに貼る。発生時刻、対象プロジェクト、直前の変更、接続数やディスクなどの要約、再現手順をそろえる。そのうえで、開発者やSREが通常のログやDBメトリクスを見る。

これにより、非エンジニアが「遅いです」とだけ報告する状態から、少なくとも「接続数増加が疑わしい」「ディスクが逼迫している可能性がある」「特定操作でタイムアウトする」という報告に近づく。AIアプリビルダーを現場に広げるなら、この初動圧縮は価値がある。

## 日本企業向けの運用設計

第一に、MCP server registryを作る。接続名、提供元、接続先URL、認証方式、読み取り/書き込み権限、対象データ分類、利用部門、責任者、有効期限、レビュー日を記録する。全プランで使えるからこそ、個人任せにしない。

第二に、接続先の環境を分ける。PoCや社内検証では、本番CRMや本番DBへ直接つながない。テスト環境、マスク済みデータ、読み取り専用API、サンプルissueから始める。プロトタイプの速度と本番データ接続を同時に解禁すると、事故時の説明が難しくなる。

第三に、OAuthとAPI keyの失効手順を決める。MCP serverがOAuthを使う場合、誰のアカウントで認可されるのか、退職時にどう無効化するのかを確認する。API keyやBearer tokenを使う場合は、共有キーの棚卸し、ローテーション、漏えい時の停止手順が必要だ。

第四に、Lovable MCP serverの研究プレビュー利用を別管理にする。外部AIからLovableを操作できる経路は、便利な自動化であると同時に、実プロジェクト編集とクレジット消費を伴う。個人がClaude DesktopやCursorから接続する場合でも、業務ワークスペースでは承認制にしたほうがよい。

第五に、database health checkの結果保存場所を決める。チャット内に残すだけでは、障害報告、再発防止、監査に使いにくい。Jira、Linear、Notion、GitHub Issueなど、チームの正式な作業記録へ転記する運用が必要だ。

第六に、生成されたアプリの公開基準を分ける。部門内の一時ツールなら軽いレビューでよいかもしれない。顧客向け、個人情報あり、決済あり、社外公開ありの場合は、通常のアプリ開発と同等のレビュー、脆弱性確認、アクセス制御、バックアップ、ログ保全を要求する。

第七に、教育資料では「MCPを使う方法」より「MCPで何をつながないか」を先に書く。便利な接続方法だけを教えると、利用者は接続を増やす方向に動く。禁止データ、承認が必要な接続、読み取り専用にすべき接続、削除すべき接続を明示したほうがよい。

## 導入シナリオ

現実的な導入は、三段階がよい。

第一段階は、公開情報とサンプルデータだけを使うプロトタイプである。マーケティングページ、社内フォーム、簡単な管理画面など、実データを使わなくても検証できるものから始める。この段階ではCustom MCPを使うとしても、ドキュメントやサンプルissueに限定する。

第二段階は、読み取り専用の社内文脈を使う。たとえば、マスク済みの要件管理、公開可能な社内ガイドライン、テスト用APIをつなぐ。ここで、生成物の精度が上がるか、誤った仕様を取り込まないか、権限の説明ができるかを確認する。

第三段階は、限定部門での実運用である。Lovable Cloudを使うなら、database health check、バックアップ、権限、公開URL、退職者対応、障害時連絡を手順化する。外部AIからLovable MCP serverを使う場合は、利用者、接続クライアント、許可ツール、実行ログ、クレジット消費を管理対象にする。

この段階分けをせず、いきなり「社内データにつながるAIアプリを誰でも作れる」状態にすると、成果より先に統制不備が問題になりやすい。Lovableの更新は導入を楽にするが、企業の責任を消すわけではない。

## まとめ

LovableのCustom MCP全プラン開放とCloud database health checkは、AIアプリビルダーの役割が広がっていることを示す。生成、接続、運用診断が同じ作業面に近づくことで、プロトタイプから業務アプリへの距離は短くなる。日本の開発・プロダクトチームにとって、これは試す価値がある。

同時に、試し方には注意が必要だ。MCP serverは権限境界であり、Lovable MCP serverは外部AIから実プロジェクトを動かす経路であり、database health checkは監視の代替ではなく初動調査の補助である。この三つを分けて理解し、接続台帳、データ分類、失効手順、障害記録、公開基準を作る。そこまでできれば、Lovableは単なる「速い試作ツール」ではなく、管理されたAIアプリ開発基盤の一部として評価できる。
