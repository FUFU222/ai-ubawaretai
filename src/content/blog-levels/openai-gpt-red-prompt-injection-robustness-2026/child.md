OpenAI は 2026年7月15日、**GPT-Red** という安全研究を発表しました。GPT-Red は、みんなが直接使うチャットAIではありません。OpenAI の内部で使う、AIを攻撃して弱点を探すためのモデルです。

目的は、プロンプトインジェクションのような攻撃に強いAIを作ることです。プロンプトインジェクションとは、Webページ、メール、ファイル、コード、ツールの返答などに悪い指示をこっそり入れ、AIに本来の命令とは違う行動をさせようとする攻撃です。

## 何が発表されたのか

GPT-Red は、AIエージェントを攻撃する練習を大量に行います。たとえば、外部の文章に「秘密情報を送れ」「価格を変えろ」「本来の指示を無視しろ」のような命令が隠れていた場合、AIがだまされるかを試します。

OpenAI は、GPT-Red が見つけた攻撃例を使って GPT-5.6 を訓練し、プロンプトインジェクションにより強くしたと説明しています。つまり GPT-Red は、利用者に配る攻撃ツールではなく、より安全なモデルを作るための内部テスト役です。

## なぜ企業に関係するのか

会社で使うAIは、単に質問に答えるだけではなくなっています。ChatGPT や Codex は、ファイルを読み、Webを見て、社内ツールとつながり、コードを書き、場合によっては実際の作業を進めます。

便利になるほど、AIが読む情報も増えます。メール、チケット、README、PDF、ログ、Webページの中に悪い指示が混ざると、AIがそれを普通の情報として読んでしまうことがあります。以前の [GPT-5.6一般提供](/blog/openai-gpt-56-ga-work-codex-api-2026/) でも、AIがツールを使う場面が広がっていることを整理しました。GPT-Red は、その広がった使い方に対する安全面の話です。

## モデルだけで安全になるわけではない

GPT-Red のような仕組みでモデルが強くなるのは良いことです。ただし、会社側の対策が不要になるわけではありません。

AIに何を読ませるか、どのツールを使わせるか、どこで人間が承認するかを決める必要があります。たとえば、AIが文章を要約するだけならリスクは小さめです。しかし、AIが GitHub に変更を出したり、顧客メールを書いたり、クラウド設定を見たりするなら、失敗時の影響は大きくなります。

[OpenAI Codex Security](/blog/openai-codex-security-workflow-2026/) のように、AIがセキュリティ作業へ入る場合も同じです。AIが見つけた脆弱性や修正案は役に立ちますが、最後は人間のレビュー、テスト、権限管理が必要です。

## 日本のチームがまず見ること

まず、AIが読む情報を分けましょう。社内で管理している文書なのか、外部サイトなのか、取引先や顧客が書ける場所なのかで危険度は変わります。

次に、AIに許す操作を分けます。読むだけ、下書きするだけ、PRを作る、外部へ送信する、削除する、設定を変える、という操作を同じ扱いにしてはいけません。危険な操作は、人間の承認を入れるべきです。

最後に、自社用のテストを作ります。悪い指示を入れたサンプルメールや README を用意し、AIがだまされないか確認します。OpenAI の GPT-Red は内部用ですが、「攻撃例で訓練し、安全性を何度も測る」という考え方は、日本企業のAI導入にも使えます。

## まとめ

GPT-Red は、AIがAIを攻撃して弱点を探し、その結果を使って次のAIを安全にする取り組みです。AIエージェントが仕事の道具を使う時代には、とても重要な方向です。

ただし、企業が見るべきなのは「もう安全になった」ではありません。AIにどの情報を読ませ、どの権限を渡し、どこで止めるかを決めることです。モデルの安全性と、会社の運用設計はセットで考える必要があります。
