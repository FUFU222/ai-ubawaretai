Anthropic の Project Glasswing は、AI を使って重要ソフトウェアの脆弱性を見つけ、防御側が先に修正できるようにする取り組みです。2026年5月22日の初期アップデートでは、Claude Mythos Preview という未公開モデルを使い、パートナー全体で高・重要度の脆弱性を1万件超見つけたと説明されました。

大事なのは、AI が「危ない場所を見つける力」をかなり高めた一方で、人間側の処理が追いつかなくなっていることです。脆弱性は見つけただけでは安全になりません。本当に問題があるかを確認し、保守者に知らせ、修正を作り、利用者がアップデートするところまで進める必要があります。

## 何が起きたのか

Project Glasswing には、AWS、Apple、Google、Microsoft、Cisco、Palo Alto Networks、Linux Foundation など、重要なソフトウェアやインフラに関わる組織が参加しています。Anthropic は、これらのパートナーに Claude Mythos Preview を提供し、防御目的でコードやシステムを調べてもらっています。

今回の発表では、Project Glasswing のパートナーが多くの高・重要度脆弱性を見つけたことが示されました。さらに Anthropic は、1,000件を超える OSS プロジェクトもスキャンし、多数の脆弱性候補を見つけています。

ただし、未修正の脆弱性の詳しい内容は広く公開されていません。悪用される前に利用者が更新できる時間を確保するためです。脆弱性情報は、見つけた瞬間にすべて公開すればよいものではありません。

## なぜ日本企業にも関係するのか

日本企業は、自社で書いたコードだけでシステムを動かしているわけではありません。OS、ブラウザ、クラウド、コンテナ、OSSライブラリ、SaaS、ネットワーク機器など、多くの外部ソフトウェアに依存しています。Project Glasswing が示したように、AI がそれらの脆弱性を大量に見つけるようになると、利用企業も影響を受けます。

たとえば、ある OSS ライブラリに重要な脆弱性が見つかったとします。そのとき自社のどのサービスがそのライブラリを使っているのか、どのバージョンなのか、外部から攻撃できる場所にあるのかがすぐ分からないと、対応が遅れます。

そのため、AI時代の脆弱性対応では、スキャナーを入れるだけでなく、依存関係の把握、影響範囲の確認、緊急パッチ、顧客説明の準備が必要になります。

## 何から始めればよいか

最初にやるべきことは、脆弱性情報を受け取る入口を整理することです。AI が出した結果でも、従来のSASTや依存関係スキャンでも、まずは本当に再現するか、自社に影響があるか、急ぐべきかを判断する流れが必要です。

次に、重要なシステムの依存関係を見えるようにします。SBOM やパッケージ管理、コンテナイメージの棚卸しが役立ちます。どのソフトウェアを使っているか分かっていれば、CVE や GitHub Security Advisory が出たときに素早く調べられます。

最後に、パッチがすぐ出ない場合の一時対策を決めておきます。設定変更、機能停止、WAF、認証強化、ログ監視、アクセス制限などでリスクを下げられる場合があります。すべてを即日修正できなくても、被害を減らす方法はあります。

## まとめ

Project Glasswing のポイントは、AI が脆弱性を大量に見つけられるようになったことだけではありません。見つかった後に、誰が確認し、誰が直し、いつ利用者へ届けるのかが新しい問題になっていることです。

日本企業は、AI セキュリティを便利な新ツールとして見るだけでなく、脆弱性対応全体の処理能力を上げるきっかけとして捉えるべきです。自社コード、OSS、クラウド、委託先を含めて、発見から修正までの流れを早めることが重要になります。
