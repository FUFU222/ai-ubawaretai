OpenAI の 2026年6月12日付 ChatGPT Release Notes は、GPT-5.2 models の ChatGPT 退役を示した。対象は GPT-5.2 Instant、GPT-5.2 Thinking、GPT-5.2 Pro で、既存会話は対応する GPT-5.5 model で継続される。これは model launch ではなく model lifecycle event であり、enterprise ChatGPT operations の観点ではかなり重要である。

今回の更新は、[OpenAI o3/GPT-4.5退役、ChatGPT運用棚卸し](/blog/openai-o3-gpt45-chatgpt-retirement-2026/) と同じ系列にある。5月28日の退役告知では、o3 / GPT-4.5 / Canvas 依存をどう棚卸しするかが主題だった。6月12日の GPT-5.2 退役では、より最近の標準的な利用履歴が GPT-5.5 系へ移るため、長期会話、GPTs、shared prompts、enablement materials の管理が焦点になる。

また、[GPT-5.5 InstantでChatGPT標準運用はどう変わるか](/blog/openai-gpt-55-instant-chatgpt-default-2026/) で扱った標準モデル更新の後続イベントとしても読める。OpenAI は GPT-5.5 を現行 ChatGPT 体験の中心へ寄せ、旧モデルを段階的に下げている。これは自然な製品運用だが、企業側がモデル名を業務手順に埋め込んでいるほど、退役のたびに運用負債が表に出る。

## 事実: GPT-5.2系はChatGPTから外れた

OpenAI の説明では、2026年6月12日以降、GPT-5.2 models は ChatGPT で利用できない。対象は GPT-5.2 Instant、GPT-5.2 Thinking、GPT-5.2 Pro である。既存会話が失われるわけではなく、GPT-5.2 を使っていた会話は対応する GPT-5.5 model で自動的に継続される。

この「既存会話は続くが、継続モデルは変わる」という点が運用上の核心である。ChatGPT の会話は、単発の質問応答ではなく、業務文脈の容器になっていることがある。数週間にわたる製品企画、法務論点、採用文面、営業提案、開発設計、障害分析の会話では、過去のやり取りを前提に次の回答を求める。モデルが GPT-5.5 系へ切り替わると、同じ文脈でも推論の粒度、出力の長さ、曖昧さへの反応、ツール利用判断が変わる可能性がある。

OpenAI は、GPT-5.2 退役が以前から告知されていたと説明し、ChatGPT では後継モデルのリリース後、旧モデルを概ね 90 日残すという考え方も示している。つまり、旧モデルの長期残存を前提にした手順は、今後も維持しにくい。企業側は、モデル名が変わる前提で運用を設計する必要がある。

関連する Help Center では、GPT-5.5 Thinking が ChatGPT の各種ツールをサポートし、GPT-5.5 Instant は Canvas を除く現在の ChatGPT ツールをサポートするとされている。この点も無視できない。GPT-5.2 から GPT-5.5 へ移るとき、単に回答品質だけでなく、どのタスクを Instant / Thinking / Pro に割り当てるかが変わる。

## ChatGPT UI と API を分離して扱う

最初に分けるべきなのは、ChatGPT UI と API である。今回のリリースノートは ChatGPT のモデル利用と既存会話の継続先を説明している。API の production workload が、6月12日の ChatGPT 更新だけで同じように置き換わると読むべきではない。

ただし、API チームにも影響はある。多くの開発組織では、最初のプロンプト設計や評価観点の探索を ChatGPT で行い、その後 API に実装する。ChatGPT 側のモデルが GPT-5.5 系へ寄ると、探索時の出力と、本番 API の固定モデル出力がずれる可能性がある。プロンプト評価に ChatGPT を使うなら、モデル名、実行日、入力、期待出力、合格基準を残すべきだ。

本番 API では、固定モデル、`chat-latest` のような追随モデル、評価環境を明確に分ける必要がある。最新モデルを早く試したいチームは追随モデルを検証環境で使う。一方、顧客向け機能、社内決裁、法務・金融・医療・人事に近い用途では、固定モデルと評価データを中心にする。モデル更新のたびに出力が変わること自体がリスクになるからだ。

この分離は、[OpenAIのモデルピッカー刷新と推論制御の見直し](/blog/openai-chatgpt-model-picker-reasoning-controls-2026/) で扱った UI 整理にもつながる。利用者向けにはモデル選択が分かりやすくなっていく一方、組織の運用では「Instant 相当」「Thinking 相当」「Pro 相当」「API 固定モデル」の境界を明文化する必要がある。

## 壊れやすいのはコードより運用知

GPT-5.2 退役で壊れやすいのは、アプリケーションコードだけではない。むしろ、社内に散った運用知のほうが影響を受けやすい。

API 実装なら、モデル名は設定ファイル、環境変数、 IaC、ログ、PR に残る。検索しやすく、変更レビューもしやすい。一方、ChatGPT の現場利用では、モデル名が社内 wiki、Notion、Google Drive、Slack、研修スライド、スクリーンショット、口頭説明、部門ごとのプロンプト集に散る。そこに `GPT-5.2 Thinking 推奨` や `5.2 Pro で最終確認` と書かれていると、退役後に利用者が迷う。

長期会話も同じである。ChatGPT を「作業ごとの一時的な相談窓口」として使うだけなら、モデル退役の影響は小さい。しかし、会話をプロジェクトの補助記憶として使っている場合、モデル切替は業務文脈の継続性に影響する。過去の結論を踏まえて続きを依頼したとき、同じ形式で返るとは限らない。

GPTs と shared prompts も運用負債になりやすい。GPTs の説明文や社内案内にモデル名が書かれていても、owner がいなければ誰も直さない。共有プロンプトが古いモデルの癖に合わせて作られている場合、新モデルでは冗長になったり、逆に情報が足りなくなったりする。これはモデル性能ではなく governance の問題である。

## モデル名から用途分類へ移す

実務上の対策は、社内向けの説明をモデル名中心から用途分類中心へ移すことである。モデル名は検証台帳や技術資料には必要だが、一般利用者向けの手順では長持ちしにくい。

たとえば、日常的な下書き、要約、翻訳、短い調査は Instant 相当とする。複雑な論点整理、規程や契約に近いレビュー、複数案の比較は Thinking 相当とする。重要な最終確認、複雑な多段推論、外部提出前の高精度チェックは Pro 相当とする。この分類なら、具体的なモデル名が GPT-5.2 から GPT-5.5 へ変わっても、手順の考え方は残る。

もちろん、用途分類だけでは足りない。高リスク業務では、モデル名、実行日、評価セット、期待出力、承認者、ロールバック先を持つべきである。ここでのポイントは、一般利用者向け説明と、管理者・開発者向け検証を分けることだ。前者は用途で安定させ、後者はモデル名で厳密に管理する。

この考え方は、[OpenAI新セキュリティ設定でChatGPTとCodex運用はどう変わるか](/blog/openai-advanced-account-security-codex-2026/) で扱ったセキュリティ設定とも相性がよい。アカウント保護、セッション管理、モデル分類、GPTs owner、API モデル指定は別々の管理面だが、最終的には「誰が、どの用途で、どの AI に、どの情報を渡したか」を説明できる状態へ集約される。

## 日本企業向けの棚卸し手順

第一段階は検索である。社内 wiki、Notion、Google Drive、Slack、GitHub、研修資料から `GPT-5.2`、`5.2 Thinking`、`5.2 Pro`、`Legacy Models`、`モデルピッカー` を検索する。検索結果は、業務影響あり、教育資料、古い議論、個人メモに分類する。全件を同じ優先度で直そうとすると止まるため、まず業務影響ありを拾う。

第二段階は影響度の採点である。外部提出物、顧客対応、法務・金融・医療・人事、経営判断、コード変更、セキュリティ判断に近いものを高リスクにする。単なる個人の壁打ちや雑な下書きは低リスクでよい。高リスク業務ほど、代替モデルの確認と owner の設定を先に行う。

第三段階は長期会話の確認である。部署ごとに、継続利用している重要会話を数件選ぶ。続きを GPT-5.5 系で実行し、回答の長さ、構成、根拠、前提の保持、不要な断定が変わらないか見る。過去の GPT-5.2 出力と完全一致する必要はないが、業務で使える品質かは確認する。

第四段階は GPTs と shared prompts の owner 設定である。社内配布している GPTs には、owner、用途、最終検証日、想定モデル分類、問い合わせ先を付ける。プロンプト集にも同じ情報を持たせる。モデル退役のたびに全社通知だけで済ませるのではなく、owner が自分のワークフローを直す形にする。

第五段階は API チームへの周知である。今回の更新は ChatGPT 側のモデル退役であり、API 本番移行の直接指示ではない。ただし、ChatGPT を使ったプロンプト検証の再現性には影響する。API 側は、Platform docs、model deprecation、pricing、評価セットを別途見る。ChatGPT の model picker と API の model id を混同しない。

## ベンダー管理として見るポイント

今回の退役は、OpenAI のモデルライフサイクル管理を評価する材料でもある。後継モデル公開後に旧モデルを一定期間残し、既存会話を対応モデルへ移すという方針は、ユーザー体験を壊しにくくするための設計である。一方、企業側から見ると、ChatGPT は SaaS の UI であり、モデル棚の変更は一般的な機能変更と同じく周知・教育・検証が必要になる。

AI ベンダー選定では、最高性能のモデルだけでなく、退役告知、移行期間、既存会話の扱い、API と UI の差分説明、管理者ログ、モデル固定性、ユーザー教育のしやすさを見るべきだ。日本企業では、AI 利用が現場から広がりやすいため、モデル退役が正式なシステム変更より先に現場へ影響する。

調達や契約だけで解決しようとしないほうがよい。情シス、AI 推進、法務、開発、人材育成が同じ台帳を見る必要がある。最低限の項目は、モデル名、用途分類、利用部門、owner、リスク、代替、検証日、次回確認期限である。これだけでも次のモデル退役時に慌てにくくなる。

## まとめ

OpenAI の 2026年6月12日更新は、GPT-5.2 Instant / Thinking / Pro を ChatGPT から退役させ、既存会話を GPT-5.5 系で継続するものだった。ChatGPT のモデル棚は GPT-5.5 系へ整理され、旧モデルを前提にした業務手順は更新が必要になる。

日本企業がやるべきことは、GPT-5.2 の性能を懐かしむことではない。長期会話、GPTs、shared prompts、研修資料、ChatGPT での API 事前検証が、古いモデル名に依存していないかを確認することだ。モデル名は今後も変わる。用途分類、owner、検証日、API との切り分けを持つ組織ほど、ChatGPT のモデルライフサイクルを安定して吸収できる。
