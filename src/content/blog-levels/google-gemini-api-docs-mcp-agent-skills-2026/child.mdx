Googleが、コーディングエージェント向けに**Docs MCP**と**Developer Skills**を出しました。

これは簡単にいうと、「AIが古い書き方でコードを書いてしまう問題」を減らすための仕組みです。

最近のAIはコードを書くのが上手ですが、時々かなり困ったことをします。見た目はそれっぽいのに、実は古いAPIの使い方だったり、今はない引数を使っていたりします。

今回のGoogleの発表は、そこにかなり正面から向き合ったものです。

## 何が増えたの？

1つ目は**Gemini API Docs MCP**です。

これは、コーディングエージェントがGemini APIの最新ドキュメントを見にいける仕組みです。エージェントは自分の学習だけに頼るのではなく、今の公式情報を見ながらコードを書けるようになります。

2つ目は**Developer Skills**です。

こちらは、今の推奨パターンや資料の見方、書き方のコツをまとめたガイドのようなものです。情報を見に行くだけでなく、「どう考えて使うとよいか」を整える役割があります。

## どうしてそれが必要なの？

AIのコード生成でよくある失敗は、「考える力が足りない」ことよりも、**持っている知識が古い**ことです。

APIの名前、引数、SDKの作法、認証の方法は、時間がたつと変わります。人間なら公式ドキュメントを見直せばよいですが、エージェントはそうしないと昔の知識で書き始めてしまいます。

その結果、

- 動かないコードが出る
- 余計な修正時間がかかる
- なんとなくAIを信用しにくくなる

ということが起きます。

## Docs MCPは何が良いの？

Docs MCPの良いところは、エージェントが**公式の最新情報に直接つながれる**ことです。

何となくWebを探すより、どこを見ているかがはっきりしています。これは正しさの面でも安心感の面でも大事です。

たとえば人間でも、古いブログ記事より公式ドキュメントを見たほうが確実です。エージェントでも同じで、ちゃんとした情報源に接続させたほうが壊れにくくなります。

## Skillsは何をしてくれるの？

Skillsは、「情報がある」だけでは足りないところを補います。

ドキュメントが多いと、どこを見るか迷ったり、古い癖のまま変な書き方を選んだりすることがあります。Skillsは、今の推奨に沿ってエージェントを誘導する役割を持ちます。

つまり、

- Docs MCPは**最新情報を届ける**
- Skillsは**その情報をうまく使わせる**

という分担です。

この2つがそろうと、エージェントの出すコードの初手がかなり良くなりやすいです。

## 数字はどう見る？

Googleは、この2つを組み合わせると**96.3%のpass rate**、**正答あたり63%少ないトークン**だったと説明しています。

この数字はGoogle自身の評価なので、そのまま何でも信じるというより、「少なくともGoogleの中ではかなり効果が見えている」という受け止め方がよさそうです。

でも方向性としては納得しやすいです。古い知識で遠回りするより、最初から最新ドキュメントを見たほうが、失敗もムダな説明も減るからです。

## 何がそんなに大きい話なの？

大きいのは、ドキュメントの役割が変わってきたことです。

昔は、ドキュメントは人が読むものでした。今はエージェントも読みます。しかも読むだけでなく、その内容をもとにコードを書きます。

そうなると、ドキュメントは説明書というより、**AIに正しい行動をさせるための材料**になります。

この考え方は、Gemini APIだけで終わらないはずです。他のAPI会社や社内ライブラリも、将来は似た形を取り入れていく可能性があります。

## 開発者にとっての意味

実際の開発では、エージェントに全部任せるというより、まず叩き台を書かせて、人が整える使い方が多いです。

このとき最初の叩き台が古いと、結局かなり直すことになります。だから「最初から今の書き方で出てくる」ことには大きな意味があります。

今回の仕組みは、そこを改善しやすくするものです。

## まとめ

GoogleのDocs MCPとDeveloper Skillsは、コーディングエージェントをもっと最新情報に近づけるための仕組みです。

ポイントは、AIをただ賢くするのではなく、**正しい情報につなぎ、正しい書き方へ導く**ことにあります。

これからのAI開発では、モデルの性能だけでなく、「どんな文脈を与えるか」がますます大事になっていきそうです。
