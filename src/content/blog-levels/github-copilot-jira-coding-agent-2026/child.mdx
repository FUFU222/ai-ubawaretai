GitHub Copilot が Jira のチケットを読んで、その内容からコードを書き始めて PR までつなげる方向へ進んでいる、というのが今回の話です。

つまり「仕事のメモ」と「実際のコード」が、前よりかなり近くなってきています。

## 何が起きたの？

標準記事では、Copilot が Jira チケットや Confluence の情報を読んで、モデル選択まで含めて実装を進め、自己レビューやセキュリティチェックも行う流れが紹介されていました。

ここで大きいのは、AI がただコードを補完するだけでなく、仕事の管理ツールそのものとつながり始めたことです。

## 何が便利なの？

これまでは、人が Jira を読み、必要なら Confluence を見て、作業内容を整理し、それから実装に入るのが普通でした。

もし AI がこの最初の立ち上がりをかなり肩代わりできるなら、着手までの時間が短くなります。

チケットに要件や受け入れ条件がちゃんと書かれていれば、そのまま AI への指示に近い形になります。

## どこが一番大きい変化？

「コードを書く前の説明」が、そのまま実装の入口になるところです。

Jira、Confluence、実装、PR が 1 本につながると、開発の始まり方自体が変わります。

コードを書く前の仕事が、単なる管理作業ではなく、AI を動かすための設計にもなります。

## 気をつけるところは？

一番大きいのは、入口の情報の質です。

チケットがあいまいだと、AI もあいまいな実装を返しやすくなります。

雑な仕様から雑な PR が高速で出るなら、速くなるどころか、あとで直す手間が増えます。

だから、AI が便利になるほど、良いチケットを書く力や、前提を明確にする力が大事になります。

## 実務ではどう変わりそう？

これからは、コードを書く力だけでなく、「AI が動きやすい形で仕事を整理する力」が重要になります。

受け入れ条件を書くこと、調べる資料をつないでおくこと、どこまで終われば完了かをはっきりさせることが、そのまま開発速度に効きます。

## まとめ

GitHub Copilot と Jira の連携は、AI がコードの中だけにいる時代から、仕事の流れ全体に入ってくる時代への変化です。

便利さは大きいですが、それ以上に、入口の情報をちゃんと整えることの大切さが増しています。
