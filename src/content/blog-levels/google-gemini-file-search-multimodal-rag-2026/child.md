Google が **Gemini API File Search** を更新し、画像を含む検索、メタデータでの絞り込み、ページ単位の引用に対応しました。

これは「AIがもっと賢く答えるようになった」というより、**AIが答える前に、正しい資料を探しやすくするための更新**です。RAG は、社内文書やPDFを検索して、その結果をAIの回答に使う仕組みです。ただ、実際の仕事では文書だけでなく、画像、図表、スクリーンショット、PDFが混ざります。今回の更新は、その混ざった資料を扱いやすくする方向です。

以前の記事で扱った [Gemini Embedding 2 のマルチモーダルRAG](/blog/google-gemini-embedding-2-multimodal-rag-2026/) は、画像や文書を意味で検索する土台の話でした。今回の File Search は、その土台を使って、開発者が検索機能として組み込みやすくする話です。

## 何が変わったのか

大きな変更は3つです。

1つ目は、画像を検索対象にできることです。Gemini API の File Search では、`gemini-embedding-2` を指定すると、テキストだけでなく画像も検索対象にできます。たとえば、商品写真、UI画面、現場写真、図版などを、自然文の質問で探しやすくなります。

2つ目は、メタデータで絞り込めることです。資料に「部署」「状態」「文書種別」のようなラベルを付けておくと、質問するときに検索範囲を狭められます。これにより、関係ない資料が混ざる問題を減らせます。

3つ目は、引用情報です。PDFのようにページがある資料では、どのページを根拠にしたかを返せます。画像の場合も、参照した画像チャンクを示す情報を扱えます。

## なぜ実務で重要なのか

AIの回答で困るのは、間違った答えそのものだけではありません。「どの資料を見てそう言ったのか分からない」ことも大きな問題です。

ページ引用があれば、ユーザーは回答を読んだあとに元資料へ戻れます。社内規程、契約書、製品マニュアル、稟議資料のような文書では、これはかなり重要です。AIの答えをそのまま信じるのではなく、根拠を確認しながら使えるからです。

また、日本の企業では、資料がPDFや画像付き文書として残っていることが多いです。営業資料、製造現場の写真、店舗の画像、操作マニュアルなどを検索できると、チャットAIの使い道は単なる要約から、業務検索へ広がります。

[Gemini のファイル直接生成](/blog/google-gemini-generate-files-2026/) と組み合わせると、資料を探し、根拠を確認し、その内容を文書にまとめる流れも見えてきます。ただし、そのためには検索対象と権限管理をきちんと決める必要があります。

## 注意点

マルチモーダル対応といっても、何でもそのまま入れられるわけではありません。Google のドキュメントでは、File Search は現時点で音声と動画形式には対応していません。画像は PNG/JPEG、最大 4K x 4K、1リクエスト最大6画像という条件があります。

そのため、動画を使いたい場合は必要な場面を画像として切り出す、音声を使いたい場合は文字起こしする、といった前処理が必要です。

もう一つ大事なのは、権限です。検索できるようにする資料には、個人情報、顧客情報、社外秘情報が含まれる可能性があります。便利だから全部入れるのではなく、まずは部署や用途を絞って始めるほうが安全です。[Workspace Intelligence の管理者制御](/blog/google-workspace-intelligence-admin-controls-2026/) でも触れたように、AIが何を見てよいかを先に決めることが大切です。

## どう始めるべきか

最初は小さく始めるのがよいです。

たとえば、1つの製品マニュアル群、1つの社内規程集、1つのサポートFAQ、1つの画像素材ライブラリだけを対象にします。そして、検索結果が本当に役に立つか、ページ引用が確認しやすいか、メタデータでノイズが減るかを見ます。

うまくいけば、次に対象部署や資料種別を広げます。逆に、根拠が見にくい、古い資料が混ざる、権限が曖昧だと分かったら、そこで設計を直すべきです。

今回の File Search 更新は、RAG を本番に近づけるための地味だけれど重要な一歩です。日本の開発チームにとっては、画像やPDFが多い業務から試す価値があります。
