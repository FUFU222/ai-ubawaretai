Google が 2026年5月5日に出した Gemini API File Search の更新は、RAG を本番運用へ寄せる上でかなり重要だ。見た目は「画像検索に対応しました」という開発者向けアップデートに見える。しかし中身を見ると、Google が retrieval 層を、モデルの周辺機能ではなく、エージェントや業務AIの中核部品として固めに来ていることが分かる。

事実として、今回 File Search にはマルチモーダル検索、カスタムメタデータフィルタ、ページ単位引用が入った。File Search ドキュメントでは、画像とマルチモーダル埋め込みに `gemini-embedding-2` を使う構成が示され、File Search store にアップロードした資料を `generateContent` の tool として使う流れが説明されている。リリースノートでも、2026年5月5日の更新として、画像を埋め込み検索できること、visual citation のための `media_id`、情報が見つかった場所を示す `page_numbers` が明記されている。

この流れは、直前に扱った [Gemini Embedding 2 の一般提供とマルチモーダルRAG](/blog/google-gemini-embedding-2-multimodal-rag-2026/) の実装面だ。Embedding 2 が「画像や文書を同じ意味空間に置く」土台だとすれば、File Search は「それを開発者が検索ツールとして使う」ための入口になる。さらに [Gemini Enterprise Agent Platform](/blog/google-gemini-enterprise-agent-platform-2026-04-23/) が企業内エージェントの実行・統制面を押さえ、[Workspace Intelligence](/blog/google-workspace-intelligence-admin-controls-2026/) が仕事の文脈利用と管理者制御を押し出していることを考えると、Google はかなり一貫している。モデルを売るだけでなく、文脈を集め、絞り、根拠を返し、業務の中で使わせる基盤を積んでいる。

## 事実: File Searchは何を肩代わりするのか

File Search は、Gemini API の中で RAG を構成するための built-in tool だ。ドキュメント上の説明では、ファイルを取り込み、チャンク化し、埋め込みを作り、インデックス化し、プロンプトに応じて関連情報を検索する。検索された情報はモデルのコンテキストとして使われる。

従来なら、この流れは開発者が複数の部品で組んでいた。ファイル保存、PDF分割、OCR、画像処理、埋め込み生成、ベクトルDB、メタデータ管理、検索、再ランキング、引用表示をそれぞれ設計する必要があった。もちろん本格的な検索基盤では今後も自前構成が必要な場面はある。ただ、File Search はその最小構成をかなりまとめてくれる。

今回の更新で特に変わったのは、検索対象と検証可能性だ。File Search ドキュメントは、File Search が `gemini-embedding-001` によるテキスト埋め込みと、`gemini-embedding-2` による画像・マルチモーダル埋め込みを使えると説明している。マルチモーダル File Search では、store 作成時に `models/gemini-embedding-2` を指定する。画像は PNG/JPEG に対応し、最大 4K x 4K、1リクエスト最大6画像という条件がある。

引用面では、応答の `grounding_metadata` に検索根拠が入り、ページを持つ文書では `retrieved_context.page_number` を扱える。画像チャンクを参照した場合には `media_id` が返り、その画像チャンクを取得する導線も説明されている。ここは、回答品質だけでなく、ユーザーの確認行動を設計する上で重要だ。

## 分析: メタデータフィルタは精度改善より統制に効く

今回の発表で一番実務的なのは、画像検索そのものより、メタデータフィルタかもしれない。

RAG の失敗は「検索できない」だけではない。むしろ本番では「検索できすぎる」ことが問題になる。似た文書が多すぎる。古い版が混ざる。法務レビュー前の草稿と最終版が同じように出てくる。A顧客向けの資料がB顧客の問い合わせで引かれる。営業資料と技術仕様が同じ言葉を使っていて、どちらを根拠にしたのか分からなくなる。

カスタムメタデータは、この問題をかなり現実的に扱える。Google の公式ブログは、部署や状態のような key-value を unstructured data に付けて、問い合わせ時に検索範囲を絞る例を示している。これは単なる検索精度のチューニングではない。RAG における **情報境界の設計** だ。

日本企業で最初に設計すべきメタデータは、たぶん次のようなものになる。

- `department`: 所管部署
- `document_type`: 規程、提案書、マニュアル、FAQ、契約書など
- `confidentiality`: 公開、社内限定、部門限定、顧客限定など
- `version_status`: draft、reviewed、final、deprecated など
- `effective_date`: 有効開始日
- `customer_or_project`: 顧客名や案件ID

このうち、AI検索の精度に直接効くのは `document_type` や `version_status` だろう。しかし統制に効くのは `confidentiality` と `customer_or_project` だ。ユーザー権限と検索メタデータを組み合わせないと、RAG は便利な検索ではなく、情報漏えいの入口になる。

## 分析: ページ引用は信頼性のUI問題である

ページ引用は、RAG の信頼性を上げる。ただし、引用があるだけで信頼できるわけではない。

ユーザーが本当に必要としているのは、AIの回答に「出典あり」と書かれることではなく、回答の根拠へ短時間で戻れることだ。PDFのページ番号が取れるなら、UIでは少なくとも資料名、ページ、更新日、抜粋、アクセス権の状態を見せたい。ページ番号だけだと、同名ファイルや版違いがある環境ではすぐ混乱する。

また、ページ引用は回答の妥当性を保証しない。検索結果が正しいページを指していても、モデルがその内容を過度に一般化する可能性はある。複数ページの条件を組み合わせる場面では、根拠の一部だけを引用して結論を出してしまうこともある。したがって、法務、医療、金融、セキュリティのような領域では、引用表示に加えて、回答のスコープ制限、低確信時の保留、レビュー導線が必要になる。

ここで重要なのは、File Search のページ引用を「監査機能」と呼びすぎないことだ。正確には、監査や確認に使える材料を返しやすくする機能だ。監査ログ、承認フロー、権限チェック、参照資料の版管理は、アプリケーション側で設計しなければならない。

## 日本企業での有望ユースケース

最初の候補は、画像やPDFが混ざるサポート検索だ。製品マニュアル、トラブル写真、過去問い合わせ、修理手順書を検索対象にすると、顧客サポートやフィールドサービスで効きやすい。テキストだけのFAQより、写真や図表が入る領域のほうが File Search の差分が出る。

次に、営業・提案資料の検索がある。日本企業では、過去提案書、業界別資料、価格表、導入事例、図版素材が散らばりがちだ。自然文で「製造業向けで、在庫管理に近い図版を使った提案」を探せるなら、営業企画やプリセールスの負担を下げられる。ただし、顧客別の機密性が強いので、メタデータと権限管理は必須になる。

3つ目は、規程・手順書検索だ。ページ引用はここで効く。回答だけでなく、規程の該当ページへ戻れることが重要だからだ。これは [Gemini のファイル生成](/blog/google-gemini-generate-files-2026/) のような文書作成機能と接続すると、検索結果から報告書や社内説明資料へ落とす流れも作りやすい。

4つ目は、開発チーム向けの設計資料検索だ。アーキテクチャ図、ER図、シーケンス図、ADR、README、Issue、PR説明が混ざる環境では、画像とテキストをまたぐ検索が効く。公式ブログでも、コードや設計図の文脈をエージェントへ渡す例が紹介されている。ここはコーディングエージェントとの相性が高い。

## 既存RAGから移行するべきか

すぐ全面移行する必要はない。むしろ、既存のRAG基盤があるチームほど、評価軸を分けて検証したほうがいい。

まず見るべきは、検索品質だ。テキストだけのFAQであれば、既存のベクトルDBやキーワード検索とのハイブリッドで十分な場合も多い。File Search の価値が出るのは、画像、PDF、ページ引用、メタデータフィルタが効く業務だ。

次に、運用コストを見る。File Search ドキュメントでは、ファイル保存とクエリ時の埋め込み生成は無料で、最初の索引化時の埋め込み生成と通常のモデル入出力トークンに課金されると説明されている。これは入り口として分かりやすい。しかし、本番では更新頻度が重要になる。毎日大量のPDFや画像が更新される業務では、インデックス再作成の粒度を設計しないと、コストだけでなく反映遅延も問題になる。

さらに、データライフサイクルを見る必要がある。ドキュメントによると、File API 経由でアップロードされた raw file は48時間後に削除される一方、File Search store に取り込まれたデータは手動削除またはモデル非推奨まで残る。これは便利だが、削除要求や文書廃止にどう対応するかを決めておかないと危険だ。特に個人情報や契約文書を扱う場合、store からの削除手順とログ確認を運用に入れるべきだ。

## 実装チェックリスト

本番前のチェックは、モデル選定よりも運用設計に寄せたほうがよい。

まず、store の分割単位を決める。部署別か、顧客別か、プロダクト別か、機密区分別か。ここを曖昧にすると、あとで検索範囲をメタデータだけで無理に制御することになる。

次に、メタデータを最小セットで始める。最初から20項目を付けると運用されない。検索と統制に効く5項目程度に絞り、評価しながら増やすべきだ。

3つ目は、引用UIを作る。回答の末尾にURLを出すだけでは足りない。資料名、ページ、抜粋、版、アクセス権、更新日をまとめて見せる設計が必要だ。

4つ目は、拒否と保留を設計する。検索結果が薄い、古い版しかない、引用が複数文書で矛盾する、ユーザー権限が不足する。こうしたケースでは、モデルに無理に答えさせず、確認依頼やエスカレーションへ送るほうがよい。

5つ目は、評価データを作る。日本語の社内資料では、英語の一般ベンチマークはほとんど役に立たない。実際の問い合わせ、過去の検索ログ、サポート履歴から、正解資料と正解ページを持つ評価セットを作るべきだ。

## まとめ

Gemini API File Search のマルチモーダル対応は、RAG の「作りやすさ」より「運用しやすさ」に効く更新だ。画像検索は分かりやすい目玉だが、本当に重要なのは、メタデータで検索範囲を絞り、ページ引用で根拠へ戻り、File Search store を業務単位で管理できることにある。

日本の開発チームにとっては、全社検索の夢を見るより、画像やPDFが多く、根拠確認が重要で、対象範囲を絞れる業務から始めるのが正しい。製品サポート、規程検索、営業資料検索、設計資料検索は特に相性がよい。

Google は Gemini Embedding 2、File Search、Workspace Intelligence、Gemini Enterprise Agent Platform を通じて、文脈基盤を上から下まで揃えようとしている。今回の更新はその中でも、開発者が直接RAGの品質と運用を変えられる層だ。既存RAGを持つチームも、これから作るチームも、画像、メタデータ、引用という3点で検証しておく価値がある。
