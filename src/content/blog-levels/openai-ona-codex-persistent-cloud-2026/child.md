OpenAI は 2026年6月11日、Ona という会社を買収する合意を発表した。Ona は、AI エージェントがクラウド上の安全な環境で作業し続けるための技術を持つ会社だ。買収が完了すると、Ona のチームは OpenAI の Codex チームに加わる予定である。

今回のニュースをかんたんに言うと、Codex に「長い仕事をするための作業部屋」を用意する動きだ。これまでの AI コーディング支援は、人間がパソコンの前で指示し、短い作業を返してもらうイメージが強かった。けれども、実際の開発では、テストを直す、原因を調べる、古いコードを移す、脆弱性を確認する、といった長い仕事が多い。

OpenAI は、Codex の仕事が数分ではなく数時間から数日に広がっていると説明している。そのため、ノートPCを閉じても、別の場所へ移動しても、Codex が企業の管理されたクラウド環境で作業を続けられることが重要になる。

## 何が変わるのか

大事なのは、Codex がただ賢くなる話ではないことだ。長い仕事を任せるには、AI がどこで動くのか、どのファイルを読めるのか、どの認証情報を使えるのか、どんな操作をしたのか、誰が最後にレビューするのかを決める必要がある。

Ona は、企業が管理できるクラウド環境、スコープされた認証情報、監査ログ、エージェントの実行管理などを重視している。OpenAI も、Ona の customer-controlled execution model によって、企業が自分たちのクラウド環境の中で agent を動かしやすくなると説明している。

つまり、Codex は「コードを書いてくれるチャット」から、「企業の開発環境の中で、長い作業を進める AI 作業者」へ近づいている。

## 日本企業に関係する理由

日本の会社では、開発環境が単純ではないことが多い。GitHub、社内チケット、CI、オンプレミス、クラウド、委託先、セキュリティ部門、情シスが関わる。AI に長い仕事を任せるなら、便利さだけでなく、責任と安全性を決めておく必要がある。

たとえば、Codex がバグ修正を進めるとしても、本番へ入れる前には人間のレビューが必要だ。脆弱性対応なら、どのリポジトリを読ませるか、どの結果を誰が確認するかを決めなければならない。この点は、[OpenAI Codex Security](/blog/openai-codex-security-workflow-2026/) の話ともつながる。

また、長時間実行は費用にも関係する。AI が長く動けば、モデル利用、クラウド環境、テスト実行、外部サービス利用が増える可能性がある。[OpenAI Codex支出管理](/blog/openai-codex-business-spend-controls-2026/) のように、上限や予算を決める考え方も必要になる。

## まず確認したいこと

最初に確認したいのは、どんな作業を任せるかだ。コードの調査、テスト修正、ドキュメント整理、移行作業、セキュリティ確認では、必要な権限が違う。最初から重要な本番システムを任せるより、低リスクなリポジトリや read-only の調査から始めるほうが安全である。

次に、AI が使う権限を小さくする。人間の管理者権限をそのまま渡すのではなく、読むだけ、Pull Request を作るだけ、テストを実行するだけ、というように分けるべきだ。

最後に、ログとレビューを残す。AI が何を読み、何を実行し、どんな変更を作ったのかが分からなければ、あとで説明できない。特に金融、医療、製造、公共のような分野では、便利さよりも説明責任が重要になることがある。

## まとめ

OpenAI の Ona 買収は、Codex を長時間動く企業向けエージェントに近づける動きだ。ポイントは、モデルの賢さだけではなく、安全なクラウド作業環境、認証情報、監査ログ、人間レビューをどう組み合わせるかである。

日本の開発チームは、今回の発表を「AI がもっと自動で開発する」という話だけで受け止めないほうがよい。Codex に長い仕事を任せるなら、どこで動かすか、何を読ませるか、どこまで書かせるか、誰が止めるかを先に決める必要がある。
