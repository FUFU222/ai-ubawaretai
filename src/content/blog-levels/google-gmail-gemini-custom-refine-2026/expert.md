Google Workspace Updates は **2026年7月14日**、Gmail の **Help me write** に任意の refine instructions を追加できるようにした。ユーザーは、Gmail のメール下書きに対して、プリセットの Formalize、Friendly、Shorten だけでなく、プロンプトバーから自由な追加指示を書いて再編集できる。Rapid Release と Scheduled Release の両方で展開が始まり、**2026年7月20日** までに完了予定とされている。

この更新は、Gmail の小さな使い勝手改善として消費しないほうがよい。営業、CS、採用、広報、購買、情シスのメール業務では、AIが初稿を作るだけでなく、会社の言い回し、顧客との関係、リスクのある表現、参照すべき情報に合わせて文面を調整する段階へ進むからだ。つまり、メールAIは「文章作成ツール」から「顧客接点の品質管理ツール」に近づく。

この位置づけは、Google Workspace のAI機能全体とつながる。[Workspace Intelligence の管理者制御](/blog/google-workspace-intelligence-admin-controls-2026/) は、Gmail、Drive、Calendar、Chat などを Gemini が文脈として使う管理線だった。[Google SheetsのGemini多言語化](/blog/google-sheets-fill-gemini-languages-governance-2026/) は、表計算の分類、要約、生成を扱った。[Workspace Studioの管理者制御](/blog/google-workspace-studio-admin-controls-2026/) は、業務フローのスターターとステップをどう許可するかという話だった。今回の Gmail 更新は、これらより顧客接点に近い「送信前の言葉」を扱う。

## 事実整理: 任意修正でできるようになったこと

Google の発表では、Gmail の Help me write で、ユーザーが email draft を prompt bar から custom refine instructions によって edit and revise できるようになったと説明されている。従来は Polish、Formalize、Shorten のような preset options が中心だった。今回の更新により、下書きが期待どおりでない場合、ユーザーは自分の言葉で不足情報の追加や表現調整を指示できる。

発表で示されている例は分かりやすい。2行目に不足している詳細を足す。依頼文に締切を含める。undo / redo で編集の前後を戻す。これはメール作成の実務に近い。現場のメール作業では、最初の下書きよりも、相手、目的、締切、温度感、社内ルールに合わせた微調整に時間がかかるからである。

対象エディションは広い。Business Starter / Standard / Plus、Enterprise Starter / Standard / Plus、Google AI Plus / Pro / Ultra、Frontline Plus、Google AI Pro for Education、AI Expanded Access などが含まれる。個人向けだけでなく、企業・教育・Frontline の利用者にも関係する更新である。

管理者条件も重要だ。発表では、この機能は Gemini for Workspace in Gmail と Workspace Intelligence access to Gmail の両方が有効な場合に既定で利用できると説明されている。エンドユーザー側には既定で見える更新でも、実際には管理者がどのOU / groupにどのAI機能を許すか、Gmail を Workspace Intelligence のデータソースとして有効にするかに左右される。

## Help Center から読むデータ利用と制約

Gmail Help は、Help me write が新しいメール下書きの生成と、既存下書きの tone and clarity 改善に使えると説明している。対象は eligible Google Workspace or Google AI plan で、複数の対応言語においてグローバルに利用できる。個人アカウントでは対応言語でも米国制限があるとされており、企業利用と個人利用で条件が異なる点は押さえたい。

同じ Help Center は、Help me write がより関連性が高く個人化された下書きを作るために、ほかのメールや Drive ファイルの情報を使う場合があると説明している。さらに、Gemini がどのメールや文書を使ったかを Sources から確認できる。これは、企業利用では監査の入口になる。AIが「それらしい」文面を出したとき、金額、日付、顧客名、添付ファイル、契約名がどこから来たのかを確認できるからだ。

Workspace Intelligence の管理者ガイドでは、管理者が Gmail、Drive and Docs、Calendar、Chat をデータソースとして個別に有効・無効化できると説明されている。既定はオンで、変更には最大48時間かかる場合がある。さらに、DLP ルールやユーザーの smart features 設定も関係する。つまり、Gmail のメールAIは、単純な「Gmail内のボタン」ではなく、Workspace 全体の文脈利用設定の上にある。

また、Gmail Help は、Gemini feature suggestions が Google の見解ではないこと、医療・法律・金融などの専門助言として依存すべきではないこと、不正確または不適切な提案があり得ることを説明している。営業メールや採用メールは専門助言そのものではないが、契約条件、返金、謝罪、採否、法的主張、障害対応を含むなら、同じ慎重さが必要になる。

## 日本企業での価値: 文章品質の平準化

日本企業で最も大きい価値は、メール文面の平準化である。営業担当ごとの文章力、CS担当ごとの謝罪表現、採用担当ごとの候補者対応、購買担当ごとの交渉文面にはばらつきが出る。テンプレートを用意しても、状況に合わせて少し直す段階で個人差が戻る。任意修正ができる Help me write は、この「テンプレートと個別対応の間」を埋める。

営業では、商談後のお礼、資料送付、見積もり依頼、稟議支援、失注フォロー、更新案内がある。ここでは、相手の役職、商談段階、製品カテゴリ、次のアクションに合わせて文面を調整する必要がある。任意修正なら、「決裁者向けに短く」「価格条件は本文に書かず、添付見積書を参照」「先方の検討期限を明記」といった指示が可能になる。

CSでは、問い合わせ回答、障害告知、仕様説明、解約防止、アップセル案内がある。ここでは、謝罪と説明のバランスが重要だ。「原因未確定なので断定しない」「回避策を先に書く」「次回更新予定を明記」「法的責任を認める表現は避ける」といった指示を、部門標準として配れる。

採用では、面接調整、候補者へのフォロー、辞退連絡、内定後案内がある。ここでは、評価理由や機微情報を不用意に書かないことが重要だ。「評価理由は書かない」「次のアクションだけを明確に」「候補者に心理的負担を与えない表現」といった指示は、採用体験を保つうえで効く。

## リスク: 自由な指示は危ない文面も作れる

任意修正のリスクは、自由に指示できることそのものにある。プリセットの Formalize や Shorten では、変化の範囲が比較的狭い。しかし、自由入力では「返金を約束して」「契約上問題ないと書いて」「競合より必ず安いと強調して」「候補者の評価を詳しく伝えて」「障害原因は弊社ではないと断定して」といった危険な指示も可能になる。

AIは安全上の制約を持つとしても、企業ごとの契約条件、顧客別の例外、社内承認ルール、業界規制までは自動で完全に理解しない。特に、過去メールやDriveファイルの文脈を参照する場合、古い資料、別顧客の条件、検討段階の文面、社内だけのメモが混ざる可能性を考える必要がある。

もう一つのリスクは、自然な文章ほど人間が確認を省くことだ。粗いAI下書きなら、人は違和感に気づく。だが、Gmail上で滑らかに整えられ、トーンも相手に合っていると、内容の妥当性より文章の自然さに引きずられる。これはメールAIに限らず、生成AI全般のリスクである。

[Googleのファイル生成機能](/blog/google-gemini-generate-files-2026/) でも、AIが完成物に近い出力を作るほどレビュー責任が重要になると整理した。Gmailでは、その完成物がそのまま外部に送られる。したがって、メールAIの運用は、プロンプト例よりも送信前レビューの分類が重要になる。

## 推奨する管理設計

第一に、メール種別ごとのリスク分類を作る。低リスクは日程調整、お礼、社内の一般連絡、資料送付の案内である。中リスクは顧客問い合わせ、営業フォロー、採用連絡、請求確認である。高リスクは契約条件、返金、謝罪、障害、法務、医療・金融・人事評価、プレス対応である。高リスクでは、AI下書きの利用を禁止するか、責任者レビューを必須にする。

第二に、部門別の refine prompt 集を作る。営業、CS、採用、社内IT、購買、広報で求める文体は違う。たとえば営業なら「次のアクションを1つに絞る」「価格条件は添付資料参照」。CSなら「原因未確定の場合は断定しない」「顧客の作業手順を番号付きで示す」。採用なら「評価理由を含めない」「候補者への敬意を保つ」。自由入力を完全に自由にせず、良い指示例を配るのが現実的だ。

第三に、禁止表現を明文化する。「必ず」「保証します」「法的に問題ありません」「返金します」「弊社に責任はありません」「採用しない理由は」など、部門ごとの危険表現を短いリストにする。長い規程より、送信前に担当者が確認できる表がよい。AIに「この禁止表現を避けて」と指示するだけでなく、人間レビューでも使う。

第四に、Sources 確認の条件を決める。AI下書きに固有名詞、日付、金額、契約名、ファイル名、顧客名、個人情報が入ったら、Sources を確認する。Sources が出ない、または根拠が曖昧な場合は、本文から削るか、人間が一次情報で確認する。これは、AIメールのファクトチェック手順として最小限である。

第五に、管理者設定の台帳を作る。Gemini for Workspace in Gmail、Workspace Intelligence の Gmail ソース、Drive and Docs ソース、Calendar ソース、Chat ソース、Workspace smart features、DLP、対象OU / group、対象プランを一覧化する。問い合わせが来たときに「なぜ使えないか」「なぜ情報を参照しないか」を説明できるようにする。

## 部門別ユースケースと運用例

営業部門では、商談後のお礼メールが最初の候補になる。AIに会議後の要点を反映させ、次回アクションを明確にし、相手の社内稟議に必要な資料を案内する。ただし、割引率、契約開始日、SLA、個別条件は本文で断定しない。見積書や契約書を参照する文面に限定する。

CS部門では、問い合わせの一次回答に使える。顧客の質問を要約し、確認手順、暫定回避策、次回連絡予定を書く。高リスクなのは障害や返金を含むケースだ。原因が未確定なら断定しない。返金や補償は承認済みテンプレート以外を書かない。ここはレビュー必須にするべきだ。

採用部門では、面接調整や候補者フォローに使える。候補者体験を損なわない文体に整える効果は大きい。一方、合否理由や評価コメントをメールに含める用途は避ける。AIが過去の面接メモや社内評価を参照できる環境なら、意図せず機微情報が文面に出る可能性を考える必要がある。

社内ITでは、問い合わせ回答、メンテナンス告知、障害一次案内に使いやすい。定型的な手順案内を分かりやすくする用途は低リスクだが、セキュリティインシデントやアカウント停止、監査関連の連絡は慎重に扱う。誤った手順や不適切な情報開示が影響を広げるためである。

## 30日以内の導入チェックリスト

最初の1週間は、対象部門と設定を確認する。どのユーザーが対象エディションを持つか、Gemini in Gmail が有効か、Workspace Intelligence の Gmail ソースが有効か、Smart features がどう設定されているかを見る。Rapid Release と Scheduled Release の差も案内する。

2週目は、低リスクメールで試す。日程調整、お礼、資料送付案内、社内通知などを使い、AI下書きの修正量、誤情報の頻度、Sources確認の必要性を見る。ここで高リスク文面を試す必要はない。

3週目は、部門別の指示例と禁止表現を作る。実際のメールを匿名化し、良い refine prompt と悪い refine prompt を並べる。AIに何を頼むと便利か、何を頼むと危ないかを、現場が自分で判断できる形にする。

4週目は、レビュー運用を決める。低リスクは本人確認、中リスクはチーム内確認、高リスクは責任者または専門部門確認に分ける。すべてを承認制にすると使われない。高リスクだけを絞ることが重要だ。

## 今回の更新の読み方

今回の Gmail 更新は、Google Workspace のAI機能が、文書・表・チャット・自動化だけでなく、実際の対外コミュニケーションに深く入ることを示している。AIがメールを作ること自体は以前からあった。変わったのは、現場ユーザーが自由な指示で文面を細かく直し、より実務に近い状態へ持っていける点である。

この自由度は、現場の生産性には効く。一方で、管理者や法務、セキュリティにとっては、メールAIを「個人の便利機能」として放置しにくくなる。特に Workspace Intelligence と組み合わさると、Gmail の文面は他のメール、Drive ファイル、Calendar、Chat の情報とつながり得る。データソース、アクセス権、DLP、Sources確認、送信前レビューを一体で見る必要がある。

日本企業にとっての現実解は、全社禁止でも無条件解放でもない。低リスクのメール作成から始め、部門別の指示例を配り、高リスク文面だけレビューを残す。管理者は設定台帳を持ち、現場はSources確認と禁止表現を覚える。この程度の運用なら、メールAIの便利さを潰さずに、事故の確率を下げられる。

## まとめ

Gemini in Gmail の任意修正対応は、Help me write をより実務的なメール編集支援にする更新だ。営業、CS、採用、社内ITでは、メール下書きを相手や目的に合わせて素早く整えられる。日本企業では、文面品質の平準化と処理時間の短縮に効きやすい。

ただし、自由な修正指示は危険な文面も作れる。価格、契約、返金、謝罪、採用、法務、障害のメールでは、AIの自然な文章をそのまま信用しないほうがよい。Workspace Intelligence のデータソース、Sources確認、DLP、スマート機能、送信前レビューをセットで設計する必要がある。

今回の更新は、Google Workspace AIガバナンスの一部として扱うべきだ。メールAIは、ユーザーの文章作成を助けるだけでなく、会社の顧客接点そのものに影響する。便利さを活かすには、部門別の指示例、禁止表現、参照元確認、レビュー分類を先に決めることが重要である。
