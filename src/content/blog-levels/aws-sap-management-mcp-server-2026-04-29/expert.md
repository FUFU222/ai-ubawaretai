AWS が 2026年4月29日に公開した **AWS For SAP Management MCP Server** は、MCP の新ネタとして流すには惜しい発表だ。理由は単純で、今回 AWS が MCP を持ち込んだ先が、一般的な開発支援や検索補助ではなく、**SAP on AWS という高統制・高属人化しやすい運用領域** だからだ。

MCP という言葉だけを見ると、AI エージェントにツールをつなぐためのインターフェース追加に見える。しかし、SAP 運用の文脈に置くと意味が変わる。そこでは、アプリケーション一覧、コンポーネント状態、設定不備、バックアップ、メトリクス、定期実行、停止開始などが別々の画面や API、手順書、スクリプトに分散している。AWS は今回、その断片を **会話型の一つの操作面** に束ねる方向をはっきり示した。

以下では、一次ソースに基づく事実を先に整理し、その後で日本企業の SAP 運用、情シス統制、SI 依存、PoC 設計にとって何が変わるのかを考える。

## 事実: 発表の中心は「SAP-aware な運用操作を AI に開くこと」

AWS for SAP ブログによれば、この MCP Server は AWS Systems Manager for SAP を基盤とし、SAP アプリケーションの状態確認、設定チェック、ライフサイクル管理、定期スケジュール、ヘルスレポート生成を AI アシスタントから扱えるようにする。記事の冒頭では、SAP HANA の本番運用で **6 以上の AWS サービスにまたがる 10 以上の API 呼び出しや確認操作** が必要になるケースを例示し、その複雑さを自然言語の会話に寄せると説明している。

ここで重要なのは、「AI が AWS API をただ叩ける」ではなく、**SAP-aware** という表現が繰り返されていることだ。ブログ記事では、MCP Server は SAP のコンポーネント依存関係、検証済みパターン、クロスサービスのロジックを一つの会話面に統合するとされる。つまり、一般的な AWS API 経由の操作よりも、SAP の構成を前提とした文脈付きの応答が狙いになっている。

GitHub の README でもこの性格は明確で、ツール群は次の4カテゴリに分かれる。

- SAP Application Management
- Configuration Checks
- Scheduling
- Health Summary

具体的には、一覧取得、個別アプリ詳細、コンポーネント取得、登録、開始、停止、設定チェック起動、サブチェック結果確認、スケジュール作成、スケジュール停止、包括的ヘルスサマリ、Markdown レポート生成などが並ぶ。README では **20 以上のツール** が用意されていることも確認できる。

## 事実: Systems Manager for SAP を前提にしているため、既存資産に寄せて導入できる

今回の MCP Server は、SAP 管理基盤を置き換える製品ではない。ブログ記事では、**SSM for SAP に登録された SAP アプリケーションが基盤** と明記されている。README でも最初の前提として、AWS account と Systems Manager for SAP の設定が必要だとしている。

この構成は、日本企業にとって採用障壁を下げる。なぜなら、すでに SAP on AWS を運用している企業は、アプリ登録、開始停止、設定チェック、バックアップ連携といった基礎を SSM for SAP 側に寄せ始めていれば、その上に **会話型レイヤーだけを足して試せる** からだ。新しい監視製品や運用 SaaS を別途契約しなくても、既存 AWS の中で検証しやすい。

さらに、ブログ記事ではこの MCP Server が **stdio transport でローカル実行** され、`~/.aws/config` の既存認証情報を利用すると説明される。README でも「LLM client と同じホストで動かす」ことが前提になっている。これは SaaS 一体型より保守的な構成だ。日本企業の情シス目線では、外部クラウドに新しい運用平面を増やすのではなく、**運用端末やジャンプホスト上で既存認証を使って動かせる** という理解になる。

## 事実: 必要権限は広く、参照専用ツールではない

導入判断で見落としてはいけないのが権限だ。README の Required IAM Permissions には、`ssm-sap:*` 系の操作に加えて、**EventBridge Scheduler、IAM、SSM、AWS Backup、CloudWatch** の権限が並ぶ。これは当然で、ツール群に「開始」「停止」「スケジュール作成」「レポート生成」「コマンド実行」が入っているからだ。

特に `scheduler:CreateSchedule` や `iam:CreateRole`、`ssm:SendCommand` が含まれる点は、これが単なる観察用の AI インターフェースではないことを示している。導入の初期段階で「とりあえず運用チーム全員に渡す」やり方は危うい。設計としては、**参照系中心の最小権限プロファイル** と、必要なときだけ使う **操作系プロファイル** を分ける発想のほうが自然だろう。

この広い権限セットは欠点ではない。むしろ AWS が、運用の現実に合わせて **読むだけでなく、点検し、予定し、動かす** ところまで含めている証拠だ。ただし、その分だけ統制設計を先にやる必要がある。

## 事実: 価格構造は「MCP は無料、実処理は基盤課金」

コストも実務上は重要だ。ブログ記事では、MCP Server 自体は **free** で、ローカルで動くと説明される。一方で、呼び出し元となる AWS サービスの通常利用料は発生する。ここで手がかりになるのが Systems Manager for SAP のドキュメントだ。

同ドキュメントでは、次のように分かれている。

- SAP アプリケーション登録と管理: 無料
- Application-aware な start / stop: 無料
- Basic monitoring and insights: 無料
- SAP configuration management: **1 回 1 アプリケーションあたり 0.25 USD**

これはかなり重要だ。AI で会話しながら操作できるようになると、利用者はチェックを「軽く回す」感覚になりやすい。しかし、設定チェックを定期化すれば、その回数だけ基盤課金は積み上がる。PoC の段階から、**何を単発で試すのか、何を定期ジョブにするのか** を区別しておかないと、導入評価が雑になる。

反対に言えば、価格構造が比較的読みやすいのは利点でもある。MCP のための新たな製品課金ではなく、既存サービスの従量課金に寄るため、情シスや購買部門が理解しやすい。

## 事実: 東京リージョン対応は日本企業の初期採用を後押しする

AWS のブログでは、この MCP Server は **Systems Manager for SAP の対応リージョンで使える** と書かれている。対応リージョンの裏取りとして、AWS General Reference の endpoints and quotas ページを見ると、Systems Manager for SAP の対象に **Asia Pacific (Tokyo) / ap-northeast-1** が含まれる。

これは日本企業向け記事として無視できない。多くの国内企業では、PoC でも本番でも「東京リージョンに置けるか」が最初の判定材料になるからだ。まして SAP のような基幹系であれば、リージョン非対応の時点で選択肢から外れることも多い。今回は少なくとも、**東京リージョンに公式 endpoint がある** ので、既存の東京リージョン SAP 資産に寄せた検証は組みやすい。

もちろん、これだけで「日本企業に最適」とは言えない。社内ネットワーク、PrivateLink 方針、踏み台構成、外部委託契約、運用責任分界によって現実は変わる。ただし、**技術的に最初の一歩を踏み出しやすい条件は揃っている** と言える。

## 考察: この発表の本質は、SAP 運用の属人知識を対話面に押し出すこと

ここからは考察だ。

日本の SAP 運用では、「状態を確認する方法」そのものが手順書やベテランの頭の中に分散していることが多い。たとえば、本番 HANA の健康状態を見るだけでも、コンソール、CloudWatch、バックアップ状況、設定チェック結果、OS レベルの確認、定期ジョブの有無などを横断する必要がある。これ自体は自動化できても、**どの順番で何を見るか** の知識が個人依存になりやすい。

今回の MCP Server は、その知識を完全に置き換えるものではないが、少なくとも **対話の入口に押し出して再利用しやすくする**。AI に「本番 HANA の状況を教えて」「金曜18時にチェックを回して」「前回の失敗内容をまとめて」と言えること自体が、作業者の入り口を揃える。これは属人化を一気に解消する魔法ではないが、**属人知識を再現可能な操作に近づける** 効果はある。

特に、日本企業で外部 SI に一次運用を委ねつつ、社内情シスは監督や調整だけ担う体制では、この種の対話面は有効だ。社内側が AWS コンソールや API を深く知らなくても、状況の把握や次の依頼を出しやすくなるからだ。

## 考察: 「AI に任せる」より「参照系から始める」が現実的

今回の発表を読んですぐに「SAP 運用自動化が来た」と受け取るのは早い。むしろ現実的な導入順は逆だ。最初に価値が出るのは、開始停止の自動化よりも、**参照、要約、定期点検、レポート化** の部分だろう。

理由は3つある。

1つ目は、権限が重いからだ。停止やスケジュール変更まで開放すると、ガバナンスが急に難しくなる。2つ目は、効果測定がしやすいからだ。参照系なら、問い合わせ対応時間、障害初動の確認時間、定例レポート作成時間の短縮で効果を見やすい。3つ目は、外部 SI や運用受託先との責任分界を崩しにくいからだ。まずは「社内で見える化を進める」使い方に留めるほうが政治的にも通しやすい。

したがって、PoC の初期ユースケースとしては次が現実的だ。

- SAP アプリ一覧と構成の自然言語参照
- 設定チェック結果の要約
- 週次レポートの Markdown 出力
- 東京リージョン環境の定例点検スケジュール作成

ここで価値が見えたら、その次に start / stop やスケジュール変更の自動化に広げるのがよい。

## 考察: 日本企業の導入論点は「AI 活用」ではなく「責任境界の再設計」

この発表を日本向けに読むなら、論点は AI の賢さより **責任境界** にある。具体的には、誰がこの MCP Server を使うのか、誰の AWS プロファイルで実行するのか、誰が設定チェック課金を管理するのか、誰が start / stop を承認するのか、という話だ。

MCP Server はローカル実行で、既存の `AWS_PROFILE` を使える。これは便利だが、そのままでは「端末ごとに見えるものが違う」「個人ごとの権限がそのまま出る」状態にもなりうる。日本企業で運用するなら、**専用プロファイル、専用端末、専用 role、監査ログ方針** を先に決めたほうがよい。

また、外部 SI との関係も変わる。これまで「状況確認は SI に聞く」が基本だった組織では、社内側が自分で状況を引けるようになるだけでも役割分担は変化する。SI の価値が消えるわけではないが、単純な状況回答より、設計改善や障害根本原因の分析に重心が移る可能性がある。

## まとめ

AWS For SAP Management MCP Server は、MCP の流行を SAP に当てはめた小粒な周辺発表ではない。むしろ、AWS が **基幹運用のような高統制領域にも「会話型の操作面」を持ち込める** と示した点で意味が大きい。Systems Manager for SAP を前提に、20 以上の SAP-aware tools、スケジュール、ヘルスレポート、ローカル実行、既存認証流用、そして東京リージョン対応まで揃っているため、日本企業でも検証の初手を打ちやすい。

実務的には、いきなり全面自動化を狙うより、**参照系と設定チェック系から始める** のが妥当だ。そのうえで、権限境界、課金回数、責任分界、運用受託先との役割整理を設計する。今回の発表が示しているのは、AI が SAP を勝手に運用する未来ではない。**SAP 運用の理解・点検・報告を、より再現可能で会話的な形に寄せる現実的な一歩** である。
