OpenAI は 2026年7月9日、**GPT-5.6** を一般提供しました。名前は Sol、Terra、Luna の3つです。6月の時点では一部の組織だけが試せる限定プレビューでしたが、今回は ChatGPT、ChatGPT Work、Codex、API で使える範囲がはっきりしました。

大事なのは、「新しいモデルが出た」だけではありません。仕事の難しさ、費用、スピードに合わせてモデルを選び、さらにAIがツールやアプリを使って仕事を進める範囲が広がったことです。

## Sol・Terra・Lunaとは

**Sol** は一番強いモデルです。複雑なコード修正、長い調査、難しい分析、セキュリティの確認などに向いています。

**Terra** は性能と費用のバランスを取るモデルです。日常的な業務文書、通常のコード作業、要約や比較などで最初に試す候補になります。

**Luna** は安く速く使うためのモデルです。短い分類、定型文、たくさんの要約など、量が多い仕事に向いている可能性があります。

ただし、安いモデルを何度もやり直すと、結局高くなることがあります。会社で使うときは、単価だけでなく「何回で終わるか」「人間がどれだけ直すか」も見る必要があります。

## ChatGPT WorkとCodexで何が変わるのか

ChatGPT Work は、チャットで答えるだけでなく、接続されたアプリ、ファイル、ブラウザ、デスクトップ上の作業をまたいで仕事を進める方向の更新です。たとえば、毎朝ダッシュボードを見て変化をまとめる、Slack の更新を見て会議アジェンダを直す、顧客フィードバックを整理する、といった使い方が想定されています。

Codex でも、複雑な作業を長く続けたり、複数の subagent に分けて進めたりしやすくなります。この流れは、以前の [Codex長時間運用](/blog/openai-codex-maxxing-long-running-work-2026/) とつながっています。

ただし、AIができることが増えるほど、会社はルールを先に決める必要があります。AIが資料を作るのはよいとしても、顧客へ送る、本番データを書き換える、課金を発生させる、外部へファイルを共有する、といった操作は人間の承認が必要です。

## APIではツールの使い方が重要になる

API では、Programmatic Tool Calling と Multi-agent beta が重要です。Programmatic Tool Calling は、AIが小さなプログラムを書き、ツールを呼び出し、途中結果を整理しながら進める仕組みです。

たとえば、社内ドキュメントを検索し、関連するログを見て、必要な部分だけをまとめるような作業では便利です。すべての中間結果をAIへ戻さずに済むため、tokenを減らせる可能性があります。

Multi-agent は、仕事を複数のAIに分けて同時に進める考え方です。大きな調査では便利ですが、費用と確認量も増えます。会社では、どの仕事だけに使うか、何人分の agent まで許すか、最後に誰が確認するかを決めておくべきです。

## 会社が最初に見るべきこと

まず、今 GPT-5.5 や他のモデルで動かしている仕事を並べます。次に、それぞれを Sol、Terra、Luna のどれで試すか決めます。

そのうえで、費用を記録します。入力token、出力token、cache write、cache read、tool call、やり直し回数を分けて見ると、どのモデルが本当に安いか分かりやすくなります。

最後に、承認ルールを作ります。AIが下書きや調査をするところまでは許しても、送信、公開、削除、権限変更、本番操作は人間が確認する。この線を決めてから使うと、便利さと安全性を両立しやすくなります。

GPT-5.6 は、とても強いモデルが増えたという話であると同時に、AIにどこまで仕事を任せるかを考えるニュースです。日本企業は、モデル名だけを差し替えるのではなく、費用、権限、承認、監査まで含めて移行を考える必要があります。
