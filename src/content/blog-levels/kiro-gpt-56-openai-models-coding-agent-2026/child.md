Kiro は 2026年7月14日、OpenAI の **GPT-5.6 Sol、Terra、Luna** を使えるようにしました。Kiro は AI コーディング用の IDE、CLI、Web を持つ開発ツールです。今回の更新で、Kiro の中で OpenAI モデルを選べるようになりました。

大事なのは、単に新しいモデルが増えたことではありません。開発チームは、どの作業に強いモデルを使うか、どの作業では安いモデルで十分か、会社のコードや設計情報がどこで処理されるかを考える必要があります。

## Sol、Terra、Lunaの違い

Kiro の説明では、**Sol** は一番難しい作業向けです。長いリファクタリング、複雑な実装、terminal を使う多段作業などに向いています。

**Terra** は、日常的な開発作業向けのバランス型です。通常のバグ修正、テスト補修、仕様を読んでコードを直すような仕事で最初に試す候補になります。

**Luna** は、速さと安さを重視するモデルです。短い要約、差分説明、軽い調査、たくさんの小さな作業に向きます。

Kiro の model docs では、Kiro credit の倍率も示されています。Sol は 2.4x、Terra は 1.2x、Luna は 0.6x です。ただし、これは目安です。安いモデルでも何度もやり直すと、結果的に高くなることがあります。

## なぜ日本の開発チームに関係するのか

日本企業では、AI コーディングツールを個人が自由に試し、そのまま業務に広げてしまうことがあります。しかし、会社のコード、顧客情報、設定ファイル、ログを扱うなら、使う前にルールが必要です。

Kiro は GPT-5.6 対応を experimental support と説明しています。また、提供リージョンは `us-east-1` と `eu-central-1` で、cross-region inference を使います。データの保存場所と、推論処理される場所は同じとは限りません。

そのため、金融、医療、公共、受託開発のようにデータの扱いが厳しい仕事では、保存場所だけでなく処理場所も確認する必要があります。個人アカウントで試す場合と、Enterprise として使う場合でも条件が変わります。

## どう使い分けるべきか

最初は、仕事を小さく分けると考えやすくなります。

短い説明や差分要約は Luna、普通のコード修正は Terra、複雑な設計や大きな変更は Sol から試す。これだけでも、強いモデルを何でも使う状態を避けやすくなります。

次に、結果を記録します。どのモデルで何分かかったか、どれくらい credit を使ったか、人間がどれだけ直したか、勝手に余計な変更をしなかったかを見ます。モデルの賢さだけでなく、レビューしやすさも重要です。

最後に、操作のルールを決めます。AI がコードの下書きを作るのはよくても、本番設定を変える、依存ライブラリを追加する、外部に送信する、秘密情報を読む、ファイルを削除するような操作は人間が確認すべきです。

Kiro の GPT-5.6 対応は、AI 開発ツールが複数モデルを選ぶ時代に入ったことを示しています。日本のチームは、どのモデルが一番強いかだけでなく、どの仕事にどのモデルを使い、どのデータを渡してよいかを決めることが大切です。
