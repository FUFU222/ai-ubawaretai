Anthropic が **2026年4月29日** に **Responsible Scaling Policy 3.2（RSP 3.2）** を有効化した。表面だけ見ると、また安全ポリシーの文言が少し変わった程度に見えるかもしれない。しかし実際には、**誰が Risk Report を監督し、誰が外部レビューを要求し、誰がそのレビュー担当者を承認するのか** という、AI ベンダー統治のかなり重要な部分に手が入っている。

日本の実務でこのニュースが効くのは、生成AIの導入がすでに「モデル比較」だけのフェーズを過ぎているからだ。PoC では性能がよく見えても、本番導入では法務、監査、リスク管理、情シス、購買、経営が **そのベンダーの統治をどこまで信じられるか** を問う。特に高リスク業務や社内横断導入では、モデルのベンチマーク以上に、**事故が起きたときにどう止まり、どう報告され、誰が説明責任を負うか** が重い。

以下では、RSP 3.2 の一次ソースをもとに変更点を整理し、その意味を日本企業の調達、監査、ベンダー管理、開発組織の観点から考える。

## 事実: RSP 3.2の差分は小さく見えて、統治の実効性に直結する

Anthropic の RSP 更新ページは、4月29日の変更点を簡潔にまとめている。そこで示されているのは次の3点だ。

1. **LTBT が Risk Report の external review を request できる**
2. **LTBT が external reviewers の選定を approve できる**
3. **LTBT に regular briefings を行うことが formal requirement になった**

この書き方は短いが、意味は重い。重要なのは、「安全を重視する」といった理念が足されたのではなく、**誰が、どの文書に対して、どの手続き権限を持つか** が追加された点にある。

RSP 3.2 の PDF でも changelog に同内容があり、4月29日の更新として正式に記録されている。つまり今回の改定は、単なるブログ記事上の補足や FAQ ではなく、**RSP 本文の管理対象として差分が刻まれた更新** である。

## 事実: Risk ReportはAnthropicの安全説明の中核であり、System Cardより重い文書として扱われている

RSP 3.2 本文では、Risk Report は公開中の全モデルと、必要に応じて重大リスクを持つ内部モデルを対象にした文書とされている。ここで言う Risk Report は、単に「このモデルはこういう機能があります」と説明する system card より一段重く、**なぜ開発・配備を続ける判断をしたのか** まで含めて説明する枠組みだ。

本文上では、Risk Report に少なくとも以下が含まれる。

- どの threat model を評価対象にするか
- その脅威モデルをどう定義したか
- どのような評価や証拠を使ったか
- どの軽減策を入れているか
- 全体としてどの程度のリスクと見ているか
- そのリスクが便益で正当化されると判断した理由
- 今後どう監視と軽減を続けるか

さらに、RSP 3.2 の本文では Risk Report を **3〜6か月ごとに公開** するとされている。新モデルごとに毎回出すわけではないが、大幅に能力が上がったモデルや内部で大きな意味を持つモデルについては追加説明を行う構造になっている。

この前提の上で LTBT の権限強化を見ると、今回の更新が単なる governance cosmetics ではないと分かる。外部レビュー要求権と選定承認権は、**Anthropic が出す安全説明の中心文書に対して、独立統治主体が実務的に介入する入口** だからだ。

## 事実: LTBTはブランド装飾ではなく、取締役会の構成に影響する独立主体である

LTBT を理解しないと、今回の改定の重みは見えない。Anthropic の 2023年の説明では、LTBT は **five financially disinterested members** から成る独立主体で、AI safety、national security、public policy、social enterprise などの知見を持つ人で構成されるとしている。

さらに重要なのは、LTBT が単なる advisory board ではないことだ。Anthropic は Class T stock を Trust に持たせ、この Trust が段階的に取締役会の選任・解任へ関与し、最終的には **4年以内に過半数のボードを選ぶ** と説明している。加えて、Trust は会社の重要変更について notice を受ける protective provisions も持つ。

これはかなり異例の設計だ。多くの AI 企業が ethics board や safety council を置いても、それらは経営判断を止める実権を持たないことが多い。Anthropic の LTBT は少なくとも制度上は、**経営やボードのインセンティブへ長期的に影響を与えるための統治装置** である。

したがって、RSP 3.2 で LTBT が Risk Report の外部レビューを要求し、レビュアー選定も承認できるようになったというのは、外から見る以上に意味がある。これは「安全に関する広報文書を、社外有識者に見てもらいます」ではなく、**ボード統治に関わる独立主体が、危険度評価の説明品質に触れる回路を強めた** ということだからだ。

## 事実: Anthropicは完全な安全保証ではなく、説明責任の質と比較可能性を上げようとしている

RSP 3.2 の本文を読むと、Anthropic は「これで危険はなくなる」とは言っていない。むしろ、AI の catastrophic risk は複数企業の行動に依存し、単独企業が unilateral に止まっても全体として安全になるとは限らない、とかなり率直に書いている。

そのため RSP 3 系では、以前よりも **強い safety argument をどう構成し、どう記録し、どうレビューにかけるか** に寄っている。Risk Report はその中核であり、外部レビューはその説明の質を上げる補助輪として位置づけられている。

RSP 本文では、将来的には standards-setting organizations や auditors のような独立した第三者が voluntary mechanism で高い品質基準を担う可能性にも触れている。これは今すぐ法定監査を受けるという意味ではないが、少なくとも Anthropic が **将来の規制・監査・標準化の接続点** を意識していることを示す。

## 考察: 日本企業にとっての実益は「AIベンダー評価の質問票」を作りやすくなること

ここからは考察だ。

日本企業では、現場の開発者や事業部が生成AIを高く評価しても、組織全体導入では次の壁にぶつかることが多い。

- 誰がベンダーのリスク説明を確認するのか
- ベンダーの社内判断を誰が止められるのか
- 更新頻度が高いモデルをどう監査するのか
- 事故や逸脱が起きたとき、どの文書を根拠に社内説明するのか

RSP 3.2 は、それ自体が答えを全部くれるわけではない。ただし、**何を質問すべきかを具体化する材料** としてはかなり使いやすい。たとえば日本企業の購買・監査・情シスは、今後 AI ベンダーに対して次のような確認をしやすくなる。

1. あなたの会社には Risk Report に相当する文書があるか。
2. それはどの頻度で更新されるか。
3. 外部レビューは誰の要求で始まり、誰がレビュー担当者を承認するか。
4. 経営陣と独立した統治主体は存在するか。
5. その主体は文書閲覧だけでなく、選任権や報告要求権を持つか。
6. モデル能力が急変したとき、どの文書が追加公開されるか。

性能比較が飽和しつつあるなかで、こうした governance due diligence は今後ますます重要になる。特に金融、製造、公共、医療、BPO、重要インフラでは、**モデルが賢いことより、社内説明に耐えること** が先に問われる場面が多い。

## 考察: 日本市場では「第三者レビューの存在」より「実際の権限設計」の方が重要

ただし、日本企業がこの種のニュースを見るときに気をつけたい点がある。多くのベンダーは「社外アドバイザーがいます」「独立委員会があります」と言う。しかし、実務上効くのは名称ではなく、**その主体が何を止められるか、何を承認できるか、何を受け取れるか** である。

その意味で Anthropic の今回の更新は比較的分かりやすい。追加されたのが review request 権限、 reviewer approval 権限、 regular briefing 義務だからだ。いずれも、ただ意見を述べるだけではなく、**情報フローや手続きの起点に入れる権限** である。

日本企業が他ベンダーを見るときも、同じ視点で比較するとよい。

- 独立主体は advisory か approval authority か
- 監督対象は marketing material か risk document か
- 単発会議か定期ブリーフィングか
- ボード統治と接続しているか、切り離されているか

ここを見ないと、「第三者レビューあり」という同じ表現でも、実質はかなり違う。

## 考察: とはいえ、RSP 3.2だけでベンダーリスクが解消されるわけではない

過大評価も禁物だ。RSP はあくまで Anthropic の自主枠組みであり、会計監査のような法定保証ではない。Risk Report の実際の内容がどこまで率直か、外部レビューがどこまで独立か、LTBT がどこまで厳しく動くかは、これからの公開文書と運用実績を見ないと判断できない。

さらに、日本企業のベンダー審査では次の論点も残る。

- データの保存と二次利用
- リージョンと越境
- ログと監査証跡
- アクセス制御と権限分離
- モデル更新通知
- 障害時や安全上の重大変更時の通知 SLA
- 委託先や下請けの構造

つまり RSP 3.2 は、安全統治の一部を見せる重要材料ではあるが、導入判断の全部ではない。それでも価値があるのは、これまで「安全への姿勢」という抽象論で語られがちだったものが、**手続き権限と文書運用のレベルまで降りてきた** からだ。

## 考察: 日本の開発チームも「ベンダーの統治を前提に設計する」段階に入った

開発者視点でも意味はある。これまで API の選定は、速度、価格、精度、コンテキスト長、ツール実行能力が中心だった。しかし社内展開が広がるほど、開発者自身も「どのベンダーなら社内承認を通しやすいか」を考えざるを得なくなる。

今後は、技術選定の初期段階から次のような質問が必要になる。

- このベンダーは能力向上時の安全説明を定期公開しているか。
- 社内と独立した統治主体があるか。
- リスクの高いモデルの扱いに review gate があるか。
- その情報を稟議、監査、顧客説明に流用できるか。

これは面倒に見えて、結果的には導入を速くする。後から法務や監査に止められるより、最初から **統治の説明可能性込みでベンダーを選ぶ** 方が、プロダクト実装も社内展開も進めやすいからだ。

## まとめ

Anthropic の RSP 3.2 改定は、AI 安全論の抽象的な議論ではなく、**Risk Report を誰が監督するのかという実務の話** である。LTBT が外部レビュー要求権、レビュアー承認権、定期ブリーフィングの受領を持つようになったことで、Anthropic の安全説明に対する独立統治の接続が一段強まった。

日本企業にとって重要なのは、これを「Anthropic は安全」という広告として読むことではない。むしろ、**AI ベンダーの統治をどう比較するかの具体例** として読むべきだ。今後の調達や監査では、モデル性能の差よりも、どのベンダーがどの文書を出し、誰がそれを監督し、どこまで独立した統治が制度化されているかが採用可否を左右する場面が増える。

RSP 3.2 はその流れを象徴する更新であり、日本の開発組織も「性能の良いモデルを選ぶ」から一歩進んで、**説明責任のあるベンダーを選ぶ** フェーズに入ったと見るべきだろう。
