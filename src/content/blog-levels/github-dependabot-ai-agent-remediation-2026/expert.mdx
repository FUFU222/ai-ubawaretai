GitHubが2026年4月7日に出した[Dependabot alerts are now assignable to AI agents for remediation](https://github.blog/changelog/2026-04-07-dependabot-alerts-are-now-assignable-to-ai-agents-for-remediation/)は、見た目以上に重要です。表面的には「DependabotからCopilot、Claude、Codexへ脆弱性修正を投げられるようになった」という話ですが、本質はもっと別のところにあります。GitHubがついに、**サプライチェーンセキュリティの実務を“検知”から“修復オーケストレーション”へ押し進めた**という点です。

脆弱性管理の現場では、検知だけ先に自動化され、修復は人力のまま残ることが多いです。SBOM、依存関係グラフ、アラート、優先度付けまでは進んでも、最後の「安全にアップグレードし、壊れたコードを直し、テストを通し、レビュー可能なPRにする」は属人的になりがちです。今回GitHubがエージェントを差し込んだのは、まさにその一番高コストな最後の区間です。

## 何が新しくなったのかを分解して見る

GitHub Changelogに書かれているフローは短いですが、かなり示唆に富んでいます。Dependabotアラートの詳細画面から `Assign to Agent` を選び、Copilot、Claude、Codexのようなコーディングエージェントへ割り当てると、エージェントは次の流れで動きます。

1. 脆弱性アドバイザリの内容を読む
2. リポジトリ内でその依存関係がどのように使われているかを確認する
3. 修正案を含むドラフトPRを開く
4. 更新によって生じたテスト失敗の解消を試す

ここで面白いのは、GitHubがわざわざ「リポジトリのdependency usageを見る」と明示していることです。これは単なるテキスト生成ではなく、**アラート文脈とコードベース文脈を束ねて修正案を作る**ことを意図しています。つまり、DependabotはCVEやアドバイザリを起点に「何を直すべきか」を渡し、エージェントは「このリポジトリではどう直すべきか」を考える役目です。

さらに重要なのが、**同じアラートに複数エージェントを独立に割り当てられる**ことです。各エージェントは別々のドラフトPRを開くので、開発チームはモデルやエージェントの違いを理屈ではなく差分として比較できます。これはかなり大きい。AIの評価をベンチマークの数字ではなく、**自分たちの本物の脆弱性修正タスク**で比較できるからです。

## Dependabotとエージェントは競合ではなく、役割の切り分けがきれい

今回の発表を深く読むと、GitHubはDependabotとAIエージェントの役割をかなり明確に分けています。

Dependabotが担うのは、主に以下です。

- 脆弱な依存関係の検知
- 利用中バージョンと安全な更新先の把握
- パッチがある場合の自動セキュリティPR
- アラートの優先度付けや管理

一方、エージェントが担うのは次です。

- メジャー更新で壊れた呼び出し側コードの修正
- 非推奨APIから新APIへの移行
- 型変更や設定変更への追随
- パッチ未提供時の安全なダウングレード提案
- テスト失敗の一次解消

これはかなり合理的な分割です。Dependabotのルールベース更新は、**最小安全バージョンへ機械的に寄せる**ことに強い。しかし、それで壊れる部分の修正は、リポジトリごとの事情に強く依存する。そこはルールではなく、コード理解と試行錯誤が必要になります。だからGitHubは、アラートと修正版PRの間にエージェントを置いた。

## なぜこの領域は今まで自動化しにくかったのか

依存関係の脆弱性対応が難しい理由は、更新作業が「同じことの繰り返し」に見えて実際にはそうではないからです。

簡単なケースなら、Dependabotが出したPRをそのまま通せます。でも難しいケースでは、次のような連鎖が起きます。

- バージョンを上げる
- コンパイルやビルドが壊れる
- 型やAPI変更に合わせてコードを書き換える
- テストが壊れる
- 振る舞い差分を確認する
- 場合によっては設定値やインフラ定義まで直す
- レビュー担当者へ「なぜこの差分が必要か」を説明できる形に整える

この最後の説明可能性が、実はかなり重要です。セキュリティ修正は緊急度が高くても、レビュー側から見ると「この変更は本当に脆弱性修正に必要なのか」「単なるリファクタが混ざっていないか」が知りたい。AIエージェントを使うと、便利な一方で差分が広がる危険があります。だからドラフトPRとして出し、比較し、人間が絞り込む設計になっているのは理にかなっています。

## GitHubの広い製品戦略の中で見ると、かなり筋が通っている

最近のGitHubを追っていると、今回の発表は突然ではありません。

2月26日の[What’s new with GitHub Copilot coding agent](https://github.blog/ai-and-ml/github-copilot/whats-new-with-github-copilot-coding-agent/)では、coding agentに**モデル選択、自己レビュー、ビルトインのセキュリティスキャン、カスタムエージェント、CLI handoff**が追加されたと案内されていました。3月から4月初旬にかけては、Copilot SDK、cloud agentのrunner制御、firewall設定、署名付きコミット、CLIの `/fleet`、Rubber Duck、BYOK/ローカルモデル、VS CodeのAutopilotまで連続投入されています。

この並びを見ると、GitHubはAIを単なるIDE補完機能としてではなく、**既存のGitHub運用面の中で働くエージェント群**として組み立てています。以前このサイトで書いた[GitHub Copilot SDKがパブリックプレビュー公開。自社アプリにCopilotエージェントを組み込める時代へ](/blog/github-copilot-sdk-public-preview-2026/)は、そのランタイムを外へ配る話でしたし、[GitHub CopilotにAutopilot登場。VS Codeが「多エージェント開発OS」になるのか](/blog/github-copilot-autopilot-vscode-2026/)は、日常の開発面に自律性を持ち込む話でした。今回のDependabot連携は、そこへ**セキュリティ修復フロー**を接続したものです。

言い換えると、GitHubは「コードを書くAI」ではなく、**issue、PR、security alert、review、CI、branch protection を横断する作業面**を取りにきています。脆弱性修正の入口がDependabotで、実行役がエージェントで、出口がドラフトPRという設計は、その戦略に非常にきれいに一致しています。

## Octoverseの数字と合わせると、この機能の狙いがよく見える

GitHubの2025年Octoverseでは、Dependabotが設定で有効化されているリポジトリは**266.8万件超**、前年比約24%増でした。重大脆弱性の平均修正時間は**37日から26日へ短縮**し、30%改善した一方で、Dependabot関連の修正は**同じ月にマージされるのが約3分の1**という状態です。

この数字から読み取れるのは、GitHubにとっての次の課題が「もっと検知する」ことではなく、**見つかった修正を最後まで前へ進めること**だという点です。検知が増えるほど、未対応アラートの山も増えます。そこでGitHubは、アラートの詳細画面そのものをエージェント起動面に変えた。これはUI改善ではなく、運用のボトルネックを狙った設計変更です。

さらに言えば、GitHubはこの領域をすでに「AIが効く」と見ています。Octoverseでは、AI利用と自動化の組み合わせで重大アラートを受けるリポジトリが26%減ったこと、Broken Access Controlのような難しい分類でもAI生成修正が一定の採用を得ていることが示されています。今回のDependabot連携は、その延長線上にあります。

## 日本市場での意味は、単なる時短ではない

日本の開発組織でこの機能が効く理由は、時間短縮だけではありません。むしろ大きいのは、**脆弱性修正の説明責任を保ったまま初動を速められる**ことです。

日本の金融、通信、公共、製造では、依存関係更新ですらレビュー記録、変更理由、承認者、CI結果が求められます。完全自動マージは難しくても、ドラフトPRなら既存フローに乗せやすい。しかも今回の機能は、GitHub Code Securityとcoding agentアクセスを含むCopilotプランが前提です。つまり無秩序な野良AIではなく、**GitHubの統制面の内側で動く修復案**として扱える。

もう一つ日本で重要なのは、人手不足です。セキュリティ専任者が少ない組織では、AppSecチームが全リポジトリを直接直すことはできません。開発チームへ直してもらう必要がありますが、そこでは「何から手を付けるか」「更新で何が壊れるか」の初動調査が最大の負担になりがちです。DependabotからAIへ渡してドラフトPRを出せるなら、セキュリティ担当は優先度付けに集中し、開発チームはレビューと仕上げに集中しやすくなる。

## ただし、ここには新しいリスクもある

この機能を過大評価すると危ないです。GitHub自身が明言している通り、AI修正は不完全であり得ます。特に脆弱性修正では、次の失敗パターンが現実的です。

- テストが薄いので、見えている失敗だけ潰して根本原因を外す
- 脆弱性そのものは解消しても、互換性や性能を落とす
- 不必要な大規模差分を出してレビュー負荷を増やす
- 修正案の根拠が曖昧で、監査や障害時の説明が弱い
- モノレポや複数サービス依存で、1つのリポジトリ内だけでは真の影響範囲を閉じられない

さらに、依存関係の脆弱性は「ライブラリを上げれば終わり」とは限りません。認証、暗号、入力検証のような領域では、ライブラリ更新に合わせて**アプリケーション側の利用方法そのものを変える必要**がある場合があります。そこでエージェントが雑な後方互換コードを足してしまうと、脆弱性は塞がっても技術的負債が増えます。

だから実務上は、次のようなガードレールが必要です。

1. ドラフトPRのまま出させる
2. CI、型検査、統合テストを必須化する
3. セキュリティ担当かコードオーナーのレビューを必須にする
4. 低リスク依存から段階的に導入する
5. 複数エージェント比較を使う場合は、差分の小ささと説明しやすさも評価軸に入れる

## 技術的には「比較可能な修復」が一番おもしろい

今回の機能で自分が最も重要だと思うのは、実は複数エージェント比較です。多くのAI機能は、ユーザーが1つのモデルに固定され、良し悪しを体感で判断するしかありません。でもDependabotアラートを起点にした修復なら、同じ入力、同じコードベース、同じ脆弱性で**複数の修正案を並べられる**。

これは今後かなり効くはずです。日本企業では「どのモデルを採用するか」を契約や稟議で大きく決めたがりますが、本来はタスクごとに得手不得手が違う。依存関係更新のような狭い領域では、どのエージェントが

- 最小差分で直すか
- テスト修正まで丁寧か
- 型や設定変更への追随がうまいか
- 余計な変更を混ぜないか

を、PRレベルで比較できます。これは単なる便利機能ではなく、**エージェント評価の現場化**です。

## まとめ

Dependabotの今回の更新は、サプライチェーン防御のワークフローを1段前へ進めました。脆弱性を見つけ、更新先を示し、さらにAIエージェントへ渡して修復案のPRを作らせる。これは「検知」と「修復」の間に残っていた人力ゾーンを埋める動きです。

GitHubはすでに、Copilot coding agent、SDK、Autopilot、cloud agent、CLI handoffを通じて、開発の実行面をエージェント化しようとしてきました。今回それがDependabotへ接続されたことで、GitHubの狙いはかなり明確になりました。AIをコード補完として売るのではなく、**GitHub上で起きる仕事全体のオーケストレーション面**を取ろうとしている。

日本の開発組織にとっての論点も同じです。AIが脆弱性を直せるかどうか、ではありません。**脆弱性対応の最後の詰まりを、既存のレビューと監査の枠内でどこまで圧縮できるか**です。その観点では、今回のDependabot連携はかなり本質的なニュースだと思います。
