Cursor が、Jira から AI エージェントを呼び出せるようにしました。Jira の work item を Cursor に割り当てたり、コメントで `@Cursor` と書いたりすると、Cursor の Cloud Agent がそのチケットを読んで作業を始めます。終わると、Jira に進捗や完了情報が出て、作られた pull request へのリンクも表示されます。

これは、ただのチャット連携ではありません。これまで人間の開発者が「チケットを読んで、コードを書いて、PR を出す」流れだったものに、AI エージェントが入ってくるということです。

## 何ができるようになるのか

Cursor の発表では、Jira の work item から bug fix、feature 追加、test 更新、調査のような仕事を依頼できるとされています。Cursor はチケットのタイトル、説明、コメント、チームのリポジトリ設定を見て、何をするべきかを判断します。

たとえば、Jira に「ログイン画面でエラーが出る。再現手順はこれ」と書かれていれば、Cursor が関連コードを調べ、修正案を作り、PR を出す流れが考えられます。完了したら Jira 側で PR へのリンクを確認できます。

使うには Cursor の管理者権限が必要で、Jira 側は Rovo が有効な Commercial Cloud が前提です。つまり、個人が勝手に入れるというより、チームや会社で導入する機能です。

## なぜJira連携が重要なのか

開発チームでは、Jira チケットが仕事の入口になっていることが多いです。何を直すのか、なぜ必要なのか、誰が確認するのか、いつまでに必要なのかがチケットに集まります。

そこから AI エージェントを起動できると、AI コーディングが個人のエディタ内の便利機能ではなく、チームの開発フローの一部になります。PM や QA は Jira で進捗を見られ、エンジニアは PR で差分をレビューできます。

ただし、チケットが曖昧だと AI も迷います。「バグを直して」だけでは足りません。再現手順、期待する動き、対象画面、テスト方法、やってはいけないことをチケットに書く必要があります。

## 日本のチームで気をつけたいこと

最初に決めるべきなのは、誰が Cursor を呼べるかです。全員が自由に `@Cursor` できると、同じ作業が何度も走ったり、まだ承認されていない変更が始まったりします。まずは小さなチームで試すのが安全です。

次に、任せる仕事を絞ることです。小さな bug fix、テスト追加、調査、ドキュメント更新のような作業から始めると評価しやすいです。大きな設計変更や、まだ要件が決まっていない機能をいきなり任せるのは危険です。

最後に、PR の責任者を決めることです。Cursor が PR を作っても、マージしてよいかを判断するのは人間です。レビュー担当者、テスト担当者、最終確認者を Jira に残しておくと、あとで混乱しにくくなります。

## まとめ

Cursor の Jira 連携は、Jira チケットから AI エージェントに作業を渡し、PR までつなげるための更新です。うまく使えば、開発チームは小さな修正や調査をより早く回せます。

一方で、AI に任せるほど、チケットの書き方、権限、レビュー責任が大事になります。まずは対象を絞り、チケットテンプレートを整え、人間が必ず PR を確認する形で試すのがよいでしょう。
