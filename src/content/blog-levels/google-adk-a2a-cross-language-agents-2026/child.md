Google が公開した ADK と A2A の記事は、いろいろな言語で作られた AI エージェントをつなぐ方法を説明しています。たとえば、全体をまとめるエージェントは Python で作り、特定の処理を担当するエージェントは Go や Java で作る、という構成です。

ADK は Agent Development Kit のことで、AI エージェントを作るための道具です。A2A は Agent2Agent の略で、エージェント同士が通信するためのプロトコルです。大事なのは、A2A が「エージェントからツールを呼ぶ仕組み」ではなく、「エージェント同士をつなぐ仕組み」だという点です。

## 何ができるようになるのか

Google の記事では、Python のエージェントから Go のエージェントを呼ぶ例が紹介されています。ADK の `RemoteA2aAgent` を使うと、相手のエージェント情報を確認し、パラメータを渡し、ネットワーク越しに処理を呼ぶ部分を扱いやすくできます。

これは、1つの大きなエージェントに全部の処理を入れるのとは違います。得意な言語や既存のシステムに合わせて、複数のエージェントに役割を分けられます。たとえば、業務ロジックは Java、速い処理は Go、AI との会話や全体調整は Python という分け方ができます。

## なぜ日本企業に関係するのか

日本企業では、すでに多くのシステムが Java、Go、Python、TypeScript などで作られています。AI エージェントを導入するために、すべてを1つの言語で作り直すのは現実的ではありません。

A2A を使うと、既存のシステムやチームの境界を残したまま、エージェント同士をつなぐ設計を取りやすくなります。各チームが自分の担当エージェントを作り、共通の通信ルールで連携するイメージです。

## 注意すべきこと

まず、A2A と MCP を混同しないことが重要です。MCP はエージェントがツールを呼ぶための仕組みとして使われます。一方、A2A はエージェント同士が会話するための仕組みです。業務システムをただ呼び出すだけなら tool でよい場合があります。判断や複数ステップの処理を持つなら、agent として分ける意味が出ます。

次に、契約を決める必要があります。相手のエージェントは何を受け取り、何を返すのか。失敗したときはどんなエラーを返すのか。どのバージョンの約束で動くのか。こうしたことを決めないままつなぐと、動いたように見えても運用で壊れやすくなります。

権限も大切です。あるエージェントが別のエージェントを呼べるとして、その先でどのデータを読めるのか、どの操作をしてよいのかを決める必要があります。AI エージェントだからといって、何でも見せてよいわけではありません。

## 小さく試すなら

最初は、2つのエージェントだけで試すのがよいです。Python のまとめ役エージェントと、Go または Java の専門エージェントを1つ作ります。まずはサンプルデータで、入力、出力、エラー、ログを確認します。

次に、実際の業務に近い小さなデータで試します。見るべきなのは、答えが正しいかだけではありません。どのエージェントが何を判断したのか、ログで追えるか。エラーのときに止まれるか。権限外のデータを読みに行かないか。ここまで確認してから範囲を広げるべきです。

## まとめ

ADK と A2A は、AI エージェントを1つの大きな仕組みにするのではなく、役割ごとに分けてつなぐための考え方です。日本企業のように複数の言語やチームが混在する環境では、実務的な意味があります。

ただし、A2A を使えば自動的に安全になるわけではありません。契約、権限、ログ、テストを決めることが必要です。AI エージェントをチームで運用するなら、「作る」だけでなく「どうつなぐか」を設計する段階に入っています。
