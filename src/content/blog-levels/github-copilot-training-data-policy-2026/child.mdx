GitHub Copilot の方針変更で、「Copilot を使ったときのやり取りが学習に使われることがある」という点が大きな話題になりました。

しかも、自分で opt-out しないとそのまま使われるように見えたため、多くの開発者が強く反応しました。

## 何がそんなに気になるの？

開発者は、AI の答えが便利かどうかだけでなく、「自分たちの情報がどう扱われるか」もかなり気にしています。

コードそのものだけでなく、質問のしかた、エラーの内容、貼り付けた断片、社内 API 名、未公開機能の相談などにも会社の事情がにじむからです。

だから「学習に使うかどうか」は、かなり大きな問題になります。

## private repo なら大丈夫？

そこが一番ややこしいところです。

標準記事でも、private repo や enterprise 契約では扱いが違う可能性がある一方で、利用者から見ると説明や設定の分かりにくさが不安を残すと書かれていました。

つまり、「たぶん大丈夫そう」で済ませるのが一番危ないです。

自分が使っているプラン、組織設定、個人設定がどうなっているかを確認しないと、本当にどこまで守られるのかが分かりません。

## なぜここまで反発が起きるの？

Copilot のような道具は、日々の開発の中に深く入り込みます。

だからこそ、あとから「そのやり取りも学習に使うかもしれません」と見える形になると、信頼が揺れやすいです。

AI は便利なほど、内部の仕事や考え方がそこに流れ込みやすいので、便利さと引き換えに何を渡しているのかが前よりずっと重要になります。

## 使う側はどうすればいい？

まず、自分の設定を確認することです。

次に、会社で使うなら、組織として opt-out の方針や入力ルールを決める必要があります。

機密情報、顧客情報、未公開機能、脆弱性の詳細など、どこまで AI に入れてよいかをあいまいにしないことも大事です。

「便利だから個人判断で使う」だけでは、後で問題になりやすいです。

## まとめ

今回の話は、Copilot が便利かどうかより、「便利な道具をどこまで信用してよいか」という話です。

AI ツールを仕事に入れるなら、性能だけでなく、データの扱いも同じくらい真剣に見ないといけません。
