GitHub Copilot の学習データ方針変更が炎上したのは、単にプライバシー懸念が強いからではなく、開発者が AI アシスタントを「高い trust boundary の内側」に置いているからです。

コード補完ツールは日常的に機密断片、障害情報、設計意図、未公開機能に触れます。

そこで default-on に見える学習利用が入ると、期待していた境界と運営側の境界定義がずれます。

## 問題はコードそのものだけではない

多くの利用者は「private repo の全文が学習されるか」だけを心配しているように見えますが、実際には会話内容や補助入力も十分センシティブです。

エラーメッセージ、社内 API 名、障害時の暫定対応、顧客向け仕様の断片など、周辺情報からでも内部事情は推測できます。

そのため、interaction data の扱いは enterprise procurement の論点になります。

## opt-out 前提は信頼コストが高い

法務的に説明可能であっても、運用として opt-out が必要な設計は、利用者側に確認負荷を押し付けます。

特にチーム導入では、誰がどの設定を見たか、契約プランで何が違うか、組織設定が端末設定より優先されるかまで確認が必要になります。

この確認コストは、単なる UI 問題ではなく導入障壁です。

## 比較対象は他社モデルの privacy posture

Copilot の議論は、Claude、ChatGPT Enterprise、各種 self-host 系ツールとの比較で見ないと意味がありません。

企業が選ぶのは「最も賢いツール」ではなく、「情報境界の説明責任を果たしやすいツール」でもあるからです。

その意味で、データ利用ポリシーは性能とは別の競争軸になっています。

## 実務上の対応

組織で使うなら、opt-out だけでなく、入力ルール、機密情報分類、監査ログの確認方法までセットで決めるべきです。

AI 導入は個人の便利設定ではなく、情報管理の運用設計です。

## まとめ

今回の Copilot 政策変更は、生成 AI ツールの勝負が「能力」だけでなく「情報境界の扱い」に移っていることをよく示しています。

信頼を一度削ると、機能追加より回復のほうがずっと難しいです。
