GitHub Copilot の学習データ方針変更が炎上したのは、単にプライバシー懸念が強いからではなく、開発者が AI アシスタントを「高い trust boundary の内側」に置いているからです。

コード補完ツールは、日常的に機密断片、障害情報、設計意図、未公開機能に触れます。

そこで default-on に見える学習利用が入ると、利用者が期待していた境界と、運営側の境界定義がずれます。

## 問題はコード全文だけではない

多くの利用者は「private repo の全文が学習されるか」だけを心配しがちですが、実際には interaction data 全体が十分センシティブです。

エラーメッセージ、社内 API 名、障害時の暫定対応、顧客向け仕様の断片、設計の悩み方そのものまで、周辺情報から内部事情はかなり推測できます。

標準記事が指摘していた通り、これはコード保存の話というより、会話と補助入力の扱いの話でもあります。

## opt-out 設計は確認コストを押し付ける

法務的には説明可能でも、運用として opt-out が必要な設計は利用者側へ確認負荷を押し付けます。

特にチーム導入では、契約プランごとの差、個人設定と組織設定の優先関係、private repo の扱い、学習利用の対象範囲まで確認が必要です。

この確認コストは単なる UI 問題ではなく、導入障壁そのものです。

「設定で切れるから問題ない」とは簡単に言えません。

## 他社比較で見るべきポイント

Copilot の議論は、Claude、ChatGPT Enterprise、self-host 系ツールなどの privacy posture と比較して初めて意味を持ちます。

企業が選ぶのは「一番賢いツール」だけではありません。

情報境界を社内で説明しやすいか、法務やセキュリティレビューを通しやすいかも同じくらい重要です。

その意味で、データ利用ポリシーは性能とは別の競争軸になっています。

## 組織で必要な対応

組織で使うなら、opt-out の有無だけでなく、入力ルール、機密情報分類、利用ログ、監査方法までセットで決めるべきです。

AI 導入は個人の便利設定ではなく、情報管理の運用設計です。

ここを曖昧にすると、便利さは得られても、後から説明責任で詰まりやすくなります。

## まとめ

今回の Copilot 政策変更は、生成 AI ツールの勝負が「能力」だけでなく「情報境界の扱い」に移っていることをよく示しています。

信頼は機能より遅く積み上がり、ずっと速く壊れるので、この論点は今後さらに重くなるはずです。
