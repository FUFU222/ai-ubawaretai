Google Workspace Updates は **2026年5月27日**、Workspace Studio のステップとスターターに対して、より細かな管理者制御を追加した。Rapid Release と Scheduled Release の両方で、**2026年5月26日** から 1〜3日でフルロールアウトされる更新だ。

この記事では、Workspace 管理者、情シス、セキュリティ担当、業務改善チーム向けに、今回の更新を「AI自動化を現場へ広げるための制御面」として整理する。Workspace Studio は、単なるノーコード便利機能ではない。Gmail、Chat、Drive、Calendar、Meet、Gemini、外部連携を業務フローとしてつなぐ入口になり得る。つまり、管理者が先に制御粒度を決めないと、現場の小さな自動化がそのまま業務実行基盤になる。

まず関係を切り分けたい。[Workspace Intelligence の管理者制御](/blog/google-workspace-intelligence-admin-controls-2026/) は、Workspace 内の文脈を Gemini がどう扱うかという基盤の話だった。[Gemini Chat の日本語Refine](/blog/google-chat-gemini-refine-japanese-2026/) は、送信前文面のAI補助だった。[Google AI Studio の Workspace 連携](/blog/google-ai-studio-android-workspace-2026/) は、開発・試作側から Workspace データへ近づく動きだった。Workspace Studio は、より現場の定型業務に近いところで、AIと自動化を実行する面になる。

## 事実整理: 今回の更新で管理者が制御できるもの

Google の発表では、Workspace Studio の steps and starters に対し、管理者がより細かな制御を定義できるようになったと説明されている。具体的には、ユーザーがフロー作成に使えるステップやスターターを、Workspace サービス別または個別に制御できる。

スターターは、フローを開始する条件だ。メール受信、会議出力、ファイル更新、スケジュールなど、業務イベントを起点にフローが動く。ステップは、フロー内で実行される処理だ。通知、文書生成、データ更新、AIによる要約や分類、外部連携などがここに入る。

今回の発表で重要なのは、Workspace Studio のスターターとステップは既定でオンになり、管理者がドメイン、OU、グループ単位で無効化できる点だ。ユーザー側に個別設定はない。管理者が無効化したステップはユーザー画面で disabled として表示され、既存フローがそのステップを使っている場合はエラーになる。

これは運用上かなり重要である。管理者が後から制限をかけると、既存フローが止まる可能性がある。したがって、制限変更は「設定変更」ではなく「業務影響がある変更」として扱うべきだ。対象フロー、所有者、影響範囲、代替手順を確認してから無効化する運用が必要になる。

利用可能エディションも広い。発表では、Business Starter / Standard / Plus、Enterprise Standard / Plus、Education Fundamentals / Standard / Plus、Google AI Pro for Education、Teaching and Learning、AI Expanded Access などが挙げられている。さらに、AI Expanded Access の Workspace Studio 利用上限は 2026年6月1日から引き上げられる予定とされている。

## 管理者ガイドから読む制御レイヤー

Google Workspace Help の管理者ガイドは、Workspace Studio のアクセス管理を複数のレイヤーに分けている。Gemini を使ったフロー作成、AI-powered steps、Workspace サービス向けステップ、連携やカスタムステップを制御できるという整理だ。

この分け方は実務的だ。多くの企業では、「AIは許可するが外部連携は止める」「Gmail と Calendar は許すが Drive 書き込みは制限する」「人事部門には自動送信系を許可しない」「開発部門にはカスタムステップを許可するが、営業部門には公開済みテンプレートだけにする」といった管理が必要になる。Workspace Studio 全体のオン/オフだけでは粗すぎる。

管理者ガイドは、Workspace サービス向けステップについて、ユーザーがアクセスできるサービスのステップは基本的に許可し、悪用リスクがある場合にだけブロックするという考え方を示している。ここは、過度に保守的な全停止を避けるヒントになる。自動化は、現場が使わないと価値が出ない。一方で、重要サービスや高リスク処理は個別に絞るべきだ。

また、Marketplace allowlist により、連携や公開済みカスタムステップを管理する考え方も示されている。ここはセキュリティ担当が見るべき点だ。AIステップだけでなく、外部SaaS、社内API、Webhook、Marketplace 経由の連携がデータ移動の経路になる。Studio の画面で作れることと、会社として接続を許可できることは別である。

さらに管理者ガイドでは、Studio activity に関するアラートも案内されている。高頻度実行の検知、Ask Gemini など生成AIステップの利用監視が例として挙げられている。これは、Workspace Studio が「作って終わり」のツールではなく、実行状況を監視する対象であることを意味する。

## 日本企業で起きやすい失敗パターン

第一の失敗は、現場改善ツールとして無条件に開放することだ。日本企業では、Excel やスプレッドシートのマクロ、Apps Script、RPA、Zapier 系ツール、SaaS のワークフロー機能が、現場主導で広がってきた歴史がある。Workspace Studio も同じ流れで使われる可能性が高い。便利な個人フローが増え、数カ月後には誰も所有していない業務依存になる。

第二の失敗は、AIだけをリスクとして見て、スターターや書き込み先を見落とすことだ。Ask Gemini のようなAI-powered steps は目立つ。しかし、実際の事故は、メールをきっかけに不要な通知を送る、Driveファイルを広く共有する、Chat に機密を投稿する、外部連携へデータを流す、といった非AI部分でも起きる。AIガバナンスは、モデル利用だけではなく、起動条件と出力先まで含めるべきだ。

第三の失敗は、全社一律のルールにすることだ。営業、採用、法務、経理、SRE、カスタマーサクセスでは、扱う情報と許容できる自動化が違う。営業は顧客メールとCalendar連携の効果が大きいかもしれない。SREはインシデント通知と記録の自動化を求める。法務は外部向け文面や契約情報の扱いに厳格なレビューを必要とする。OU / group ごとに設計しなければ、現場には使いにくく、管理者には危ない設定になる。

第四の失敗は、実行上限を軽視することだ。Workspace Studio Help では、作成できるフロー数、24時間内の実行回数、Gmail starter を使う active flow 数、1フローのステップ数に制限があると説明されている。上限に達すると active flow が止まり、実行失敗やエラー通知が起きる。重要業務に使うなら、上限到達時の運用が必要だ。

## 推奨する導入設計

最初に、Workspace Studio の利用を3段階に分けるとよい。第1段階は個人または小チームの低リスク自動化だ。社内通知、日程調整、個人のタスク整理、限定されたDocsやSheetsへの記録などを対象にする。ここでは外部連携を止め、AIステップも要約や分類など低リスク用途に限定する。

第2段階は部門テンプレート化だ。営業、CS、採用、社内ITなど、効果が見えやすい部門で、承認済みフローをテンプレートとして配布する。フロー共有を許す場合は、所有者、レビュー者、更新手順を明確にする。共有されたフローには、入力データ、出力先、AIステップ、外部連携、実行頻度を明記する。

第3段階は業務基盤化だ。ここでは、監査ログ、アラート、所有者の引き継ぎ、障害時の停止、上限管理、変更審査を入れる。重要な業務プロセスを Workspace Studio で動かすなら、RPA や Apps Script と同じように、台帳と運用責任を持たせる必要がある。

制御項目としては、少なくとも次を棚卸しするべきだ。どの OU / group に Workspace Studio を許すか。どのスターターを許すか。どの Workspace サービス向けステップを許すか。AI-powered steps をどの用途で許すか。Marketplace allowlist をどう管理するか。カスタムステップを誰が作り、誰がレビューするか。フロー共有を許すか。高頻度実行やAIステップ利用のアラートをどう置くか。

## セキュリティ・法務・現場の責任分担

情シスは、設定、対象ユーザー、OU / group、サービス別ステップの許可、上限とエラーの監視を持つ。セキュリティは、データ分類、外部連携、認証情報、カスタムステップ、Marketplace allowlist、アラート条件を持つ。法務は、顧客向け文面、契約、個人情報、公式見解、監査証跡の扱いを確認する。現場部門は、実際のフロー所有者と業務影響を持つ。

この分担を決めないまま進めると、Workspace Studio は「誰でも作れるが誰も責任を持たない」状態になりやすい。特に日本企業では、現場の業務改善担当が善意で作ったフローが、いつの間にかチームの標準作業になることがある。作成者が異動した後に止まり、誰も直せない。これは、AI以前からある自動化の典型的な負債だ。

生成AIが入ると、この負債は見えにくくなる。AIが要約した内容、分類した問い合わせ、生成した通知文が、人間の判断を通らず次の処理へ流れる可能性があるからだ。したがって、AIステップを含むフローには、人間レビューを置く場所、ログを見る場所、誤分類時に戻す手順を先に決めておく必要がある。

## 具体的な利用シナリオ

営業部門では、問い合わせメールを起点に、顧客名、製品カテゴリ、緊急度を整理し、担当チームへ Chat 通知する用途が考えられる。ただし、AIが分類した緊急度をそのままSLA判断に使うのは危うい。最初は通知文の下書きやラベル補助にとどめ、人間が確認する運用がよい。

採用部門では、面接日程、候補者連絡、面接官へのリマインド、フィードバック収集の自動化に使える。ここでは個人情報が含まれるため、外部連携と共有範囲を強く絞るべきだ。AIステップで候補者評価を自動生成するような使い方は、慎重に扱う必要がある。

社内ITでは、問い合わせ受付、障害通知、メンテナンス案内、FAQ誘導に使いやすい。AIステップは問い合わせ分類や回答候補の生成に効くが、実際の設定変更やアカウント操作は人間承認を残したほうがよい。フロー実行上限に達するとサポート運用が止まるため、重要フローは監視対象にする。

プロダクト・開発組織では、会議出力の整理、仕様変更の通知、リリース準備チェック、サポート問い合わせの分類に使える。ただし、GitHub、Jira、社内APIなどへ接続する場合は、Workspace Studio だけでなく接続先の権限も見る必要がある。フローが便利でも、開発ワークフローの変更責任は別途残る。

## 今回の更新の読み方

今回の細かな管理者制御は、Google Workspace がAI機能を単発のアプリ機能から、組織が管理する業務面へ移していることを示している。Workspace Intelligence は文脈、Gemini Chat は日常文面、Connected Sheets は現場分析、Workspace Studio は自動化実行面だ。それぞれの機能を別々に見ると小さく見えるが、組み合わせると、Google Workspace 上で仕事の入力、理解、判断補助、実行がつながる。

日本企業が注目すべきなのは、Google が管理者向け制御を先に増やしている点だ。これは、AI自動化が無秩序に広がるリスクを Google 側も認識しているということでもある。管理者がこの制御を使わなければ、既定オンの便利機能が現場に先に広がり、後から止めるほど業務影響が出る。

したがって、今回の更新は「Workspace Studio が少し便利になった」ではなく、「Workspace 上のAI自動化を部品単位で許可する準備ができた」と読むべきだ。導入済み企業は、全社展開前に設定を見直す価値がある。未導入企業も、今後 Workspace Studio を使う前提で、OU / group、外部連携、AIステップ、フロー台帳の設計を先に作っておくとよい。

## まとめ

Workspace Studio のステップとスターター単位の管理者制御は、日本企業が Google Workspace 上で業務AI自動化を扱ううえで重要な更新だ。現場の小さな改善を止めずに、危ない部品だけ絞るための制御粒度が増えた。

実務上は、次の順で進めるのがよい。まず、OU / group ごとの対象範囲を決める。次に、スターター、Workspace サービス向けステップ、AI-powered steps、外部連携、カスタムステップを分けて許可する。さらに、フロー共有、実行上限、高頻度実行、AI利用アラート、所有者引き継ぎを運用に入れる。

Workspace Studio は、Google Workspace の中で現場が自動化を作れる強い入口になる。便利さは大きいが、業務を動かす以上、責任も発生する。今回の管理者制御を使い、全社禁止でも無条件解放でもない、部品単位のガバナンスを作ることが、日本企業にとっての現実解だ。
